title: AI Functions
---

Vane Data AI Functions have two semantic output models: the **Expression API** and the **Relation API**. They are not three syntax surfaces. For one row-preserving AI result beside source columns, begin with the SQL entry point inside Expression.

Both models use the same provider and UDF execution layer. The distinction is whether the operation is one projection result or a specialized table transformation.

## Expression API

The Expression API produces one row-preserving AI column in a `SELECT` projection. SQL is the recommended default entry point: stable identifiers and source fields remain explicit next to the model result.

### SQL Entry Point (Recommended)

Use SQL `ai_prompt` or `ai_embed` when one model result belongs beside each input row. This short projection keeps the identity and prompt that produced each response:

```python
import vane

con = vane.connect()
source = con.sql("""
    SELECT *
    FROM (VALUES
        (1, 'Summarize the refund request'),
        (2, 'Summarize the shipping delay')
    ) AS t(document_id, prompt)
""")

reviewed = con.sql("""
    SELECT
        document_id,
        prompt,
        ai_prompt(prompt) AS response
    FROM source
""")
```

The SQL call is still Expression API. Its provider options are constant planning inputs, while `document_id` and `prompt` continue to flow as ordinary relation columns.

### Python Entry Point

Python Expression is equivalent syntax with the same row-preserving projection semantics. It is useful when the surrounding projection is already being assembled with Python expressions:

```python
projected = source.select(
    vane.col("document_id"),
    vane.col("prompt"),
    vane.ai.prompt(vane.col("prompt")).alias("response"),
)
```

Changing from SQL to Python does not change the output model. Both forms add one AI result column and keep source fields by projecting them explicitly.

## Relation API

Move to Relation AI only when a capability needs a table transformation rather than one ordinary projection result:

- `rel.embed_text` provides built-in chunking through `max_chunk_chars`.
- `rel.classify_text` provides built-in classification.
- `rel.prompt` provides structured `return_format`, multimodal `image_columns`, and a named Relation output through `output_column`.
- Relation methods can select an explicit `execution_backend` and provider `concurrency` or `actor_number` controls.

Built-in classification requires Relation API, so this final illustration uses the Relation method rather than pretending classification is a SQL table function:

```python
classified = source.classify_text(
    "prompt",
    labels=["refund", "shipping", "other"],
    output_column="label",
)
```

Vane exposes no direct SQL Relation or table-function API. SQL can filter and project the input before a Relation call and can consume its result afterward, but the specialized Relation method itself is currently Python.

## Provider and Runner Lifecycle

Provider descriptors cross the plan boundary instead of carrying live clients or models with the plan. The selected client or model is instantiated lazily on workers and reused within each actor, avoiding setup for rows that never execute and amortizing setup across batches.

For SQL and Python Expression, the active runner chooses the actor backend when the query is planned: local execution uses a subprocess actor, while a Ray run uses a Ray actor. Relation methods can additionally choose their backend explicitly. Provider defaults and exact parameters belong in the [AI Function API Reference](/docs/data/reference/ai-api), not in this conceptual lifecycle description.

## Credentials and External Effects

Credentials belong in the worker environment because provider clients are created on workers. SQL option binding rejects credential-like fields, so API keys, tokens, and secrets should not be embedded in SQL plans.

Provider calls are external effects outside SQL transactions. They are not exactly-once: a failed or retried query can repeat a request, and transaction rollback cannot undo a completed provider call. Carry stable IDs through the projection, retain reviewable inputs and outputs, and make downstream writes idempotent where repetition matters.

## Choosing a Model

- Choose SQL Expression first for one prompt response or embedding per input row.
- Choose Python Expression for the same row-preserving result when Python composition is clearer.
- Choose Relation only for built-in chunking or classification, structured or multimodal prompts, named Relation output, or explicit Relation execution controls.

For exact signatures and provider parameter rules, use the [AI Function API Reference](/docs/data/reference/ai-api). For a complete task with setup, validation, and output handling, use the [AI Functions Guide](/docs/data/guides/ai-functions).
