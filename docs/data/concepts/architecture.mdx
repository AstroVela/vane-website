title: Architecture
---

Vane Data is a multimodal-native data engine. It combines DuckDB-compatible SQL, Python expressions, table-shaped Python and AI transformations, and local or Ray execution in one lazy relational plan.

The public architecture has two semantic API models: the **Expression API** and the **Relation API**. SQL and Python are entry points into Expression, not two additional top-level models.

## System view

The layers separate what an operation returns from how and where it runs:

```text
Public entry points                         Semantic API model       Planned work
SQL Expression (recommended) ──────────┐
Python Expression ─────────────────────┴──> Expression API ──┐
Python Relation methods ──────────────────> Relation API ────┴──> shared logical/physical planning
                                                                       │
                                                                local or Ray runner
                                                                       │
                                                               task/actor execution
```

Both models remain part of a table-shaped pipeline. Ordinary SQL still performs scans, filters, joins, projections, and surrounding writes. The semantic model determines whether custom work contributes one projection result or owns a table transformation; later layers decide its physical execution.

## Public API models

Choose the public model from the output contract, then choose its entry point and execution controls.

### SQL Entry Point to Expression API (Recommended)

SQL Expression is the default path when one typed result belongs beside each input row in a `SELECT` projection.

- `vane.attach_function(...)` gives a compatible Python UDF a registered SQL alias, and `vane.detach_function(...)` removes it.
- The registered alias is called in SQL as a projection expression.
- SQL `ai_prompt(...)` and `ai_embed(...)` add one AI result column through the same model.

Registration controls the SQL entry point to Expression; it is not a third API model. Keeping identifiers and source fields in the same projection makes the row-preserving contract and result provenance visible.

### Python Entry Point to Expression API

Python Expression is alternate syntax for the same projection model. `vane.col(...)`, `vane.lit(...)`, and `vane.sql_expr(...)` build inputs for `vane.func`, `vane.func.batch`, `vane.cls`, `vane.cls.batch`, `vane.ai.prompt`, and `vane.ai.embed` expressions.

The normal composable form produces one typed result for each input row, just like SQL Expression. Python batch Expression has a narrowly defined cardinality-changing exception; its placement and output rules belong in the API Reference.

### Relation API

Relation API is the specialized model for operations that own table shape. Custom Relation methods include `rel.map_batches(...)`, `rel.flat_map(...)`, and `rel.map(...)`. AI Relation methods include `rel.prompt(...)`, `rel.embed_text(...)`, and `rel.classify_text(...)` for capabilities such as structured or multimodal output, built-in chunking or classification, and explicit Relation execution controls.

Vane does not expose a direct SQL Relation or table-function API. SQL can produce the input relation before a Python Relation call and consume its result afterward, but the table-shaped method itself is currently Python Relation API.

## Shared planning and execution

All public forms build lazy relational work. Registered UDF aliases and SQL AI calls are lowered to the same logical projection execution model as Python Expression; they do not maintain separate SQL runtimes. Relation operators instead own their declared table-shaped output and cardinality contract.

| Public form | Logical contract | Execution ownership |
| --- | --- | --- |
| Registered SQL aliases, `ai_prompt`, `ai_embed` | Expression projection | Shared projection execution layer |
| `vane.func` / `vane.cls` / `vane.ai` expressions | Expression projection | Shared projection execution layer |
| Python `rel.*` UDF and AI methods | Relation transformation | Relation operator owns table shape |

Logical planning preserves this distinction. Physical planning then selects local or Ray task/actor execution without changing the public output model.

## Runner modes

Vane Data has two main execution modes:

| Mode | Best fit | Execution behavior |
| --- | --- | --- |
| Local | Development, testing, lightweight SQL, and single-node UDF jobs | Runs relational work in the driver process; Python UDF and AI execution uses `subprocess_task` or `subprocess_actor` workers on the same machine |
| Ray | Distributed scans and writes, distributed UDFs, model throughput, or explicit resource placement | Distributes suitable work and uses Ray task or actor execution |

Select Ray with `vane.configure(runner="ray")` or `VANE_RUNNER=ray`. If Ray is already initialized, the surrounding process can affect runner selection, so production scripts should configure the intended runner before creating connections and executing pipelines.

## UDF execution layer

The execution layer moves typed Arrow batches between relational operators and Python callables. Task backends run functions without long-lived state; actor backends construct one callable-class instance per actor and reuse it across batches. Multiple actors therefore own independent model, client, or cache instances.

| Backend | Callable shape | Typical role |
| --- | --- | --- |
| `subprocess_task` | Python function | Local stateless batch transforms |
| `subprocess_actor` | Callable class | Local model or client reuse |
| `ray_task` | Python function | Distributed stateless transforms |
| `ray_actor` | Callable class | Distributed model reuse or GPU-backed inference |

For Expression work, the active runner maps the logical task or actor requirement to a local or Ray backend. Relation methods can expose explicit backend controls because the Relation operator owns the complete table stage. Exact resource, state, and backend restrictions are documented in the API Reference.

## AI provider lifecycle

AI configuration crosses the planning boundary as a serializable provider descriptor, not as a live client or model. Instantiation is lazy and worker-side:

```text
Provider/options -> serializable plan descriptor -> worker's first batch -> live client/model -> actor reuse
```

This avoids initializing providers for work that never executes and amortizes setup across later batches handled by the same actor. SQL and Python AI Expression use the runner-selected actor backend; Relation AI can additionally expose explicit Relation execution controls.

Provider requests are external effects outside SQL transactions. A retry can repeat a completed request, and rollback cannot undo it, so pipelines should retain stable identifiers and use idempotent downstream handling where repetition matters.

## Configuration model

Programmatic and environment configuration select the same public runner setting:

```python
import vane

vane.configure(runner="ray")
```

```bash
export VANE_RUNNER=ray
```

`vane.configure(...)` applies registered `VANE_*` settings through the process environment. Use `vane.current_config()` or `vane.env.as_dict()` to inspect the public configuration snapshot. Configure the process before creating connections or starting work that may reach workers.

## Worker environment

Every worker needs the runtime context required by the work it executes:

- Python modules used by UDFs and AI providers.
- Object storage credentials and endpoint settings.
- Model files or accessible model download locations.
- Relevant `VANE_*`, `AWS_*`, `S3FS_*`, `DUCKDB_*`, `PYTHONPATH`, and `RAY_ADDRESS` environment values.

AI credentials belong in this worker environment because provider clients are created there. Credential-like fields are rejected from SQL AI options; API keys, tokens, and secrets should not be embedded in plans.

## Data movement

Common data paths follow the same layering:

- SQL scans local or remote data into DuckDB-compatible relations and reduces it with ordinary filters and projections.
- Expression execution sends the required argument columns through UDF or AI workers and combines the typed result with the surrounding projection.
- Relation methods receive Arrow batches and return their declared table output. If later stages need source columns, the Relation operation must include them in its output contract.
- Ray mode can materialize and move partitions through Ray workers.
- Writes use DuckDB-compatible or Vane Data relation paths according to the runner and target.

Place filters and column projections before expensive Python or model stages. This reduces data crossing into Python and, under Ray, reduces cluster movement. SQL can continue before and after a Python Relation stage without turning that stage into a SQL Relation API.
