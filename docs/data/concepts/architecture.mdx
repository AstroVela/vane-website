---
title: Architecture
---

Vane Data is a multimodal-native data engine. It exposes a DuckDB-compatible SQL and Python surface, adds Expression and Relation UDF and AI operations, and can execute selected work locally or through Ray.

The design goal is simple: keep pipelines table-shaped while allowing SQL, Python libraries, model inference, and distributed execution to live in one workflow.

## System view

A Vane Data pipeline has four user-visible layers:

1. API surface: `import vane`, `vane.connect()`, DuckDB-compatible SQL, Expression and Relation APIs, UDFs, and AI Functions.
2. Relation pipeline: SQL operations, projections, filters, joins, custom Expression results, Relation transformations, and writes are composed as relation steps.
3. Execution runner: local execution handles development and single-node jobs; the Ray runner distributes suitable scans, writes, and UDF or AI work.
4. UDF runtime: Python functions, callable classes, and AI providers run through task or actor backends.

At the API layer, Expression and Relation describe what an operation returns. Expression contributes a typed result to a projection, while Relation owns a table-shaped transformation. SQL is the recommended entry point for Expression UDFs and AI Functions; Python provides both Expression syntax and Relation methods.

This separation lets a pipeline start locally, then move selected work to Ray when data volume, model throughput, or resource placement requires it.

## Public user surface

Use these as the primary public entry points:

- `import vane`
- `vane.connect()`
- `vane.configure(...)`
- `vane.current_config()`
- `vane.env`
- DuckDB-compatible SQL through `con.sql(...)` and `con.execute(...)`
- SQL Expression UDF aliases registered with `vane.attach_function(...)` and removed with `vane.detach_function(...)`
- SQL AI Expressions through `ai_prompt(...)` and `ai_embed(...)`
- Python Expression inputs through `vane.col(...)`, `vane.lit(...)`, and `vane.sql_expr(...)`
- Python Expression UDFs through `vane.func`, `vane.func.batch`, `vane.cls`, and `vane.cls.batch`
- Python AI Expressions through `vane.ai.prompt(...)` and `vane.ai.embed(...)`
- DuckDB-compatible relation APIs
- `rel.map_batches(...)`
- `rel.flat_map(...)`
- `rel.map(...)`
- `rel.embed_text(...)`
- `rel.classify_text(...)`
- `rel.prompt(...)`

Choose Expression when custom work belongs in a projection, normally as one typed result beside each input row. Choose Relation when the operation owns the output columns or row count. Vane does not currently expose a direct SQL Relation or table-function API; SQL can prepare a relation before a Python Relation call and consume its result afterward.

Existing DuckDB-style imports remain available for compatible code, but new Vane Data projects should prefer `import vane`.

## Runner modes

Vane Data has two main execution modes.

Local mode:

- Runs relational work in the driver process on the local machine.
- Is the right starting point for development, testing, lightweight SQL, and single-node UDF jobs.
- Uses local subprocess task or actor workers for UDF and AI execution.

Ray mode:

- Selected with `vane.configure(runner="ray")` or `VANE_RUNNER=ray`.
- Uses Ray for distributed scans, distributed writes, distributed UDFs, and actor-based model workers.
- Requires every worker to have access to the same Python dependencies, storage systems, model files, and credentials needed by the pipeline.

If the process already has Ray initialized, runner selection can interact with that environment. For predictable behavior in production scripts, configure the intended runner explicitly before creating connections and running pipelines.

## UDF execution layer

UDFs reach the same execution layer through two public output contracts. Expression UDFs add a typed projection result: the recommended SQL path calls an alias registered with `vane.attach_function(...)`, while `vane.func` and `vane.cls` provide Python Expression syntax. Relation UDFs such as `rel.map_batches(...)`, `rel.flat_map(...)`, and `rel.map(...)` own a table-shaped stage and return its declared output.

Choose the backend based on callable shape and resource needs:

| Backend | Callable shape | Use case |
| --- | --- | --- |
| `subprocess_task` | Python function | Local stateless batch transforms |
| `subprocess_actor` | Callable class | Local model or client reuse |
| `ray_task` | Python function | Distributed stateless transforms |
| `ray_actor` | Callable class | Distributed model reuse or GPU-backed inference |

Task backends are for functions without long-lived state. Actor backends construct a callable class in each actor and reuse that instance across batches; separate actors keep separate state. The active runner maps Expression work to a local or Ray backend, while Relation methods can also expose explicit execution controls.

Exact signatures, cardinality exceptions, placement rules, and resource restrictions belong in the UDF API Reference.

## AI Function layer

AI Functions use the same Expression and Relation output contracts and the same worker execution model:

- SQL `ai_prompt(...)` and `ai_embed(...)` are the recommended path when one AI result belongs in a `SELECT` projection.
- Python `vane.ai.prompt(...)` and `vane.ai.embed(...)` provide alternate Expression syntax for the same projection model.
- Relation methods such as `rel.prompt(...)`, `rel.embed_text(...)`, and `rel.classify_text(...)` support table-shaped capabilities such as structured or multimodal output, chunking, classification, and explicit Relation execution controls.

AI provider configuration crosses the plan as serializable data. The live client or model is created lazily in the worker and can be reused by its actor across batches. Use AI Functions for common provider-backed operations; use custom UDFs when you need full control over model loading, batching, output schema, or provider behavior.

## Configuration model

Programmatic configuration:

```python
import vane

vane.configure(runner="ray")
```

Environment configuration:

```bash
export VANE_RUNNER=ray
```

`vane.configure(...)` applies registered `VANE_*` settings through the process environment. Use `vane.current_config()` or `vane.env.as_dict()` to inspect the public configuration snapshot.

## Worker environment

Workers need the runtime context required by the work they execute. In practice, confirm that workers can access:

- Python modules used by UDFs and AI providers.
- Object storage credentials and endpoint settings.
- Model files or model download locations.
- Relevant `VANE_*`, `AWS_*`, `S3FS_*`, `DUCKDB_*`, `PYTHONPATH`, and `RAY_ADDRESS` environment values.

AI provider clients are created in workers, so provider credentials must be available there. Prefer worker environment variables for API keys, tokens, and secrets; SQL AI options reject credential-like fields instead of embedding credentials in the plan.

Set the required environment before creating connections or running pipelines.

## Data movement

Common data paths:

- SQL reads local or remote data into DuckDB-compatible relations.
- Expression execution sends the required argument columns to a UDF or AI worker and combines the typed result with the surrounding projection.
- Relation methods stream Arrow batches into Python workers and return their declared table output. When downstream stages still need source columns, include those columns in the Relation output.
- Ray mode can materialize partitions through Ray workers.
- Writes use DuckDB-compatible and Vane Data relation write paths, depending on runner and target.

Place filters and column projections before expensive Python or model stages. This reduces the amount of data crossing into Python and, in Ray mode, reduces cluster data movement.
