---
title: Execution Model
---

Vane Data pipelines are lazy relation plans. Projection work begins with SQL Expression
by default. Python Expression is alternate syntax for the same typed
projection model, and Relation API handles specialized table-shaped stages. The
two top-level output models are Expression and Relation; SQL is an entry point,
not another model.

The execution boundaries follow the shape of the result, not whether the user
wrote SQL or Python.

## Relations and lazy plans frame execution

A relation represents a table-shaped computation. Creating one records work
without immediately reading every row or calling Python and AI workers:

```python
import vane

con = vane.connect()
source = con.sql("""
    SELECT id, text
    FROM (VALUES
        (1, '  Hello  '),
        (2, ' Vane Data ')
    ) AS t(id, text)
    WHERE text IS NOT NULL
""")
```

Each projection or table stage returns another relation. The lazy plan can
therefore retain filters, source columns, and Python or AI transformations until
a fetch or write consumer materializes it.

## SQL Expression is the default projection path

Expression API adds one typed result column in a `SELECT` projection and
normally preserves one output row for each input row. SQL is the recommended
entry point because the source fields and their derived values remain visible in
one relational statement.

### Registered UDF aliases

`vane.attach_function` gives a Python callable a SQL alias. The alias is then a
projection expression rather than a separate execution model:

```python
@vane.func(return_dtype="VARCHAR")
def normalize_text(value):
    return str(value).strip().lower()

vane.attach_function(
    normalize_text,
    alias="normalize_text_sql",
    connection=con,
    parameters=["VARCHAR"],
)

normalized = con.sql("""
    SELECT
        id,
        text,
        normalize_text_sql(text) AS normalized_text
    FROM source
""")
```

Registration associates the alias with its callable and type contract. Binding
the alias in `SELECT` records a typed projection in the lazy plan and lowers it
to the shared Expression execution path. It does not invoke the Python callable
while the query is being assembled.

### SQL AI functions

SQL `ai_prompt` and `ai_embed` follow the same projection path:

```python
ai_enriched = con.sql("""
    SELECT
        id,
        text,
        ai_prompt(text) AS response,
        ai_embed(text) AS embedding
    FROM source
""")
```

Binding each SQL AI call records one typed, row-preserving Expression result.
Provider configuration is planning data; a live client or model is not created
while the relation is being built. Registered SQL aliases remain row-preserving
in v1, and SQL AI calls use the same row-preserving projection boundary.

## Python Expression is alternate syntax

Python Expression builds the same typed projection model when the surrounding
composition is clearer in Python:

```python
python_enriched = source.select(
    vane.col("id"),
    vane.col("text"),
    normalize_text(vane.col("text")).alias("normalized_text"),
    vane.ai.prompt(vane.col("text")).alias("response"),
    vane.ai.embed(vane.col("text")).alias("embedding"),
)
```

Passing an Expression such as `vane.col("text")` builds lazy work; it does not
eagerly call the decorated function. Changing from the SQL entry point to this
Python entry point does not change typing, projection, materialization, or worker
semantics.

Python batch Expression has one explicit exception: `row_preserving=False` may
change cardinality, but it must be the sole top-level projection. It cannot sit
beside passthrough columns or another top-level result. Registered SQL aliases
cannot use this exception and remain row-preserving in v1.

## Relation API executes table-shaped stages

Use Relation API when a stage owns table shape: it returns multiple columns,
replaces the current table, or changes cardinality. It is also the model for
specialized AI work such as built-in chunking or classification,
structured/multimodal prompting, and explicit Relation execution controls.

For example, splitting one input row into multiple output rows is a Relation
operation:

```python
def split_terms(row):
    return [
        {"id": row["id"], "term": term}
        for term in str(row["text"]).split()
    ]

terms = source.flat_map(
    split_terms,
    schema={"id": "BIGINT", "term": "VARCHAR"},
    execution_backend="subprocess_task",
)
```

The public UDF methods are `rel.map_batches`, `rel.flat_map`, and `rel.map`;
`rel.map` is a row-wise scalar operation, not a pandas batch transform. The
specialized AI methods are `rel.prompt`, `rel.embed_text`, and
`rel.classify_text`. A Relation stage returns the columns and row count defined
by that stage instead of automatically retaining every input column.

Vane does not provide a direct SQL Relation or table-function API. SQL can
produce the input relation before a Python Relation call and consume its output
afterward, but the table-shaped method call itself is not a SQL projection.

## Materialization starts work

Building and composing relations remains lazy. Execution starts when a consumer
requests data or writes it, for example:

- `show()`
- `fetchall()`
- `to_arrow_table()`
- `to_record_batch()`
- `write_parquet(...)`
- `to_table(...)`

At materialization, Vane Data runs the accumulated relational operators and
schedules the Expression or Relation worker stages using the active runner. The
same lazy plan can therefore be inspected and narrowed before expensive Python
or provider work begins.

## Arrow batches connect SQL and Python

Arrow batches are the typed data boundary between relation execution and Python
workers. Registered aliases and SQL AI functions reach this boundary through
their shared Expression path; Python Expressions and Relation methods use it
directly.

- An Expression worker produces exactly one typed result column. Its normal
  row-preserving contract lets that column align with passthrough columns in the
  same projection.
- A Relation worker receives table-shaped input and returns the declared table
  shape. Its output may contain several columns or a different number of rows.
- Batch sizing changes scheduling and memory behavior, not which semantic model
  the operation belongs to.

Keeping schemas and identifiers explicit at this boundary makes SQL-to-Python
movement inspectable and lets later SQL stages consume ordinary relation
columns.

## Local execution

Local execution is the starting point for development, debugging, single-node
preprocessing, and validating a representative sample before scaling. The local
runner keeps relational work on the local machine and uses subprocess workers
for Python work:

- `subprocess_task` runs stateless functions.
- `subprocess_actor` reuses a callable class, model, or client across batches in
  one actor.

Provider-backed SQL and Python AI Expressions also resolve through the active
local runner and create their provider inside a subprocess actor when execution
actually reaches that stage.

## Ray execution

Select the Ray runner programmatically:

```python
import vane

vane.configure(runner="ray")
```

or through the environment:

```bash
export VANE_RUNNER=ray
```

Ray can distribute scans, writes, stateless UDFs with `ray_task`, and
actor-backed model work with `ray_actor`. It also places CPU and GPU requests
across the cluster. Moving a stage to Ray changes placement and parallelism; it
does not change its Expression or Relation output contract.

Every Ray worker that may run a stage must be able to import its Python modules,
reach its storage and provider endpoints, and access the required credentials or
model artifacts.

## Scheduling tasks, actors, and AI workers

Choose task or actor execution from callable lifecycle and resource needs:

| Workload | Callable or worker shape | Backend |
| --- | --- | --- |
| Simple local batch transform | Python function | `subprocess_task` |
| Local model or client reuse | Callable class | `subprocess_actor` |
| Distributed stateless transform | Python function | `ray_task` |
| Distributed model reuse or GPU inference | Callable class | `ray_actor` |

Tasks fit stateless work that does not rely on a long-lived instance. Actors fit
expensive initialization and resources that should be reused across batches.
Actor reuse is a lifecycle and throughput tool, not durable or shared state.

AI plans carry lightweight provider descriptors across the plan boundary. A
worker creates the real client or model lazily and reuses it within its actor.
Provider `concurrency` controls provider actors, while Relation `actor_number`
controls a Relation actor pool; exact defaults and restrictions belong in the
[AI Function API Reference](/docs/data/reference/ai-api) and
[UDF API Reference](/docs/data/reference/udf-api).

Stateful `vane.cls` Expression work has a narrower v1 contract: it uses exactly
one actor, and its mutable state is query-scoped, actor-local, and ephemeral.

## Streaming, blocking, and backpressure

Many scans, projections, and Relation stages can pass Arrow batches onward as
they become ready. Other stages must collect or coordinate data before they can
emit output. Common blocking or partially blocking work includes global
aggregations, exchange-heavy joins, sorts and shuffles, writes, and deliberate
UDF streaming boundaries.

Backpressure bounds admitted or in-flight batches and ready output. When a
downstream stage is slower, these limits make upstream work wait instead of
letting queued rows or objects grow without bound. Concurrency, batch size, and
backpressure therefore need to be tuned together; increasing worker count alone
does not remove a blocking boundary.

Place filters and column projections before expensive UDF and model calls. This
reduces Arrow data crossing into Python and, in Ray mode, reduces cluster data
movement. See [Performance Tuning](/docs/data/guides/performance-tuning) for the
operational workflow and the UDF reference for exact controls.

## Failure boundaries and external effects

Worker loss or a callable error can fail a query, and retry paths can execute a
call again. The query-scoped state of `vane.cls` is not checkpointed; losing its
single actor fails the query rather than recreating a replacement actor with
empty state. Distributed execution also provides no stable global row order for
state-dependent logic.

Provider requests and other external UDF effects are outside the SQL transaction
boundary and are not exactly-once. A rollback cannot undo a completed request or
write, and a retry can repeat it. Carry stable identifiers through projections,
keep results reviewable, and make external writes idempotent when repetition
matters. Credentials belong in worker environments because provider clients are
created there, not in the lazy plan.

For the full state, retry, credential, and side-effect contracts, read
[UDF Concepts](/docs/data/concepts/udfs),
[AI Function Concepts](/docs/data/concepts/ai-functions), and their API
references.

## Operational checklist

- Start with SQL Expression when one typed result belongs beside every input
  row. Use Python Expression for the same projection contract only when Python
  composition is clearer, and use Relation API for table-shaped or specialized
  work.
- Materialize a representative sample locally and verify output types, retained
  identifiers, and row counts before scaling.
- Filter rows and project unused columns away before Python or provider stages.
- Keep Expression result types and Relation output schemas explicit.
- Tune batch size with memory use, latency, streaming, and backpressure in mind.
- Use task backends for stateless work and actor backends only when reuse or
  resource placement is valuable.
- Move to Ray only when distribution, actor pools, GPUs, or cluster placement
  improve the workload.
- Confirm every worker can access storage, dependencies, credentials, and model
  artifacts.
- Measure throughput, memory, retries, external request counts, and output row
  counts after each execution change.
