Vane Data UDFs have two semantic output models: the **Expression API** and the **Relation API**. They are not three syntax surfaces. SQL and Python are entry points into the same Expression model; Relation is the separate model for table-shaped work.

This distinction is about what a query stage produces. Choose the model from the output contract, then choose syntax and execution controls.

## Expression API

The Expression API adds one typed output column to a `SELECT` projection and is normally row-preserving. SQL is the recommended default entry point because it keeps filtering, projection, source columns, and the UDF result visible in one relational statement.

### SQL Entry Point (Recommended)

Registering a Python callable with `vane.attach_function` gives the Expression API a SQL name; registration is a control for that model, not a third model. The registered alias then behaves as one projection expression. SQL aliases remain row-preserving in v1.

```python
import vane

con = vane.connect()
source = con.sql("""
    SELECT *
    FROM (VALUES
        (1, '  Hello  '),
        (2, ' Vane Data ')
    ) AS t(id, text)
""")

@vane.func(return_dtype="VARCHAR")
def normalize_text(value):
    return str(value).strip().lower()

vane.attach_function(
    normalize_text,
    alias="normalize_text_sql",
    connection=con,
    parameters=["VARCHAR"],
)
normalized = con.sql("""
    SELECT id, text, normalize_text_sql(text) AS normalized_text
    FROM source
""")
normalized.show()
vane.detach_function("normalize_text_sql", connection=con)
```

Keeping `id` and `text` in the projection makes the row-preserving contract visible and leaves ordinary SQL in charge of the surrounding table logic.

### Python Entry Point

Python Expression is alternate syntax for the same projection model. A call that receives `vane.col(...)` builds a lazy expression instead of eagerly invoking the Python function:

```python
projected = source.select(
    vane.col("id"),
    vane.col("text"),
    normalize_text(vane.col("text")).alias("normalized_text"),
)
```

The normal contract is one typed, row-preserving output column. Python batch Expression has one explicit exception: `row_preserving=False` may change cardinality, but that expression must be the sole top-level projection. It cannot be composed beside passthrough columns. This exception does not extend to registered SQL aliases, which remain row-preserving in v1.

## Relation API

Use the Relation API when the result is table-shaped: multi-column output, a complete table stage, or an explicit cardinality change. Current methods include `rel.map_batches`, `rel.flat_map`, and `rel.map`; `rel.map` is the row-wise scalar Relation API, not a pandas batch transform.

For example, token expansion belongs to Relation because one input row may produce several output rows:

```python
def split_terms(row):
    return [
        {"id": row["id"], "term": term}
        for term in str(row["text"]).split()
    ]

terms = source.flat_map(
    split_terms,
    schema={"id": "BIGINT", "term": "VARCHAR"},
    execution_backend="subprocess_task",
)
```

Vane exposes no direct SQL Relation or table-function API. SQL can prepare the input relation and consume a Relation result, but the cardinality-changing or table-shaped method call itself is currently Python Relation API.

## Shared Planning and Execution

Both models participate in the same lazy planning and execution layer. SQL aliases and Python expressions become planned projection work; Relation methods become planned table stages. Execution moves typed data through Arrow batches, so batching, worker placement, and resource choices do not change the semantic output model.

That shared layer is why the SQL and Python Expression entry points have the same projection and row-preservation rules. It is also why Relation is selected for its table contract, not simply because the callable happens to use Python.

## Actor Reuse and Mutable State

Actor-backed execution reuses a callable instance—and therefore a loaded model, client, or cache—within each actor. This is a lifecycle and throughput tool; multiple actors do not create shared state semantics.

`vane.cls` and `vane.cls.batch` provide the narrower mutable-state contract for Expression UDFs. In v1 they use `actor_number=1` so one actor instance can carry state across the batches of one query. General actor-backed callable reuse should not be described as durable application state.

## State, Failure, and Side Effects

Mutable `vane.cls` state is query-scoped, actor-local, and ephemeral. It is not checkpointed, is neither global state nor keyed state, and provides no exactly-once guarantee. For a stateful Expression UDF, actor loss makes the query fail rather than rebuilding an actor with empty state.

External effects performed by any UDF are not rolled back when the SQL transaction or query fails, and retries can repeat them. There is also no stable global row order across distributed relation execution. State-dependent logic should therefore avoid relying on input order, and external writes should use stable identifiers and idempotent handling.

## Choosing a Model

- Choose SQL Expression first when one typed result belongs beside existing rows.
- Choose Python Expression when Python composition is clearer but the same projection contract still fits.
- Choose Relation when the callable owns table shape, multiple columns, or cardinality.

For exact signatures and parameter rules, use the [UDF API Reference](/docs/data/reference/udf-api). For a complete task with setup, validation, and output, use the [Custom Python UDFs Guide](/docs/data/guides/custom-python-udfs).
