---
title: Sizing
---

Sizing Vane Data jobs is mostly about data volume, UDF cost, and model resources.

## Start with a small run

Record:

- input rows
- input bytes
- projected columns
- UDF backend
- batch size
- Relation `actor_number`, AI `concurrency`, and (for prompt providers) `max_api_concurrency`
- CPU/GPU resources
- output rows
- elapsed time

Then scale one variable at a time.

## Local CPU UDF

Use:

```python
execution_backend="subprocess_task"
```

Start with:

- `batch_size=512` for text cleaning
- `batch_size=32` for image/audio decoding

Adjust based on memory and latency.

## Local model actor

Use:

```python
execution_backend="subprocess_actor"
```

Start with:

- `actor_number=1`
- a batch size that keeps the model busy without memory pressure

Increase `actor_number` only if CPU/GPU resources remain idle.

## Ray task

Use:

```python
execution_backend="ray_task"
```

Good for CPU-heavy stateless work. Watch scheduling overhead if batches are too small.

## Ray actor

Use:

```python
execution_backend="ray_actor"
```

Good for actor-backed model inference.

Sizing rule:

```text
required GPUs = actor_number * gpus
```

If `actor_number=4` and `gpus=1`, the Ray cluster needs at least 4 available GPUs for full parallelism.

## Scan sizing

Relevant public settings:

- `ray_scan_task_size_grouping`
- `ray_max_task_backlog`
- `ray_scan_task_open_cost_bytes`
- `ray_scan_task_min_partition_num`

Use scan task grouping when many small files would otherwise create too many scan tasks.

## Output sizing

Embeddings can dominate output size.

Estimate:

```text
rows * dimensions * 4 bytes
```

For 10 million rows and 768 float32 dimensions, raw vector values alone are about 30 GB before file format overhead.

Relation actor pools use `actor_number`. Public AI provider option `concurrency` maps to the outer AI UDF actor count; for prompt providers, `max_api_concurrency` bounds in-flight provider requests inside each actor. Stateful `vane.cls` v1 always uses one actor and should not be sized as a parallel actor pool. See the [UDF API Reference](/docs/data/reference/udf-api) and [AI Function API Reference](/docs/data/reference/ai-api).

## Red flags

- One UDF batch contains unused large columns.
- Workers download model weights for every batch.
- Output is written to node-local disk from a distributed job.
- AI-backed jobs exceed provider rate limits through too many actors, too many in-flight requests per actor, or both.
- A single huge file prevents parallelism.

Fix projection, actor reuse, shared storage, provider limits, and partitioning before adding more machines.
