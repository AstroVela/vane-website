title: AI Functions
---

Vane Data is built for multimodal AI workloads. AI Functions let you run models over data: generating embeddings, classifying text, or calling large language and multimodal models.

For row-preserving work that adds one model output to each input row, start with the SQL Expression functions `ai_embed` and `ai_prompt`. Use the Python Expression functions `vane.ai.embed` and `vane.ai.prompt` when provider configuration is more convenient to construct in Python. Use the Relation helpers for specialized capabilities such as built-in text classification, long-text chunking, structured responses, or multimodal prompting.

AI Functions are a good starting point for common tasks such as embeddings, text classification, and prompting. Use [Embeddings at Scale](/docs/data/guides/embeddings-at-scale) when the work is mainly chunking and embedding many documents. Use [GPU Inference](/docs/data/guides/gpu-inference) for vLLM or GPU-backed batch prompting. Use [Custom Python UDFs](/docs/data/guides/custom-python-udfs) when you need full control over model loading, retries, output schema, preprocessing, or per-row error columns.

## Input Relation

Start with a relation that contains:

- A stable business ID, such as `document_id`, `ticket_id`, `case_id`, or `line_item_id`.
- The text column or fields needed by the model.
- Source metadata that helps review unusual outputs later, such as `source_uri`, `media_type`, `document_type`, or `created_at`.

```python
import vane

con = vane.connect()

source = con.sql("""
    select
        document_id,
        document_type,
        text,
        source_uri
    from read_parquet('data/documents/*.parquet')
    where text is not null
    order by document_id
    limit 100
""")

source.to_table("source_sample")
```

Keep the first sample small and ordered. Stable IDs let you validate AI output deterministically; ordering makes manual review repeatable but must not be used as a substitute for identity.

## Preserve IDs

SQL and Python AI Expressions are row-preserving and produce one output column. Keep IDs and source fields in the same `SELECT` or `rel.select(...)` projection as the AI Expression instead of materializing the model output separately and joining it back by row position.

After each Expression task, compare the stable IDs as well as the row count:

```python
def assert_same_ids(input_rel, output_rel):
    input_ids = sorted(row[0] for row in input_rel.project("document_id").fetchall())
    output_ids = sorted(row[0] for row in output_rel.project("document_id").fetchall())

    if output_ids != input_ids:
        raise ValueError("AI output IDs do not match source IDs")
```

The Relation helpers `embed_text(...)`, `classify_text(...)`, and the Relation form of `prompt(...)` return the configured output relation rather than an ordinary projection containing all source columns. Use them when that output shape fits the task. If the final result must also carry stable IDs and several source fields, use an Expression where one is available, or a custom row-preserving batch UDF that returns those fields in its declared schema. Do not reattach Relation output to source rows by position.

## Embed Text

Use `ai_embed` when each input row should get one vector. Because it is an Expression, the SQL projection can retain the stable ID and source metadata beside the embedding:

```python
embedded = con.sql("""
    select
        document_id,
        document_type,
        source_uri,
        text,
        ai_embed(
            text,
            struct_pack(
                provider := 'openai',
                model := 'text-embedding-3-small',
                dimensions := 512,
                normalize := true,
                concurrency := 4,
                batch_size := 64,
                on_error := 'raise'
            )
        ) as embedding
    from source_sample
    order by document_id
""")
```

The second argument to `ai_embed` is one constant options value; `struct_pack(...)` is the usual way to construct it. Setting `dimensions` also gives the output a fixed `FLOAT[512]` type.

Validation checks:

- Input column: filtered `text` should be non-null.
- Output column: `embedding` should exist, and vector length should match the model expectation.
- ID preservation: `document_id` should match the input IDs exactly.
- Result review: inspect a few rows to confirm empty or very short text was not embedded accidentally.

```python
assert_same_ids(source, embedded)
embedding_table = embedded.to_arrow_table()

if embedding_table.num_rows == 0:
    raise ValueError("empty embedding sample")

embeddings = embedding_table["embedding"].to_pylist()
if any(value is None or len(value) != 512 for value in embeddings):
    raise ValueError("embedding is null or has an unexpected dimension")
```

When the same typed configuration is easier to build in Python, use the equivalent Python Expression:

```python
embedded_python = source.select(
    vane.col("document_id"),
    vane.col("document_type"),
    vane.col("source_uri"),
    vane.col("text"),
    vane.ai.embed(
        vane.col("text"),
        provider="openai",
        model="text-embedding-3-small",
        provider_options=vane.ai.OpenAIProviderOptions(concurrency=4),
        embedding_options=vane.ai.OpenAIEmbeddingOptions(on_error="raise"),
        dimensions=512,
        normalize=True,
    ).alias("embedding"),
)
```

For long text, the Relation helper can split each input value into overlapping character chunks, embed the chunks, and combine them back into one vector:

```python
chunked_embedding_only = source.embed_text(
    "text",
    provider="transformers",
    model="sentence-transformers/all-MiniLM-L6-v2",
    max_chunk_chars=2000,
    chunk_overlap_chars=200,
    output_column="embedding",
    execution_backend="subprocess_actor",
)
```

This is an output-only Relation stage. If every chunk needs its own stable ID and row, use [Embeddings at Scale](/docs/data/guides/embeddings-at-scale) instead.

## Classify Text

Use the Relation helper `classify_text` when the label set is small and each input row should get one zero-shot label. There is no SQL or Python Expression form of this helper.

```python
label_only = source.classify_text(
    "text",
    labels=["invoice", "claim", "memo", "other"],
    provider="transformers",
    model="facebook/bart-large-mnli",
    output_column="ai_label",
    batch_size=16,
    execution_backend="subprocess_actor",
)
```

Validation checks:

- Input column: `text` should contain the text you want the classifier to see.
- Output column: `ai_label` should contain only one of the labels you provided.
- Cardinality: output row count should match input row count.
- Result review: sample by label and check whether the label set is too broad, too narrow, or missing business categories.

```python
label_table = label_only.to_arrow_table()
source_table = source.to_arrow_table()

if label_table.num_rows != source_table.num_rows:
    raise ValueError("classification row count mismatch")

labels = set(label_table["ai_label"].to_pylist())
allowed = {"invoice", "claim", "memo", "other"}
unexpected = labels - allowed
if unexpected:
    raise ValueError(f"unexpected labels: {[str(x) for x in unexpected]}")
```

Because this Relation helper returns the label output, do not join it back to `source` by row position. When the published classification artifact must contain IDs, source fields, labels, and per-row errors together, use a custom row-preserving batch UDF with an explicit multi-column schema.

## Prompt

Use `ai_prompt` when output is generated from a row-level prompt. Keep the first prompt narrow and easy to review.

```python
prompts = con.sql("""
    select
        document_id,
        document_type,
        source_uri,
        'Summarize this document in one sentence: ' || text as review_prompt
    from source_sample
    order by document_id
""")

prompts.to_table("review_prompts")

reviewed = con.sql("""
    select
        document_id,
        document_type,
        source_uri,
        review_prompt,
        ai_prompt(
            review_prompt,
            struct_pack(
                provider := 'openai',
                model := 'gpt-4o-mini',
                concurrency := 4,
                max_api_concurrency := 4,
                batch_size := 8,
                system_message := 'Return concise business review notes.',
                temperature := 0.0,
                on_error := 'raise'
            )
        ) as review_note
    from review_prompts
    order by document_id
""")
```

Validation checks:

- Input column: `review_prompt` should contain only fields the model should see.
- Output column: `review_note` should be non-empty and short enough to review.
- ID preservation: `document_id`, `document_type`, and `source_uri` remain beside the output.
- Result review: manually inspect a small sample before increasing `limit`, `batch_size`, or `concurrency`.

```python
assert_same_ids(prompts, reviewed)
review_table = reviewed.to_arrow_table()

empty = [value for value in review_table["review_note"].to_pylist() if not value]
if empty:
    raise ValueError("prompt returned empty responses")
```

The equivalent Python Expression accepts typed provider and prompt options while preserving the same one-column projection model:

```python
reviewed_python = prompts.select(
    vane.col("document_id"),
    vane.col("document_type"),
    vane.col("source_uri"),
    vane.col("review_prompt"),
    vane.ai.prompt(
        vane.col("review_prompt"),
        provider="openai",
        model="gpt-4o-mini",
        provider_options=vane.ai.OpenAIProviderOptions(
            concurrency=4,
            max_api_concurrency=4,
        ),
        prompt_options=vane.ai.OpenAIPromptOptions(
            temperature=0,
            on_error="raise",
        ),
        system_message="Return concise business review notes.",
    ).alias("review_note"),
)
```

Use the Relation form `prompts.prompt(...)` only when the task needs Relation-only options such as `return_format`, `image_columns`, `use_chat_completions`, or an explicit `execution_backend`. It returns the configured output relation, so plan stable-key preservation before using it. Expression calls do not accept those Relation-only options; rename an Expression result with `.alias(...)`.

## Provider and Execution Choices

Choose the smallest provider setup that answers the current task.

| Choice | Use when | Watch |
| --- | --- | --- |
| Hosted provider | You need a managed model and can use external API calls. | Credentials, cost, latency, rate limits, and provider-specific output behavior. |
| Local model | You need local execution through installed model libraries. | Model download, dependency versions, memory use, and worker initialization time. |
| vLLM or GPU model | You need GPU-backed prompt throughput or private LLM inference. | vLLM paths and GPU-specific tuning are covered in [GPU Inference](/docs/data/guides/gpu-inference). |

Minimum settings to make explicit:

- `provider` and `model`: record the exact provider and model used for each run.
- Provider credentials: set them through environment variables in every worker environment; do not put API keys or other credentials in SQL options.
- SQL options: pass one foldable constant options value, normally `struct_pack(...)`; configuration cannot vary by row.
- Python options: use typed provider, prompt, and embedding option objects when that makes configuration clearer.
- `batch_size` and `concurrency`: increase them only after the small sample is correct; larger values can increase memory use or hit provider rate limits.
- Execution: Expression AI Functions follow the active Vane runner. Relation helpers can additionally select `execution_backend` explicitly when needed.
- Errors and retries: choose `on_error` deliberately and validate null, empty, or malformed model output. Provider calls are external side effects, so a query retry can issue another request and incur additional cost.
