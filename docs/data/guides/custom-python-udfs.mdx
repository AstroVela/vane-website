---
title: Custom Python UDFs
---

Use Custom Python UDFs when a Vane Data pipeline needs Python logic that does not fit cleanly in SQL or a built-in AI Function: external libraries, custom rules, media processing, model preprocessing, custom inference, private clients, or a fully controlled output schema.

This guide focuses on the UDF types you can use and how to choose between them. For one typed result beside existing columns, prefer an Expression UDF: register a SQL alias for the recommended SQL workflow, or build the same projection with the Python Expression API. Use the Relation API when the UDF owns a multi-column or row-changing table stage.

## Choose a UDF Type

Choose the output shape first, then the callable shape. The execution backend is a separate choice that controls where the callable runs.

| UDF type | Preferred API | Callable shape | Use when |
| --- | --- | --- | --- |
| Stateless scalar Expression UDF | Registered SQL alias; Python `vane.func` when constructing expressions in Python | Scalar function | Add one typed, row-preserving value beside existing columns. |
| Stateless batch Expression UDF | Registered SQL alias; Python `vane.func.batch` when constructing expressions in Python | Arrow function | Batching helps, but the result is still one typed column. |
| Stateful Expression UDF | Registered SQL alias; Python `vane.cls` or `vane.cls.batch` | Callable class | One typed result needs query-scoped actor state. |
| Stateless batch Relation UDF | `map_batches` | Python function | Return a multi-column or whole-table result without reusable state. |
| Stateful batch Relation UDF | `map_batches` | Callable class | Return a table-shaped result while reusing expensive setup across batches. |
| Row-expanding Relation UDF | `flat_map` | Row function | One input row can produce zero, one, or many output rows. |
| Scalar Relation UDF | `map` | Scalar function | An explicit Relation executor and its pass-through result shape are required. |

For a normal row-preserving enrichment, start with a connection-scoped SQL alias. Filtering, joining, and source-column selection remain ordinary SQL, while the Python callable contributes one typed projection value:

```python
import vane

con = vane.connect()


def normalize_document_type(value):
    return str(value or "unknown").strip().lower()


vane.attach_function(
    normalize_document_type,
    alias="normalize_document_type_sql",
    connection=con,
    parameters=["VARCHAR"],
    return_dtype="VARCHAR",
)

try:
    normalized = con.sql("""
        select
            document_id,
            normalize_document_type_sql(document_type) as document_type
        from (values
            ('doc-001', ' Invoice '),
            ('doc-002', null)
        ) as documents(document_id, document_type)
    """)
    normalized.show()
finally:
    vane.detach_function("normalize_document_type_sql", connection=con)
```

The alias is scoped to `con` and is supported in `SELECT` projections. Apply filters, joins, and grouping before the UDF projection. Use `vane.func`, `vane.func.batch`, `vane.cls`, or `vane.cls.batch` instead when the projection needs to be assembled as Python Expression objects. See the [UDF API Reference](/docs/data/reference/udf-api) for registration signatures and placement restrictions.

## Stateless Batch UDFs

Use a stateless batch UDF when each batch can be processed independently:

- text cleanup
- rule-based labeling
- metadata checks
- lightweight parsing
- deterministic feature extraction
- Python library calls that do not need reusable state

A stateless batch UDF is a Python function that receives a `pyarrow.Table` and returns a `pyarrow.Table`. If it produces one typed column and preserves input row count, it can be registered as a SQL Expression alias or used through Python `vane.func.batch(..., row_preserving=True)`. The example below returns source fields, diagnostics, and several derived columns together, so the table-shaped `map_batches` Relation API is the better fit.

```python
import re

import pyarrow as pa
import vane

con = vane.connect()

documents = con.sql("""
    select
        cast(document_id as varchar) as document_id,
        source_uri,
        media_type,
        document_type,
        text
    from read_parquet('data/documents/*.parquet')
    where text is not null
    order by document_id
    limit 100
""")

SUPPORTED_MEDIA_TYPES = {"application/pdf", "text/plain", "text/html"}
RISK_TERMS = ("missing signature", "expired", "redacted incorrectly")


def prepare_document_text(batch: pa.Table) -> pa.Table:
    document_ids = batch.column("document_id").to_pylist()
    source_uris = batch.column("source_uri").to_pylist()
    media_types = batch.column("media_type").to_pylist()
    document_types = batch.column("document_type").to_pylist()
    texts = batch.column("text").to_pylist()

    normalized_text = []
    text_length = []
    needs_review = []
    review_reason = []
    status = []
    error = []

    for media_type, text in zip(media_types, texts, strict=False):
        try:
            clean = re.sub(r"\s+", " ", str(text or "")).strip()
            lower = clean.lower()

            normalized_text.append(clean)
            text_length.append(len(clean))

            if media_type not in SUPPORTED_MEDIA_TYPES:
                needs_review.append(True)
                review_reason.append("unsupported_media_type")
            elif not clean:
                needs_review.append(True)
                review_reason.append("empty_text")
            elif any(term in lower for term in RISK_TERMS):
                needs_review.append(True)
                review_reason.append("risk_term")
            else:
                needs_review.append(False)
                review_reason.append("none")
            status.append("ok")
            error.append("")
        except Exception as exc:
            normalized_text.append("")
            text_length.append(0)
            needs_review.append(True)
            review_reason.append("udf_error")
            status.append("failed")
            error.append(f"prepare_document_text: {type(exc).__name__}: {exc}")

    return pa.table({
        "document_id": document_ids,
        "source_uri": source_uris,
        "media_type": media_types,
        "document_type": document_types,
        "normalized_text": normalized_text,
        "text_length": text_length,
        "needs_review": needs_review,
        "review_reason": review_reason,
        "status": status,
        "error": error,
    })


prepared_schema = {
    "document_id": "VARCHAR",
    "source_uri": "VARCHAR",
    "media_type": "VARCHAR",
    "document_type": "VARCHAR",
    "normalized_text": "VARCHAR",
    "text_length": "BIGINT",
    "needs_review": "BOOLEAN",
    "review_reason": "VARCHAR",
    "status": "VARCHAR",
    "error": "VARCHAR",
}

prepared = documents.map_batches(
    prepare_document_text,
    schema=prepared_schema,
    batch_size=64,
    execution_backend="subprocess_task",
)

prepared.show()
```

The returned table contains both new fields and the source fields needed by later stages. `map_batches` returns the columns produced by the UDF; it does not automatically preserve upstream columns.

For comparison, a row-preserving Python batch Expression names its input columns explicitly and declares exactly one output column:

```python
def normalize_text_batch(table: pa.Table) -> pa.Table:
    values = table.column("text").to_pylist()
    return pa.table({
        "normalized_text": [
            None if value is None else str(value).strip().lower()
            for value in values
        ]
    })


normalized_text = vane.func.batch(
    normalize_text_batch,
    inputs={"text": vane.col("text")},
    schema={"normalized_text": "VARCHAR"},
    row_preserving=True,
).alias("normalized_text")

normalized = documents.select(
    vane.col("document_id"),
    normalized_text,
)
```

Use the registered SQL form when this projection belongs in a SQL pipeline; use this Python form when expressions are assembled dynamically.

Validate the stateless UDF before increasing the batch size or moving to Ray:

```python
prepared.to_table("prepared_documents")

summary = con.sql("""
    select
        document_type,
        media_type,
        needs_review,
        review_reason,
        count(*) as rows
    from prepared_documents
    group by document_type, media_type, needs_review, review_reason
    order by document_type, media_type, needs_review, review_reason
""")

summary.show()
```

After the local output is correct, the same function can run with `ray_task`:

```python
import vane

vane.configure(runner="ray")

ray_prepared = documents.map_batches(
    prepare_document_text,
    schema=prepared_schema,
    batch_size=256,
    execution_backend="ray_task",
)
```

Use `ray_task` only after the local contract is correct. Ray adds worker import, dependency, storage, and resource variables that can hide ordinary UDF bugs.

## Stateful Batch UDFs

Use a stateful batch UDF when setup should be reused across batches:

- model or tokenizer loading
- API client setup
- database or object-store client setup
- lookup tables or caches
- GPU model state

A stateful UDF is a callable class. For one typed, row-preserving Expression result, use `vane.cls` or `vane.cls.batch`, instantiate the decorated class, and either attach that instance as a SQL alias or call it while building a Python projection. The v1 Expression contract uses one actor per query and treats its state as query-scoped and ephemeral.

The example below returns a multi-column table, so it uses the Relation form instead: pass the class itself to `map_batches`, not an already-created instance. Actor backends construct the class inside the worker and reuse the same object for multiple batches.

Actor classes should have no required constructor arguments. Put configuration in environment variables, class-level defaults, optional arguments with defaults, or a no-argument wrapper class.

```python
import pyarrow as pa


class PromptBuilder:
    def __init__(self) -> None:
        self.template_by_type = {
            "claim": "Extract claim decision and one short reason:",
            "invoice": "Extract invoice number, supplier, and total:",
            "policy": "Summarize policy issue and required follow-up:",
        }

    def __call__(self, batch: pa.Table) -> pa.Table:
        document_ids = batch.column("document_id").to_pylist()
        source_uris = batch.column("source_uri").to_pylist()
        document_types = batch.column("document_type").to_pylist()
        texts = batch.column("normalized_text").to_pylist()
        upstream_status = batch.column("status").to_pylist()
        upstream_error = batch.column("error").to_pylist()

        prompts = []
        prompt_template = []
        status = []
        error = []

        for document_type, text, previous_status, previous_error in zip(
            document_types,
            texts,
            upstream_status,
            upstream_error,
            strict=False,
        ):
            if previous_status != "ok":
                prompt_template.append("")
                prompts.append("")
                status.append("skipped")
                error.append(previous_error)
                continue

            template = self.template_by_type.get(document_type, "Summarize this document:")
            prompt_template.append(template)
            prompts.append(f"{template}\n\n{text}")
            status.append("ok")
            error.append("")

        return pa.table({
            "document_id": document_ids,
            "source_uri": source_uris,
            "prompt_template": prompt_template,
            "prompt": prompts,
            "status": status,
            "error": error,
        })


prompt_schema = {
    "document_id": "VARCHAR",
    "source_uri": "VARCHAR",
    "prompt_template": "VARCHAR",
    "prompt": "VARCHAR",
    "status": "VARCHAR",
    "error": "VARCHAR",
}

prompts = prepared.map_batches(
    PromptBuilder,
    schema=prompt_schema,
    batch_size=64,
    execution_backend="subprocess_actor",
    actor_number=2,
)
```

In a real model UDF, load the model, tokenizer, or client in `__init__` or lazily on first call, not at module import time. That keeps imports cheap and lets each subprocess or Ray actor own its own state.

For distributed actor reuse, switch the backend to `ray_actor`:

```python
ray_prompts = prepared.map_batches(
    PromptBuilder,
    schema=prompt_schema,
    batch_size=128,
    execution_backend="ray_actor",
    actor_number=4,
)
```

For GPU-backed state, request GPU resources per actor:

```python
class GpuEmbedder:
    def __init__(self) -> None:
        # Load the tokenizer/model here so each actor owns its GPU state.
        self.model_name = "local-or-mounted-model"

    def __call__(self, batch: pa.Table) -> pa.Table:
        document_ids = batch.column("document_id").to_pylist()
        source_uris = batch.column("source_uri").to_pylist()
        texts = batch.column("normalized_text").to_pylist()

        embeddings = [[0.0, 0.0, 0.0] for _ in texts]
        return pa.table({
            "document_id": document_ids,
            "source_uri": source_uris,
            "embedding": embeddings,
            "status": ["ok"] * len(document_ids),
            "error": [""] * len(document_ids),
        })


embedded = prepared.map_batches(
    GpuEmbedder,
    schema={
        "document_id": "VARCHAR",
        "source_uri": "VARCHAR",
        "embedding": "FLOAT[]",
        "status": "VARCHAR",
        "error": "VARCHAR",
    },
    batch_size=32,
    execution_backend="ray_actor",
    actor_number=2,
    cpus=0.0,
    gpus=1.0,
)
```

`gpus=1.0` is per actor. With `actor_number=2`, Ray needs two available GPU slots for full parallelism. For more GPU execution patterns, see [GPU Inference](/docs/data/guides/gpu-inference).

Every Ray worker must be able to import the callable and access the same data paths, model files, credentials, and Python dependencies.

## Row-Expanding UDFs

Use `flat_map` when one input row can produce zero, one, or many output rows. Typical uses include document chunking, frame extraction summaries, nested record normalization, URL expansion, and filtering by returning no rows.

Python `vane.func.batch(..., row_preserving=False)` can produce a cardinality-changing batch Expression, but it must be the sole top-level projection and cannot be attached as a SQL alias in v1. Prefer `flat_map` when the transformation is naturally expressed as explicit zero-or-many rows.

The callable receives one row as a Python `dict`. It may return `None`, one output `dict`, or an iterable of output dicts.

```python
def chunk_document(row: dict) -> list[dict]:
    text = str(row.get("normalized_text") or "")
    chunks = [text[index:index + 1000] for index in range(0, len(text), 1000)]

    return [
        {
            "document_id": row["document_id"],
            "chunk_id": f"{row['document_id']}:{position}",
            "chunk_text": chunk,
        }
        for position, chunk in enumerate(chunks)
        if chunk
    ]


chunks = prepared.flat_map(
    chunk_document,
    schema={
        "document_id": "VARCHAR",
        "chunk_id": "VARCHAR",
        "chunk_text": "VARCHAR",
    },
    execution_backend="subprocess_task",
)
```

The declared schema selects keys from each output dict. Missing keys become null, and extra keys are ignored.

## Scalar UDFs

For a small row-wise value beside existing columns, prefer the registered SQL scalar Expression shown above; use Python `vane.func` when the projection is constructed in Python. Use Relation `map` only when you specifically need an explicit Relation executor and its pass-through result shape.

```python
def normalize_document_type(document_type: str) -> str:
    return str(document_type or "unknown").strip().lower()


normalized_types = documents.project("document_type").map(
    normalize_document_type,
    return_type="VARCHAR",
    execution_backend="subprocess_task",
)
```

For standard embeddings or LLM calls, prefer the built-in AI Functions. For custom model calls, media decoding, or any operation that benefits from batching, choose a batch Expression for one typed row-preserving result or `map_batches` for a table-shaped result.

## Shared Output Contract

Treat every UDF output as an explicit contract. An Expression UDF contributes one declared result column to a projection; a Relation UDF owns the table shape it returns.

| Topic | Guidance |
| --- | --- |
| Returned columns | An Expression batch schema declares exactly one output column. For Relation `map_batches` and `flat_map`, return exactly the columns declared in `schema`. |
| Source fields | Select source fields beside an Expression result. Return them explicitly from a Relation UDF when later stages need them. |
| Row diagnostics | Return explicit `status` and `error` columns when failed rows should remain reviewable. |
| Types | Use Arrow-compatible values: strings, numbers, booleans, bytes, lists, structs, or Arrow arrays. |
| Nulls | Keep null behavior explicit, especially for numeric, bytes, list, and model-output columns. |
| Row count | A row-preserving Expression must return one output row per input row. A Relation `map_batches` UDF usually does the same; use `flat_map` for explicit zero-or-many outputs. |

If Arrow cannot infer the type you want, build an explicit array:

```python
return pa.table({
    "document_id": pa.array(document_ids, type=pa.string()),
    "review_reason": pa.array(review_reason, type=pa.string()),
})
```

Avoid returning Python objects such as model classes, file handles, custom dataclasses, or raw exceptions. Convert them to strings, numbers, bytes, lists, structs, JSON strings, or explicit Arrow arrays before returning.

## Backend and Scaling

Expression UDFs derive their execution shape from the callable: `vane.func` and `vane.func.batch` use tasks, while `vane.cls` and `vane.cls.batch` use actors. Configure the Vane runner to choose local or Ray execution. Relation UDFs expose `execution_backend` directly; choose it according to callable shape:

| Backend | Callable shape | Use it for |
| --- | --- | --- |
| `subprocess_task` | Function | First stateless UDFs, local isolation, CPU transforms. |
| `subprocess_actor` | Callable class | Local model or client reuse without Ray. |
| `ray_task` | Function | Distributed stateless batch work after local correctness is proven. |
| `ray_actor` | Callable class | Distributed model/client reuse, GPU placement, or larger actor pools. |

The only valid backend names are `subprocess_task`, `subprocess_actor`, `ray_task`, and `ray_actor`. Older or shorthand options such as `local`, `subprocess`, `ray=True`, `use_process`, `actor_count`, and `prefer_actor` are not the current API.

Use this progression:

1. Build the UDF locally with `subprocess_task` or `subprocess_actor`.
2. Validate schema, row counts, IDs, source columns, and error behavior on a small real sample.
3. Increase `batch_size` locally after the output is correct.
4. Move stateless functions to `ray_task` only when distribution helps.
5. Move stateful classes to `ray_actor` only when actor reuse, GPU placement, or cluster scheduling is required.

`batch_size` controls how many input rows Vane passes to one `map_batches` call. Batch size should not change result semantics. If output changes when `batch_size` changes, the UDF is probably relying on hidden batch state.

Use `actor_number` to size `subprocess_actor` and `ray_actor` pools. Task backends do not accept this actor-pool option.

## Related Guides

- Use [Structured Transformation](/docs/data/guides/structured-transformation) when the main task is table-shaped cleanup, joins, deduplication, validation, or output shaping.
- Use [Multimodal Pipeline](/docs/data/guides/multimodal-pipeline) when the UDF reads bytes, decodes files, extracts media features, or records per-stage status and errors.
