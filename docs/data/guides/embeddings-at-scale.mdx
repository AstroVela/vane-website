---
title: Embeddings at Scale
---

Use Vane Data for embedding pipelines when you want SQL filtering, batch model execution, and table outputs in one workflow.

The usual shape is:

1. Read and filter source rows with SQL.
2. Chunk long text into stable units.
3. Embed chunks with SQL `ai_embed(...)`, or use the Python Expression API when the configuration belongs in Python.
4. Write embeddings with stable document and chunk IDs.

## Basic text embedding

The recommended entry point is the SQL Expression API. `ai_embed` adds one embedding value per input row, so identifiers and source fields can stay in the same projection:

```python
import vane

con = vane.connect()
embedded = con.sql("""
    SELECT
        id,
        text,
        ai_embed(
            text,
            struct_pack(
                provider := 'transformers',
                model := 'sentence-transformers/all-MiniLM-L6-v2'
            )
        ) AS embedding
    FROM read_parquet('data/documents/*.parquet')
    WHERE text IS NOT NULL
""")
```

`ai_embed` is evaluated in a `SELECT` projection and can be combined with ordinary columns. Without `dimensions`, its result type is `FLOAT[]`. The options argument is a constant `STRUCT` for the whole task, not a set of per-row values.

## Chunk long text first

Do not send whole documents to an embedding model unless that is intentional. Create bounded chunks with stable IDs.

```python
import pyarrow as pa

def chunk_rows(batch: pa.Table) -> pa.Table:
    doc_ids = batch.column("id").to_pylist()
    texts = batch.column("text").to_pylist()
    out_doc_id = []
    out_chunk_id = []
    out_text = []

    for doc_id, text in zip(doc_ids, texts):
        parts = [part.strip() for part in str(text).split(".") if part.strip()]
        for idx, part in enumerate(parts):
            out_doc_id.append(doc_id)
            out_chunk_id.append(idx)
            out_text.append(part)

    return pa.table({
        "id": out_doc_id,
        "chunk_id": out_chunk_id,
        "chunk_text": out_text,
    })

rel = con.sql("""
    SELECT id, text
    FROM read_parquet('data/documents/*.parquet')
    WHERE text IS NOT NULL
""")

chunks = rel.map_batches(
    chunk_rows,
    schema={
        "id": "BIGINT",
        "chunk_id": "BIGINT",
        "chunk_text": "VARCHAR",
    },
    batch_size=512,
    execution_backend="subprocess_task",
)
chunks.to_table("document_chunks")
con.sql("SELECT * FROM document_chunks").write_parquet("data/document-chunks.parquet")
```

This step uses the Relation API because one document can produce multiple output rows. The Parquet copy provides the input used by the isolated Ray example below. For production systems, use a tokenizer-aware chunker that matches the model context window and preserves enough overlap for retrieval quality.

## Use a local embedding model

Local models are usually best with an actor backend, because model initialization is expensive. SQL `ai_embed` creates that execution path while keeping the source columns in the projection:

```python
embedded_local = con.sql("""
    SELECT
        id,
        chunk_id,
        chunk_text,
        ai_embed(
            chunk_text,
            struct_pack(
                provider := 'transformers',
                model := 'sentence-transformers/all-MiniLM-L6-v2',
                batch_size := 64
            )
        ) AS embedding
    FROM document_chunks
""")
```

Use the Python Expression API as an alternative when typed configuration or Python composition is more convenient:

```python
embedded_python = chunks.select(
    vane.col("id"),
    vane.col("chunk_id"),
    vane.col("chunk_text"),
    vane.ai.embed(
        vane.col("chunk_text"),
        provider="transformers",
        model="sentence-transformers/all-MiniLM-L6-v2",
    ).alias("embedding"),
)
```

Both forms are row-preserving Expression operations. When the provider detects usable GPU resources, it may request a GPU from the UDF execution layer.

## Use hosted embedding APIs

Hosted providers are useful when you want managed models and can tolerate API latency and rate limits. The SQL form remains the preferred entry point:

```python
embedded_hosted = con.sql("""
    SELECT
        id,
        chunk_id,
        chunk_text,
        ai_embed(
            chunk_text,
            struct_pack(
                provider := 'openai',
                model := 'text-embedding-3-small',
                dimensions := 512,
                normalize := true,
                batch_size := 64,
                concurrency := 8
            )
        ) AS embedding
    FROM document_chunks
""")
```

Set credentials such as `OPENAI_API_KEY` in every worker environment; credential-like fields are rejected in SQL options. `concurrency` controls the number of AI actors, while request-level concurrency depends on the provider. Increase either only after measuring rate limits, retries, and cost on a bounded input.

## Use Ray for larger jobs

Move to Ray after the local path is correct. Configure the runner before creating the connection or input relation; the SQL Expression itself does not change:

```python
import vane

vane.configure(runner="ray")
con = vane.connect()

embedded_ray = con.sql("""
    SELECT
        id,
        chunk_id,
        chunk_text,
        ai_embed(
            chunk_text,
            struct_pack(
                provider := 'transformers',
                model := 'sentence-transformers/all-MiniLM-L6-v2',
                batch_size := 64
            )
        ) AS embedding
    FROM read_parquet('data/document-chunks.parquet')
""")
```

Every Ray worker must be able to import the provider libraries and access model files or provider credentials. Local subprocess execution inherits visible devices but does not reserve or isolate GPUs; use Ray when explicit resource scheduling is required.

## Output shape

Write a complete relation that makes downstream retrieval and auditing straightforward:

- `document_id`
- `chunk_id`
- `chunk_text`
- `embedding`
- `source_uri`
- model name and provider
- creation timestamp or run ID when needed

Because the Expression API keeps identifiers and embeddings in one projection, the result can be written directly after validation:

```python
embedded_hosted.write_parquet("output/embeddings/part-00000.parquet")
```

Before writing, validate row counts, stable IDs, nulls, and vector dimensions. For distributed writes, ensure every worker can access the output filesystem.

## Tuning checklist

- Filter and project before embedding.
- Keep chunks bounded and deterministic.
- Use actor execution for local models.
- Increase `batch_size` until latency or memory becomes the bottleneck.
- Increase actor or provider request concurrency only when workers and API quota have headroom.
- Use Ray when scan, inference, or write throughput exceeds one process, or when resource scheduling is required.
- Keep output rows deterministic with stable document and chunk IDs.
- Set fixed `dimensions` only when the provider and model support that vector size.
