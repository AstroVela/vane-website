This guide completes one reviewable vLLM text-generation task with SQL `ai_prompt` first. It then shows the equivalent Python Expression construction and reserves the specialized Relation helper for advanced capabilities at the end.

Use the [AI Function API Reference](/docs/data/reference/ai-api) for exact signatures, option fields, and return types. Use [AI Function Concepts](/docs/data/concepts/ai-functions) and the [Execution Model](/docs/data/concepts/execution-model) for the reasons behind the Expression/Relation boundary and worker lifecycle.

The task keeps this contract from source to output:

```text
select a keyed candidate slice -> generate one VARCHAR per row -> retain IDs and source -> validate -> review -> write
```

Use this guide when prompts should run against a model on the local machine or in your Ray environment, model initialization should be reused across batches, or GPU scheduling and prompt locality need deliberate validation. For arbitrary GPU model code or a source-preserving multi-column result schema, use [Custom Python UDFs](/docs/data/guides/custom-python-udfs).

## 1. Prepare a Small, Keyed Candidate Slice

Start with real rows, a deterministic order, stable business IDs, and the source fields needed to reproduce an output. Keep the first run small enough to inspect by hand.

```python
import vane

con = vane.connect()

source = con.sql(
    """
    SELECT
        CAST(case_id AS VARCHAR) AS case_id,
        CAST(document_id AS VARCHAR) AS document_id,
        CAST(document_type AS VARCHAR) AS document_type,
        CAST(source_uri AS VARCHAR) AS source_uri,
        CAST(text AS VARCHAR) AS source_text
    FROM read_parquet('data/claims/documents/*.parquet')
    """
)
source.to_table("gpu_inference_source")

candidates = con.sql(
    """
    SELECT
        'claims-vllm-2026-07-11-001'::VARCHAR AS run_id,
        case_id,
        document_id,
        document_type,
        source_uri,
        source_text,
        (
            'Extract the claim decision and one short factual reason: '
            || source_text
        )::VARCHAR AS prompt
    FROM gpu_inference_source
    WHERE case_id IS NOT NULL
      AND document_id IS NOT NULL
      AND document_type IS NOT NULL
      AND source_uri IS NOT NULL
      AND source_text IS NOT NULL
      AND trim(source_text) <> ''
    ORDER BY document_type, case_id, document_id
    LIMIT 50
    """
)
candidates.to_table("gpu_inference_candidates")

expected_keys = con.sql(
    """
    SELECT case_id, document_id
    FROM gpu_inference_candidates
    ORDER BY document_type, case_id, document_id
    """
).fetchall()

if not expected_keys:
    raise ValueError("GPU inference candidate slice is empty")
if len(set(expected_keys)) != len(expected_keys):
    raise ValueError("(case_id, document_id) must be unique")
```

`run_id` identifies this configuration and output path; `(case_id, document_id)` identifies the business row. Never use output order as identity. If the prompt template, model, or material engine settings change, use a new run ID or record an explicit version.

## 2. Run Basic Local vLLM Generation with SQL

The recommended first prompt call is a top-level SQL projection. The constant `struct_pack` selects vLLM and supplies only non-secret, query-wide configuration. The two JSON bridge fields are strings containing valid JSON; they become the provider's `engine_args` and `generate_args` dictionaries at bind time.

```python
local_prompted = con.sql(
    """
    SELECT
        run_id,
        case_id,
        document_id,
        document_type,
        source_uri,
        source_text,
        prompt,
        ai_prompt(
            prompt,
            struct_pack(
                provider := 'vllm',
                model := 'Qwen/Qwen3-1.7B',
                concurrency := 1,
                batch_size := 8,
                gpus_per_actor := 1.0,
                max_retries := 0,
                on_error := 'log',
                system_message := 'Return one concise factual business note.',
                engine_args_json := '{"max_model_len": 2048}',
                generate_args_json := '{"sampling_params": {"temperature": 0.0, "max_tokens": 128}}'
            )
        ) AS response_text
    FROM gpu_inference_candidates
    ORDER BY document_type, case_id, document_id
    """
)
local_prompted.to_table("gpu_inference_model_output")
```

`ai_prompt` returns one `VARCHAR` beside each candidate row. Its options must be foldable constants; they cannot vary by row. SQL credential-like keys are rejected recursively, including inside the JSON bridges. A local vLLM run normally needs no hosted-provider API key. If model download requires a gated registry token, place it in the worker environment rather than SQL.

With the default local runner, Vane creates a subprocess actor and initializes the real vLLM engine lazily on that worker. The actor reuses the engine for later batches in the same query. The nonzero `gpus_per_actor` value describes the intended resource, but local subprocess execution only inherits process-visible devices: it does not reserve or isolate a GPU and emits a warning for a nonzero GPU request. Set device visibility before starting the process and verify it independently.

Before materializing this query, the local worker must have a compatible Vane, vLLM, PyTorch, CUDA, and driver stack. Pre-stage the exact model/tokenizer artifacts when workers should run without outbound network access; otherwise allow only the artifact endpoint the model load requires. Treat `trust_remote_code` or a mutable model revision as a separate supply-chain decision rather than copying it into an example by default.

## 3. Validate, Review, and Write

Convert tolerant failures into an explicit status, retain the model and an audit-ready summary of the prompt template, system message, and material generation settings, and validate the keyed contract before writing.

```python
local_review = con.sql(
    """
    SELECT
        run_id,
        case_id,
        document_id,
        document_type,
        source_uri,
        source_text,
        prompt,
        response_text,
        'Qwen/Qwen3-1.7B'::VARCHAR AS model,
        '{"prompt_template_version":"claims-decision-v1","system_message":"Return one concise factual business note.","max_model_len":2048,"temperature":0.0,"max_tokens":128}'::VARCHAR
            AS config_summary,
        CASE
            WHEN response_text IS NULL OR trim(response_text) = ''
                THEN 'prompt_error'
            ELSE 'ready'
        END::VARCHAR AS ai_status
    FROM gpu_inference_model_output
    ORDER BY document_type, case_id, document_id
    """
)
local_review.to_table("gpu_inference_review")

validation = con.sql(
    """
    SELECT
        count(*) AS row_count,
        count(*) FILTER (
            WHERE case_id IS NULL OR document_id IS NULL
        ) AS null_key_count,
        count(*) FILTER (
            WHERE source_uri IS NULL OR source_text IS NULL
        ) AS null_source_count,
        count(*) FILTER (WHERE ai_status <> 'ready') AS error_output_count
    FROM gpu_inference_review
    """
).fetchone()
(
    actual_row_count,
    null_key_count,
    null_source_count,
    error_output_count,
) = validation

actual_keys = con.sql(
    """
    SELECT case_id, document_id
    FROM gpu_inference_review
    ORDER BY document_type, case_id, document_id
    """
).fetchall()

assert actual_row_count == len(expected_keys)
assert actual_keys == expected_keys
assert null_key_count == 0
assert null_source_count == 0

expected_columns = [
    "run_id",
    "case_id",
    "document_id",
    "document_type",
    "source_uri",
    "source_text",
    "prompt",
    "response_text",
    "model",
    "config_summary",
    "ai_status",
]
assert list(local_review.columns) == expected_columns
assert [str(dtype) for dtype in local_review.types] == ["VARCHAR"] * 11

if error_output_count:
    raise ValueError("Review prompt_error rows before scaling the run")

local_review.write_parquet("output/claims-vllm-2026-07-11-001/part-00000.parquet")
```

`on_error := 'log'` keeps exhausted failures visible as null output instead of dropping source rows. Use `on_error := 'raise'` while diagnosing setup failures. Do not blindly append a retry to a durable table: merge or replace by `(run_id, case_id, document_id)`, or write a new reviewable run directory.

## 4. Use Python Expression as an Alternate

Use Python Expression when the projection is assembled in Python or typed vLLM options are easier to build there. It has the same row-preserving model as SQL and still adds exactly one `VARCHAR` result column.

```python
python_provider_options = vane.ai.VLLMProviderOptions(
    engine_args={
        "max_model_len": 2048,
    },
    concurrency=1,
    gpus_per_actor=1,
)
python_prompt_options = vane.ai.VLLMPromptOptions(
    generate_args={
        "sampling_params": {
            "temperature": 0.0,
            "max_tokens": 128,
        }
    },
    on_error="log",
)

python_prompted = candidates.select(
    vane.col("run_id"),
    vane.col("case_id"),
    vane.col("document_id"),
    vane.col("document_type"),
    vane.col("source_uri"),
    vane.col("source_text"),
    vane.col("prompt"),
    vane.ai.prompt(
        vane.col("prompt"),
        provider="vllm",
        model="Qwen/Qwen3-1.7B",
        provider_options=python_provider_options,
        prompt_options=python_prompt_options,
        system_message="Return one concise factual business note.",
    ).alias("response_text"),
)
```

If you choose this alternate, apply the same ordered-key, cardinality, null/error, schema, and run-specific write checks used for the SQL result. Keep the AI Expression directly in `select(...)` with an alias. It is not a route to structured, multimodal, or table-shaped output.

## 5. Scale Without Mixing Resource Layers

### Local and Ray Execution

Use the smallest topology that answers the current question:

| Shape | What it proves | GPU behavior |
| --- | --- | --- |
| Local SQL or Python Expression sample | Prompt quality, model loading, engine reuse, output length, and first memory check. | The subprocess inherits visible devices. A GPU request is advisory, produces a warning when nonzero, and provides no reservation or isolation. |
| Ray-backed Expression or an explicit Relation Ray actor pool | Distributed imports, actor reuse, placement, and multi-GPU capacity. | Ray scheduler requests reserve resources per actor. Workers still need compatible dependencies and model artifacts. |
| Native vLLM Ray pool | vLLM-owned routing among reusable engine actors and optional named-pool reuse. | The inner pool has its own actor count and per-actor GPU requests; include these resources in capacity planning. |

SQL `concurrency=N` and typed `VLLMProviderOptions(concurrency=N)` map to the outer AI UDF `actor_number=N`. `gpus_per_actor=G` is the request for each of those actors. On Ray, full outer-pool parallelism therefore needs at least `N * G` advertised GPU resources. This outer concurrency is not an API request semaphore and is not the Relation method's `execution_backend` selector.

Native vLLM Ray execution is a separate provider topology. In Python, a raw provider-options dict can retain provider-specific `concurrency` together with `use_ray=True`; that concurrency configures the inner native vLLM pool rather than the outer AI UDF pool. `ray_actor_pool_name` also belongs to the inner topology, and a pool name alone does not enable it. SQL's same-name `concurrency` field is outer-only. Do not assume an outer actor's GPU reservation also covers inner Ray actors. Introduce one layer at a time and inspect Ray placement before combining them.

### Reuse Actors and Engines

The provider descriptor is lightweight. Each execution actor lazily constructs its model/client once and reuses it across batches handled by that actor. This is query-scoped execution reuse, not durable state, a checkpoint, or a guarantee that a later query will find the same engine.

A named native vLLM pool can let multiple distributed executors look up the same vLLM actor set and share inflight-load tracking. Keep the pool name, model, engine arguments, generation arguments, dependencies, and model artifacts consistent. A name collision with different settings is an operational error, not an automatic model upgrade.

### Group Similar Prefixes and Treat Caches as Tuning

When many prompts share the same system instruction or long template, group them by task type before inference if doing so does not change business meaning. Keep the stable key in the final artifact so execution grouping never becomes the join contract.

vLLM engine prefix caching is engine/version configuration. Native pool routing also exposes provider-specific controls such as `do_prefix_routing`, `min_bucket_size`, `prefix_match_threshold`, and `load_balance_threshold`. Establish a correctness and throughput baseline before changing them, and record every override. They are not generic Relation UDF actor controls.

### Tune Batch Size and Concurrency Together

`batch_size` controls the outer Vane batch delivered to an AI actor. Larger batches can improve vLLM continuous batching, but also increase GPU memory pressure, latency, and the number of rows affected by one failure. Increase actor count only after one actor is stable and the Ray cluster has enough GPU and host memory for another model copy.

Change one variable per benchmark: outer batch size, outer actor count, vLLM engine limits, or native-pool size. Measure warm-up separately from steady state, because loading weights and compiling kernels can dominate a small sample.

## 6. Operate Model Calls and Writes Safely

- **Dependencies and artifacts:** every selected worker must import the same compatible vLLM/PyTorch stack and read the exact model, tokenizer, custom-code revision, and cache path. Pre-stage large artifacts or verify worker network access before scheduling the run.
- **Credentials and network:** local vLLM usually needs no inference API credential. Put gated model-registry credentials in worker environment variables or the platform secret store, never SQL or serialized options. Restrict outbound access to required artifact locations.
- **Retries and errors:** this sample sets wrapper retries to zero. If retries are enabled, a timeout or worker/query retry can repeat work. `on_error="log"` requires explicit null/status review.
- **Effects and identity:** model calls and model downloads are external effects. They are not transactional or exactly-once, and rollback cannot undo completed work. Preserve stable IDs and `run_id`; make downstream writes idempotent or reviewable.
- **Determinism:** `temperature=0` improves repeatability but does not turn a distributed GPU run into an exactly-once or byte-identical computation across model, kernel, or dependency changes.

## 7. Use Relation Prompting Last for Advanced Capabilities

Use `rel.prompt` only when the task requires Relation-only structured output, supported multimodal `image_columns`, an explicit Relation backend, or explicit Relation actor-pool controls. It is a Python table transformation; Vane does not expose a direct SQL Relation API or SQL table function for it. The exact capability and output boundary is in the [AI Function API Reference](/docs/data/reference/ai-api), and the design reason is in [AI Function Concepts](/docs/data/concepts/ai-functions).

This construction selects a valid Ray actor backend, supplies a positive `actor_number`, requests one Ray GPU per outer actor, and asks vLLM for structured output:

```python
import vane

claim_decision_schema = {
    "type": "object",
    "properties": {
        "decision": {"type": "string"},
        "reason": {"type": "string"},
    },
    "required": ["decision", "reason"],
    "additionalProperties": False,
}


vane.configure(runner="ray")
ray_con = vane.connect()

advanced_candidates = ray_con.sql(
    """
    SELECT
        CAST(case_id AS VARCHAR) AS case_id,
        CAST(document_id AS VARCHAR) AS document_id,
        (
            'Return a structured claim decision and reason: '
            || CAST(text AS VARCHAR)
        )::VARCHAR AS prompt
    FROM read_parquet('data/claims/documents/*.parquet')
    WHERE case_id IS NOT NULL
      AND document_id IS NOT NULL
      AND text IS NOT NULL
    ORDER BY case_id, document_id
    LIMIT 20
    """
)

ray_provider_options = {
    "engine_args": {
        "max_model_len": 2048,
    },
    "actor_number": 2,
    "gpus_per_actor": 1.0,
    "batch_size": 8,
}
ray_prompt_options = vane.ai.VLLMPromptOptions(
    generate_args={
        "sampling_params": {
            "temperature": 0.0,
            "max_tokens": 128,
        }
    },
    on_error="log",
)

decision_only = advanced_candidates.prompt(
    "prompt",
    provider="vllm",
    model="Qwen/Qwen3-1.7B",
    provider_options=ray_provider_options,
    prompt_options=ray_prompt_options,
    return_format=claim_decision_schema,
    output_column="decision_json",
    execution_backend="ray_actor",
)
```

Here `actor_number=2` is the outer Relation UDF actor-pool size, and `gpus_per_actor=1.0` makes Ray reserve one GPU resource for each actor. Because `use_ray` is absent, each Ray UDF actor owns its local vLLM engine; this example does not create a second native vLLM Ray pool. If you deliberately choose the native pool instead, account for its separate actors and resources.

The built-in Relation helper may return only its configured `decision_json` column. The `_only` name makes that boundary visible: do not zip it back to `advanced_candidates` by row position. Use it as an isolated output-only stage when keys are not needed downstream. When the published artifact must retain IDs and multiple diagnostic columns, stay with SQL/Python Expression or use a custom row-preserving batch UDF with an explicit schema.

Add `image_columns=[...]` only for a provider/model combination that supports the chosen image representation, and validate the structured JSON against `claim_decision_schema` before publishing it. SQL can prepare the input and inspect a materialized result around this call, but that does not turn the Python Relation helper into a SQL Relation API.
