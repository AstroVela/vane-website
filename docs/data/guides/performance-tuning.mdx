---
title: Performance Tuning
---

Use this guide after your multimodal, AI, or inference pipeline produces correct results on a small, representative slice of your own data. Performance tuning should answer two questions: which stage is slow, and which single variable is worth changing next.

This is not a production benchmark, SLA guide, or advanced cluster tuning manual. The useful outcome is a clear bottleneck diagnosis and a run record that makes performance feedback actionable.

## Start With Correctness

Do not tune a pipeline that is still changing shape. First make sure the run is small, repeatable, and checkable.

- Keep stable IDs from input through output.
- Check row counts before and after every expensive stage.
- Validate the output schema and a few sample rows by hand.
- Keep explicit `status` and `error_message` columns for rows that fail.
- Use a representative sample, not only the easiest records.
- Record a baseline before changing any tuning knob.

Then tune one variable at a time. Keep the dataset slice, code, model, provider, and output path fixed while you compare runs. If a change does not improve the bottleneck you measured, revert it before trying the next knob.

## Performance vs Sizing

Performance Tuning diagnoses where time or failures are coming from in a concrete run. Sizing estimates the resources needed for the next larger run.

| Question | Use Performance Tuning | Use Sizing |
| --- | --- | --- |
| "Why is this run slow?" | Yes | No |
| "Which stage is the bottleneck?" | Yes | No |
| "Should I change batch size or concurrency?" | Yes | Maybe |
| "How many CPUs, GPUs, or workers might I need for a larger slice?" | No | Yes |
| "Can I promise production throughput?" | No | No |

Use the performance record in this guide as input when you later estimate resources in Sizing.

## Tuning Order

Move through the pipeline in order. A later tuning knob will not help if an earlier stage is sending too much data, using inaccessible paths, or writing oversized output.

| Order | Stage | What to check | Smallest useful change |
| --- | --- | --- | --- |
| 1 | Data read | Input rows, input bytes, file count, object store latency, worker access to paths | Start with a smaller real slice; use worker-visible paths; avoid reading unused files |
| 2 | SQL filter/project | Rows and columns entering the expensive stage | Filter early; project only the columns the UDF, AI function, or model needs |
| 3 | UDF batch | Per-batch time, memory, long tail batches, retry cost | Adjust `batch_size` only after input columns and rows are reduced |
| 4 | Actor reuse | Model, tokenizer, media decoder, or client initialization logs, repeated setup, low throughput | Use an actor backend for stateful model/client/GPU reuse |
| 5 | Provider rate limit | HTTP 429s, timeouts, retry waits, provider-side quota errors | Lower concurrency; batch only where the provider supports it; reduce unnecessary calls |
| 6 | GPU utilization | GPU memory, utilization, model load time, CPU preprocessing time | Reuse model actors; tune batch size; add GPUs only after one GPU is understood |
| 7 | Output write | Output rows, output bytes, wide columns, many small files, worker access to output path | Write only required columns; avoid carrying raw bytes or large intermediate objects unless needed |

## Core Knobs

These are the main knobs for an initial workflow. Change one at a time and keep the run record updated.

| Knob | What it changes | Start here | Watch for |
| --- | --- | --- | --- |
| `batch_size` | Rows per UDF, AI, or inference call | Increase when per-call overhead dominates; decrease when memory or retries dominate | Out-of-memory errors, slow long-tail batches, worse provider latency |
| UDF `concurrency` | Number of UDF batches or actors in flight, when the selected backend supports it | Increase only after each batch is reasonably efficient; do not pass it to `ray_task` | Provider rate limits, memory pressure, output write contention |
| Provider API concurrency | Hosted-provider async request concurrency, when the provider exposes a separate option such as `max_api_concurrency` | Tune against provider limits, not Ray worker count | 429s, retry storms, quota errors, higher cost |
| vLLM actor-pool concurrency | Number of vLLM actors created by the native vLLM executor or provider when Ray vLLM is enabled | Increase only when GPU slots and model memory are available | Ray pending actors, GPU OOM, low per-actor utilization |
| Task vs actor | Whether work is stateless or reuses state | Use tasks for light stateless CPU work; actors for models, clients, GPU, or connection reuse | Model reload per batch, unbounded actor memory, worker startup failures |
| Local vs Ray | Whether the pipeline stays local or uses Ray workers | Stay local for correctness and small samples; move to Ray when parallelism, worker placement, or GPUs matter | Path, dependency, credential, and output locations visible only on the driver |
| GPU count | Available GPU resources for inference actors | Tune one GPU first; add GPUs when the GPU stage is confirmed as the bottleneck | CPU preprocessing bottlenecks, output bottlenecks, low utilization across many actors |

Runner and backend are related but different. The runner controls the overall execution path, such as local execution versus Ray execution. The UDF or AI execution backend controls how a stage runs, such as task versus actor.

## Hosted API vs GPU Bottlenecks

Hosted APIs and local GPU inference fail in different ways. Separate them before changing knobs.

| Workload type | Common bottleneck | Symptoms | Useful fixes |
| --- | --- | --- | --- |
| Hosted API | Provider quota, request latency, retries, cost | 429 or quota errors, long retry waits, idle local CPU/GPU, failures tied to credentials or provider limits | Lower `concurrency`; reduce rows before the call; use provider-supported batch requests; record provider/model/error details |
| GPU inference | Model load, GPU memory, GPU utilization, CPU preprocessing, media decode, output write | GPU out-of-memory, GPU mostly idle, model initialization per batch, slow decode/tokenization before GPU work | Use actor reuse; tune `batch_size`; separate preprocessing from inference; verify one GPU before increasing GPU count |

Do not treat a hosted API rate limit as a Ray scheduling problem. Do not treat low GPU utilization as proof that more GPUs are needed. First confirm which stage is waiting and why.

## vLLM and Ray-Backed Inference

vLLM tuning has two different entry points in Vane:

| Path | What to tune |
| --- | --- |
| SQL `vllm(prompt, model, options)` | Native vLLM operator settings: `engine_args`, `generate_args`, `batch_size`, `do_prefix_routing`, `max_buffer_size`, `min_bucket_size`, `prefix_match_threshold`, `inflight_limit`, `use_ray`, `concurrency`, `gpus_per_actor`, and `ray_actor_pool_name`. |
| Relation `prompt(provider="vllm")` | AI helper and vLLM provider settings: prompt relation shape, helper output validation, `engine_args`, `generate_args`, `use_ray`, `concurrency`, `gpus_per_actor`, UDF `execution_backend`, and row-count/order recombination. |

The native vLLM executor applies throughput-oriented engine defaults when you do not override them: `max_num_batched_tokens=8192` and `max_num_seqs=256`. Its Vane-side defaults include `concurrency=1`, `gpus_per_actor=1`, `batch_size=128`, `max_buffer_size=5000`, `min_bucket_size=16`, `inflight_limit=128`, and prefix routing enabled. Record any overrides in the run log, because changes to context length, sampling, tensor parallelism, or batching can dominate throughput comparisons.

For Ray-backed native vLLM, separate the two resource layers:

- UDF or relation backend concurrency controls how many Vane tasks or actors submit work.
- vLLM actor-pool `concurrency` controls how many vLLM actors are created.
- `gpus_per_actor` is per vLLM actor, so required GPU slots are approximately `concurrency * gpus_per_actor`.
- Named pools through `ray_actor_pool_name` allow actor reuse and shared inflight tracking across distributed tasks.

For prefix-heavy prompt sets, first establish a baseline with the same model, data slice, and sampling settings. Then test prompt ordering or native prefix routing (`do_prefix_routing`, `min_bucket_size`, `prefix_match_threshold`) without changing model settings. If the prompts do not share long prefixes, prefix routing is unlikely to explain the bottleneck. Do not treat `load_balance_threshold` as a primary tuning knob unless you have verified its effect in the current execution path.

Backpressure matters when prompt submission is faster than GPU completion. In native SQL `vllm()`, `inflight_limit` bounds submitted prompts; a low value can protect memory and improve load awareness, while a value that is too low can starve the GPU. Treat it as a measured tuning variable, not a general speed knob.

## Symptoms and Fixes

| Symptom | Likely cause | Fix to try first |
| --- | --- | --- |
| Many tiny UDF or AI calls, high scheduling overhead, low model throughput | Batches are too small | Increase `batch_size` gradually while watching memory and error rate |
| Out-of-memory errors, very slow retries, or one large batch dominates elapsed time | Batches are too large | Reduce `batch_size`; project fewer columns; split decode, transform, and inference into separate stages |
| Logs show model loading or API client setup for every batch | Stateless task is recreating expensive state | Move model or client initialization into a callable class and run with an actor backend |
| Hosted API returns 429, timeout, quota, or authentication errors | Provider rate limit or credential issue | Lower `concurrency`; reduce calls with SQL filters; verify credentials in the same environment as the workers |
| Local run works, Ray run fails with file not found, import errors, or missing credentials | Worker path, dependency, or credential visibility | Put data, models, and output on worker-visible storage; install dependencies on workers; pass credentials through the Ray environment |
| GPU memory is full but throughput is unstable | Batch too large, output too wide, or actor memory growth | Reduce `batch_size`; avoid carrying unused columns; restart from a clean small run and compare |
| GPU utilization stays low while CPU is busy | CPU preprocessing or data read is the bottleneck | Time decode/tokenization separately; filter/project before inference; cache or stage expensive preprocessing output |
| Output write takes most of the run | Output is too large or too wide | Write only the columns needed for the next workflow step; avoid raw bytes and large Python object columns in final output |

## Minimal Measurement Loop

Use the same small representative slice for a short tuning loop.

1. Run the baseline and record elapsed time, errors, and stage-level observations.
2. Pick the first bottleneck in the tuning order.
3. Change one knob.
4. Run the same slice again.
5. Keep the change only if it improves the measured bottleneck without breaking correctness.

For AI and inference workloads, measure at least:

- rows entering the AI or model stage
- calls or batches sent
- successful rows and failed rows
- UDF accepted, submitted, completed, produced, and emitted row counts when runtime counters are available
- provider retry or rate-limit count, if hosted
- provider API concurrency limit, if hosted
- GPU memory and utilization, if local GPU
- vLLM actor count, batch size, inflight limit, prefix routing settings, and engine args, if vLLM
- output rows and bytes
