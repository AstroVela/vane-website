title: Performance Tuning
---

Start with a correct local pipeline, then tune the smallest number of variables.

## Runner

Local:

```python
vane.configure(runner="local")
```

Ray:

```python
vane.configure(runner="ray")
```

The local runner streams Arrow record batches. The Ray runner initializes Ray when needed and sends a logical plan to a Ray query driver.

## Batch size

For `map_batches`, `batch_size` controls how many rows are passed to one UDF call.

Small batches:

- lower memory
- worse model throughput
- more scheduling overhead

Large batches:

- better throughput
- higher memory
- slower retries

Tune with representative data.

## Actor-backed reuse

Use actor backends for model inference:

- `subprocess_actor` for local model or client reuse.
- `ray_actor` for distributed model or client reuse.

Use task backends for light stateless work:

- `subprocess_task`
- `ray_task`

## CPU and GPU resources

`map_batches` accepts:

- `cpus`
- `gpus`
- `actor_number` for actor-backed callable classes

An actor backend requires a callable class, a positive `actor_number`, and an explicit non-negative `gpus` value; use `gpus=0.0` for CPU-only actors. Ray uses resource requests for placement, so make sure the Ray cluster advertises matching resources before increasing `actor_number`. A local subprocess inherits visible GPUs but does not reserve or isolate them.

Expression `vane.func` uses a task backend. Stateful `vane.cls` v1 requires `actor_number=1`; its mutable state is query-scoped, actor-local, and ephemeral, so it is not a scale-out control. AI provider `concurrency` is a provider-facing option that maps to the AI actor count; for prompt providers, `max_api_concurrency` controls request concurrency inside each actor. Relation actor pools use `actor_number` directly.

## Ray scan task grouping

Public config:

```python
vane.configure(ray_scan_task_size_grouping=True)
```

Environment variable:

```bash
export VANE_RAY_SCAN_TASK_SIZE_GROUPING=true
```

The registry describes this as size-based scan task grouping, useful for merging small files into larger scan tasks.

Related public config:

- `ray_max_task_backlog`
- `ray_scan_task_open_cost_bytes`
- `ray_scan_task_min_partition_num`
- `ray_init_sql`

## Backpressure

`VANE_RAY_MAX_TASK_BACKLOG` limits pending Ray tasks when set to a positive value. `0` means unlimited in the public registry.

Use it when the driver can submit faster than workers can consume.

## Avoid common bottlenecks

- Project only columns needed by the UDF.
- Filter before model calls.
- Keep image/audio/video decoding separate from model inference when tuning.
- Write to a filesystem all workers can access.
- Avoid one row per UDF call for model workloads.
- Use structured output schemas instead of Python object columns.

## Measure

Keep a simple run record:

- input rows
- input bytes
- batch size
- backend
- actor number or provider concurrency
- GPUs per actor
- elapsed time
- output rows
- failure count

Benchmark workloads can be useful references for environment-driven performance experiments, but benchmark-specific file locations, storage endpoints, and model defaults are not general deployment defaults.

Read the [UDF API Reference](/docs/data/reference/udf-api) before tuning Expression cardinality, state, or GPU resources.
