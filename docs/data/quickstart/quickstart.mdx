This quickstart gets one Python function running inside a Vane SQL projection. The required path uses inline data, runs locally, and needs no Ray cluster, GPU, PyArrow, or model-provider credentials.

You will create three rows, register a Python function as a SQL alias, call it from `SELECT`, and see the result.

## Install Vane Data

```bash
python -m pip install vane-ai
```

## Run Your First SQL Expression UDF

Copy this into a Python file and run it:

```python
import vane

con = vane.connect()

con.execute("""
    CREATE TABLE documents AS
    SELECT *
    FROM (VALUES
        ('doc-001', 'Signed claim form received.'),
        ('doc-002', 'Invoice is missing approval metadata.'),
        ('doc-003', 'Policy reference has expired.')
    ) AS rows(document_id, source_text)
""")


def review_status(source_text):
    text = str(source_text).lower()
    if "missing" in text or "expired" in text:
        return "needs_review"
    return "ready"


vane.attach_function(
    review_status,
    alias="review_status_sql",
    connection=con,
    parameters=["VARCHAR"],
    return_dtype="VARCHAR",
)

reviewed = con.sql("""
    SELECT
        document_id,
        source_text,
        review_status_sql(source_text) AS review_status
    FROM documents
    ORDER BY document_id
""")

reviewed.show()
vane.detach_function("review_status_sql", connection=con)
```

The result contains the original columns plus one typed `review_status` column:

| document_id | review_status |
| --- | --- |
| doc-001 | ready |
| doc-002 | needs_review |
| doc-003 | needs_review |

## What Just Happened

- `vane.attach_function(...)` registered the raw Python callable on `con`.
- `review_status_sql(...)` became a SQL Expression that could run in a `SELECT` projection.
- The Expression added one result beside each input row, so IDs and source text stayed in the same relation.
- `vane.detach_function(...)` removed the alias after the lazy result had been materialized by `show()`.

For exception-safe cleanup, batch functions, callable classes, and Relation transformations, continue with [Custom Python UDFs](/docs/data/guides/custom-python-udfs). For exact signatures and restrictions, use the [UDF API Reference](/docs/data/reference/udf-api).

## Optional: Add an AI Function

This is not required for the Quickstart. To try a hosted provider, install its package and set credentials in the worker environment:

```bash
python -m pip install openai
export OPENAI_API_KEY="<your-key>"
```

Then add an AI result with the SQL Expression function `ai_prompt`:

```python
with_ai = con.sql("""
    SELECT
        document_id,
        ai_prompt(
            source_text,
            struct_pack(
                provider := 'openai',
                model := 'gpt-4o-mini'
            )
        ) AS review_note
    FROM documents
    ORDER BY document_id
""")

with_ai.show()
```

Provider calls can incur cost and send data outside your environment. Keep the first input small and read [AI Functions](/docs/data/guides/ai-functions) before building a larger workflow.

## Scale from Local to Ray

The Quickstart uses the default local runner. To distribute the same UDF or AI Expression work, configure Ray **before** creating the connection:

```python
import vane

vane.configure(runner="ray")
con = vane.connect()
```

The SQL projection and registered callable can stay the same. For a larger job, replace the inline `VALUES` table with a shared source such as Parquet or object storage.

Every Ray worker must be able to import the UDF and its dependencies and access the same data, model files, and credentials required by its work. See [Ray Cluster Deployment](/docs/data/deploy/ray-cluster) before running on a cluster.
