This quickstart gets one Python function running inside a Vane SQL projection. The required path uses inline data, runs on your machine, and needs no external Ray cluster, GPU, or model-provider credentials.

You will create three rows, register a Python function as a SQL alias, call it from `SELECT`, and see the result.

Before continuing, complete the [Installation guide](/docs/data/quickstart/installation).

## Run Your First SQL Expression UDF

Copy this into a Python file and run it:

```python
import vane

con = vane.connect()

documents = con.values(
    (
        vane.lit("doc-001").alias("document_id"),
        vane.lit("Signed claim form received.").alias("source_text"),
    ),
    (vane.lit("doc-002"), vane.lit("Invoice is missing approval metadata.")),
    (vane.lit("doc-003"), vane.lit("Policy reference has expired.")),
)


def review_status(source_text):
    text = str(source_text).lower()
    if "missing" in text or "expired" in text:
        return "needs_review"
    return "ready"


vane.attach_function(
    review_status,
    alias="review_status_sql",
    connection=con,
    parameters=["VARCHAR"],
    return_dtype="VARCHAR",
)

reviewed = con.sql("""
    SELECT
        document_id,
        source_text,
        review_status_sql(source_text) AS review_status
    FROM documents
    ORDER BY document_id
""")

reviewed.show()
vane.detach_function("review_status_sql", connection=con)
```

The result contains the original columns plus one typed `review_status` column:

| document_id | review_status |
| --- | --- |
| doc-001 | ready |
| doc-002 | needs_review |
| doc-003 | needs_review |

## What Just Happened

- `vane.attach_function(...)` registered the raw Python callable on `con`.
- `review_status_sql(...)` became a SQL Expression that could run in a `SELECT` projection.
- The Expression added one result beside each input row, so IDs and source text stayed in the same relation.
- `vane.detach_function(...)` removed the alias after the lazy result had been materialized by `show()`.

For callable lifecycle, batch and cardinality behavior, and Relation transformations, continue with [UDF Concepts](/docs/data/concepts/udfs).

## Optional: Add an AI Function

This is not required for the Quickstart. To try OpenAI, install its optional dependency in the same environment, then set credentials in the worker environment:

```bash
uv pip install openai
export OPENAI_API_KEY="<your-key>"
```

Then add an AI result with the SQL Expression function `ai_prompt`:

```python
with_ai = con.sql("""
    SELECT
        document_id,
        ai_prompt(
            source_text,
            struct_pack(
                provider := 'openai',
                -- Optional for OpenAI-compatible providers:
                -- base_url := 'https://api.example.com/v1',
                model := 'gpt-4o-mini'
            )
        ) AS review_note
    FROM documents
    ORDER BY document_id
""")

with_ai.show()
```

Provider calls can incur cost and send data outside your environment. Keep the first input small and review [AI Function Concepts](/docs/data/concepts/ai-functions) before building a larger workflow.

## Scale from Local Ray to a Cluster

Ray is Vane's default runner. With `RAY_ADDRESS` unset, this quickstart starts a Ray runtime on the current machine; no external cluster or explicit runner configuration is required.

To run the same job on a cluster, launch it from the head node with cluster discovery enabled:

```bash
RAY_ADDRESS=auto python quickstart.py
```

The SQL projection and registered callable can stay the same. For a larger job, replace the inline relation with a shared source such as Parquet or object storage.

Every Ray worker must be able to import the UDF and its dependencies and access the same data, model files, and credentials required by its work. See [Deployment](/docs/data/deploy/deployment) before running on a cluster.
