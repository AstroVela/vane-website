This Reference is lookup material for public UDF signatures, parameters, returns, and call restrictions. For complete tasks, use the [Custom Python UDFs Guide](/docs/data/guides/custom-python-udfs). For execution semantics and design reasons, see [UDF Concepts](/docs/data/concepts/udfs).

Vane has two UDF models. The Expression API is the default and produces one typed output column; its recommended SQL entry point comes first, followed by the equivalent Python entry point. The Relation API is the specialized model for table-shaped transformations, row expansion, and explicit Relation execution controls.

## Expression API

Expression UDFs participate in a projection and normally preserve one output row per input row. Register a Python callable as a SQL alias for the recommended entry point, or build the same model with Python expressions.

### SQL Entry Point (Recommended)

`vane.attach_function` and `vane.detach_function` are Python registration controls for SQL Expression calls. They do not define a separate API model.

**Registered-object compatibility**

| Registered object | Required or allowed arguments | Forbidden or fixed arguments |
| --- | --- | --- |
| Decorated `VaneFunction` | `parameters`; effective `return_dtype` from the decorator or attach call | `input_names`, `schema`, `batch_size`, `gpus`, and `actor_number` are forbidden. |
| Raw scalar callable | `parameters` and `return_dtype` | All batch and actor arguments are forbidden. |
| Raw batch function | `parameters`, `input_names`, and one-column `schema`; optional `batch_size` and `gpus` | `return_dtype` and `actor_number` are forbidden. |
| Zero-argument raw batch callable class | Raw batch arguments plus a positive `actor_number` | Constructor arguments and `return_dtype` are forbidden; this is actor-backed reuse, not the `vane.cls` state contract. |
| Instantiated `VaneClassInstance` | `parameters`; optional explicit `input_names` and `batch_size` override | `return_dtype`, `schema`, `gpus`, and `actor_number` cannot override the decorator; row preservation is fixed. |
| Instantiated `VaneClassBatchInstance` | `parameters` and `input_names`; optional `schema` and `batch_size` override | `return_dtype`, `gpus`, and `actor_number` cannot override the decorator; the decorator must set `row_preserving=True`. |

Uninstantiated Vane class wrappers must first capture constructor arguments, for example `Counter()`, before registration.

#### `vane.attach_function`

**Purpose**

Register a supported Python callable or Vane wrapper as a connection-scoped SQL Expression alias.

**Signature**

```python
vane.attach_function(
    fn_or_function: Any,
    alias: str | None = None,
    *,
    connection: Any | None = None,
    replace: bool = False,
    parameters: Any = None,
    return_dtype: Any | None = None,
    input_names: Any = None,
    schema: Mapping[str, Any] | None = None,
    batch_size: int | None = None,
    gpus: float | None = None,
    actor_number: int | None = None,
) -> None
```

**Parameters**

| Parameter | Contract |
| --- | --- |
| `fn_or_function` | A decorated `VaneFunction`, raw callable, zero-argument raw batch callable class, or instantiated Vane class wrapper. |
| `alias` | SQL function name. `None` derives the name from the wrapper or callable. |
| `connection` | Owning connection. `None` uses `duckdb.default_connection()`. |
| `replace` | `False` rejects an existing signature. `True` may replace only a Vane alias owned by this connection with the same SQL signature. |
| `parameters` | SQL input types as a list or tuple; required by the supported object contracts above. |
| `return_dtype` | Scalar output type. It may come from a decorator where the compatibility matrix permits. |
| `input_names` | Ordered batch SQL argument names. Names must be non-empty, match `parameters`, and be unique case-insensitively. |
| `schema` | Batch output mapping. SQL batch aliases require exactly one declared output column. |
| `batch_size` | Optional positive batch size. |
| `gpus` | Optional non-negative batch or raw-actor GPU resource request. |
| `actor_number` | Positive actor count for a zero-argument raw batch callable class; it cannot override a Vane class decorator. |

**Returns**

`None` after the alias is registered on the selected connection.

**Behavior and data contract**

The selected connection owns the alias, and only queries planned on that connection can use it. Python preflight validation completes before the catalog change is applied. Batch aliases accept case-insensitive SQL named arguments and reorder them to `input_names`; scalar aliases accept positional arguments only. `replace=True` is limited to a same-signature Vane alias owned by the same connection.

**Minimal example**

```python
import vane

con = vane.connect()

def add_one(value):
    return value + 1

vane.attach_function(
    add_one,
    alias="add_one_sql",
    connection=con,
    parameters=["INTEGER"],
    return_dtype="INTEGER",
)

try:
    rows = con.sql(
        "SELECT value, add_one_sql(value) AS next_value "
        "FROM (VALUES (1), (2)) AS items(value)"
    ).fetchall()
finally:
    vane.detach_function("add_one_sql", connection=con)
```

**Restrictions and errors**

Unsupported objects, missing types, invalid option combinations, duplicate `input_names`, and uninstantiated Vane class wrappers fail during Python preflight. Apply registration outside an active transaction: the DuckDB catalog apply can cancel the active transaction. The shared compatibility table above is normative for allowed overrides.

#### `registered_alias(...args)`

**Purpose**

Invoke a registered Python UDF as the recommended SQL entry point to the Expression API.

**Signature**

```sql
registered_alias(argument [, ...]) -> declared_return_type
```

**Parameters**

`argument [, ...]` must match the SQL `parameters` declared at registration. Raw batch aliases may also use case-insensitive named arguments declared by `input_names`; scalar aliases reject named arguments.

**Returns**

One value of the registered `declared_return_type` for each input row.

**Behavior and data contract**

The alias is connection-scoped and row-preserving in v1. It is valid in a `SELECT projection` and composes beside ordinary columns, so identifiers and source fields can remain in the same projection.

**Minimal example**

```python
import vane

con = vane.connect()

def add_tax(amount):
    return amount * 1.1

vane.attach_function(
    add_tax,
    alias="add_tax",
    connection=con,
    parameters=["DOUBLE"],
    return_dtype="DOUBLE",
)

try:
    rows = con.sql(
        "SELECT order_id, amount, add_tax(amount) AS total "
        "FROM (SELECT 1 AS order_id, 10.0::DOUBLE AS amount) AS orders"
    ).fetchall()
finally:
    vane.detach_function("add_tax", connection=con)
```

**Restrictions and errors**

Registered aliases are unsupported in `WHERE`, `GROUP BY`, `HAVING`, join conditions, aggregate arguments, and short-circuit-sensitive `CASE`, `AND`, `OR`, or `COALESCE` expressions. Planning fails rather than falling back to per-row Python execution. SQL aliases cannot use the cardinality-changing `row_preserving=False` contract in v1.

**Related pages**

[SQL-first UDF task](/docs/data/guides/custom-python-udfs) · [Expression placement model](/docs/data/concepts/udfs)

#### `vane.detach_function`

**Purpose**

Remove a registered SQL Expression alias from a selected connection.

**Signature**

```python
vane.detach_function(
    alias: str,
    *,
    connection: Any | None = None,
) -> None
```

**Parameters**

| Parameter | Contract |
| --- | --- |
| `alias` | Name of the SQL alias to remove. |
| `connection` | Connection from which to remove it. `None` uses `duckdb.default_connection()`. |

**Returns**

`None` after the alias is removed.

**Behavior and data contract**

Removal applies only to the specified connection, or to the default connection when `connection=None`. It delegates to that connection's function-removal operation.

**Minimal example**

```python
import vane

con = vane.connect()

def identity(value):
    return value

vane.attach_function(
    identity,
    alias="identity_sql",
    connection=con,
    parameters=["VARCHAR"],
    return_dtype="VARCHAR",
)
vane.detach_function("identity_sql", connection=con)
```

**Restrictions and errors**

Use the same connection that owns the alias. A missing alias or a connection mismatch is reported by the connection's removal operation; detaching on one connection does not affect aliases on another.

[SQL alias lifecycle task](/docs/data/guides/custom-python-udfs) · [Connection and execution semantics](/docs/data/concepts/udfs)

### Python Entry Point

Python constructors and wrappers build the same one-column Expression model. Calls remain lazy when an input is an `Expression`; where documented, ordinary Python values use eager execution.

#### `vane.col`

**Purpose**

Construct a reference to a named relation column.

**Signature**

```python
vane.col(name: str) -> Expression
```

**Parameters**

`name` is the column name to resolve against the relation that consumes the expression.

**Returns**

A DuckDB `Expression` representing the column reference.

**Behavior and data contract**

The reference is lazy. It is bound when used in a relation operation such as `select`.

**Minimal example**

```python
import vane

con = vane.connect()
rel = con.sql("SELECT 1 AS id, 'hello' AS text")
result = rel.select(vane.col("id"), vane.col("text"))
```

**Restrictions and errors**

`name` must resolve in the consuming relation. Missing or ambiguous columns fail during binding.

**Related pages**

[Python Expression UDF task](/docs/data/guides/custom-python-udfs) · [Expression model](/docs/data/concepts/udfs)

#### `vane.lit`

**Purpose**

Construct a constant expression from a Python literal.

**Signature**

```python
vane.lit(value: Any) -> Expression
```

**Parameters**

`value` is a Python value convertible to a DuckDB literal.

**Returns**

A constant DuckDB `Expression`.

**Behavior and data contract**

The literal is part of the lazy projection and is repeated according to the relation's row count.

**Minimal example**

```python
import vane

con = vane.connect()
rel = con.sql("SELECT 1 AS id")
result = rel.select(
    vane.col("id"),
    vane.lit("ready").alias("status"),
)
```

**Restrictions and errors**

Values without a DuckDB literal conversion are rejected when the expression is constructed or bound.

**Related pages**

[Python Expression UDF task](/docs/data/guides/custom-python-udfs) · [Expression model](/docs/data/concepts/udfs)

#### `vane.sql_expr`

**Purpose**

Construct a DuckDB expression from SQL expression text.

**Signature**

```python
vane.sql_expr(sql: str) -> Expression
```

**Parameters**

`sql` is one SQL expression, not a complete query.

**Returns**

A parsed DuckDB `Expression`.

**Behavior and data contract**

Column names in the SQL text are resolved against the consuming relation, and the result composes with other Python expressions.

**Minimal example**

```python
import vane

con = vane.connect()
rel = con.sql("SELECT 3 AS quantity, 4.5 AS price")
result = rel.select(
    vane.sql_expr("quantity * price").alias("subtotal"),
)
```

**Restrictions and errors**

Invalid SQL expression text and unresolved names fail during parsing or binding. Do not interpolate untrusted text into `sql`.

**Related pages**

[Python Expression UDF task](/docs/data/guides/custom-python-udfs) · [SQL and Python entry points](/docs/data/concepts/udfs)

#### `vane.func`

**Purpose**

Wrap a stateless scalar Python callable for eager calls and one-column Expression execution.

**Signature**

```python
vane.func(
    fn: Callable | None = None,
    *,
    return_dtype: Any | None = None,
    name: str | None = None,
) -> VaneFunction | decorator
```

**Parameters**

| Parameter | Contract |
| --- | --- |
| `fn` | Callable to wrap. `None` supports parameterized decorator use. |
| `return_dtype` | Single-column output type. Optional for eager-only calls, but required for Expression or SQL execution. |
| `name` | Display and default SQL name. `None` derives it from the callable, with `map_udf` as the final fallback. |

**Returns**

A `VaneFunction` wrapper, or a decorator when `fn=None`. Calling the wrapper returns the eager Python result or a DuckDB `Expression`, depending on its inputs.

**Behavior and data contract**

When every argument is an ordinary Python value, the original callable runs immediately in the current process. When any positional argument is an `Expression`, the call builds a lazy, stateless scalar expression using a task backend. Ordinary keyword values remain Python arguments; mutable state is not retained across batches.

**Minimal example**

```python
import vane

@vane.func(return_dtype="INTEGER", name="add_one")
def add_one(value):
    return value + 1

assert add_one(1) == 2

con = vane.connect()
rel = con.sql("SELECT 1::INTEGER AS value")
result = rel.select(
    vane.col("value"),
    add_one(vane.col("value")).alias("next_value"),
)
```

**Restrictions and errors**

Expression inputs must be positional; an `Expression` supplied in keyword arguments is rejected. Expression execution requires `return_dtype` and follows the shared projection and short-circuit placement limits.

**Related pages**

[Build a scalar UDF task](/docs/data/guides/custom-python-udfs) · [Eager and Expression execution](/docs/data/concepts/udfs)

#### `vane.func.batch`

**Purpose**

Wrap a stateless Arrow batch callable as one typed output Expression.

**Signature**

```python
vane.func.batch(
    fn: Callable[[pyarrow.Table], Any],
    *,
    inputs: Mapping[str, Any],
    schema: Mapping[str, Any] | None = None,
    name: str | None = None,
    batch_size: int | None = None,
    row_preserving: bool = False,
    gpus: float | None = None,
) -> Any | Expression
```

**Parameters**

| Parameter | Contract |
| --- | --- |
| `fn` | Callable that receives a `pyarrow.Table`. |
| `inputs` | Non-empty name-to-literal-or-expression mapping; keys become Arrow input column names. |
| `schema` | Expression output mapping; it must be non-empty and contain exactly one column. |
| `name` | UDF name. `None` derives it from the callable, with `map_batches_udf` as fallback. |
| `batch_size` | Optional positive input batch size. |
| `row_preserving` | `False` permits cardinality change as a result-only projection; `True` requires one output row per input row. |
| `gpus` | Optional non-negative execution resource declaration. |

**Returns**

The eager callable result when every input is a literal, or one DuckDB `Expression` when any input is an `Expression`.

**Behavior and data contract**

The eager path builds `pa.table(inputs)` and calls `fn` immediately. The Expression path names input columns from `inputs` and uses a stateless task backend. `row_preserving=True` checks equal input and output row counts and permits ordinary projection columns beside the result.

**Minimal example**

```python
import pyarrow as pa
import vane

def normalize_batch(table: pa.Table) -> pa.Table:
    values = table.column("text").to_pylist()
    return pa.table({"normalized": [value.strip().lower() for value in values]})

con = vane.connect()
rel = con.sql("SELECT 1 AS id, ' Hello ' AS text")
normalized = vane.func.batch(
    normalize_batch,
    inputs={"text": vane.col("text")},
    schema={"normalized": "VARCHAR"},
    row_preserving=True,
).alias("normalized")
result = rel.select(vane.col("id"), normalized)
```

**Restrictions and errors**

`inputs` must be non-empty. The Expression schema must declare exactly one output column. With `row_preserving=False`, the batch expression may change cardinality but must be the sole top-level projection; SQL registration accepts only row-preserving batch wrappers.

**Related pages**

[Batch Expression task](/docs/data/guides/custom-python-udfs) · [Cardinality contracts](/docs/data/concepts/udfs)

#### `vane.cls`

**Purpose**

Decorate a scalar callable class whose mutable state is reused by one actor during one query.

**Signature**

```python
vane.cls(
    class_: type | None = None,
    *,
    actor_number: int | None = None,
    return_dtype: Any | None = None,
    name: str | None = None,
    gpus: float | None = 0,
) -> VaneClass | decorator
```

**Parameters**

| Parameter | Contract |
| --- | --- |
| `class_` | Class to decorate. `None` supports parameterized decorator use. |
| `actor_number` | Signature default is `None`; v1 requires the exact integer `1` explicitly. |
| `return_dtype` | Required scalar output type. |
| `name` | UDF name. `None` derives it from the class, with `class_udf` as fallback. |
| `gpus` | Per-actor GPU resource request; default `0`. |

**Returns**

A `VaneClass` wrapper, or a decorator when `class_=None`. Calling the decorated class captures constructor arguments in an instance wrapper; calling that instance returns an eager value or an `Expression`.

**Behavior and data contract**

Ordinary values use one local eager instance. Expression inputs create the real class inside one worker actor and reuse it for batches handled by that actor. Expression execution is row-preserving. In Expression and SQL execution, any SQL NULL input produces NULL without invoking the class; eager Python calls retain Python behavior and pass `None` to user code.

**Minimal example**

```python
import vane

@vane.cls(actor_number=1, return_dtype="BIGINT")
class Counter:
    def __init__(self, start=0):
        self.value = start

    def __call__(self, delta):
        self.value += delta
        return self.value

counter = Counter(10)
con = vane.connect()
rel = con.sql("SELECT 1::BIGINT AS delta")
result = rel.select(counter(vane.col("delta")).alias("running_value"))
```

**Restrictions and errors**

v1 accepts only the exact integer `actor_number=1`; `True`, `1.0`, and other integers are rejected. Expression inputs must be positional, not keyword arguments. State is query-scoped, actor-local, ephemeral, not checkpointed, and not exactly-once; actor loss fails the query.

**Related pages**

[Stateful UDF task](/docs/data/guides/custom-python-udfs) · [State and failure semantics](/docs/data/concepts/udfs)

#### `vane.cls.batch`

**Purpose**

Decorate an Arrow batch callable class whose mutable state is reused by one actor during one query.

**Signature**

```python
vane.cls.batch(
    class_: type | None = None,
    *,
    actor_number: int | None = None,
    schema: Mapping[str, Any] | None = None,
    name: str | None = None,
    batch_size: int | None = None,
    row_preserving: bool = False,
    gpus: float | None = 0,
) -> VaneClassBatch | decorator
```

**Parameters**

| Parameter | Contract |
| --- | --- |
| `class_` | Batch callable class to decorate. `None` supports parameterized decorator use. |
| `actor_number` | Signature default is `None`; v1 requires the exact integer `1` explicitly. |
| `schema` | Required non-empty mapping with exactly one output column. |
| `name` | UDF name. `None` derives it from the class, with `class_batch_udf` as fallback. |
| `batch_size` | Optional positive input batch size. |
| `row_preserving` | `False` permits cardinality change as the sole top-level projection; `True` requires equal row counts. |
| `gpus` | Per-actor GPU resource request; default `0`. |

**Returns**

A `VaneClassBatch` wrapper, or a decorator when `class_=None`. Instantiating the wrapper captures constructor arguments; invoking the instance as `instance(**inputs)` returns an eager batch result or an `Expression`.

**Behavior and data contract**

Input mapping keys become Arrow column names. Ordinary values build a local Arrow table and reuse one local instance; Expression values create and reuse the class inside one worker actor. State has the same query-local lifetime as `vane.cls`.

**Minimal example**

```python
import pyarrow as pa
import vane

@vane.cls.batch(
    actor_number=1,
    schema={"normalized": "VARCHAR"},
    row_preserving=True,
)
class Normalizer:
    def __call__(self, table: pa.Table) -> pa.Table:
        values = table.column("text").to_pylist()
        return pa.table({"normalized": [value.strip().lower() for value in values]})

normalizer = Normalizer()
con = vane.connect()
rel = con.sql("SELECT 1 AS id, ' Hello ' AS text")
normalized = normalizer(text=vane.col("text")).alias("normalized")
result = rel.select(vane.col("id"), normalized)
```

**Restrictions and errors**

v1 requires the exact integer `actor_number=1`, and the schema must contain exactly one output column. `row_preserving=False` may change cardinality only as the sole top-level projection. SQL registration requires the decorator to set `row_preserving=True`. State is query-scoped, actor-local, ephemeral, not checkpointed, and not exactly-once.

**Related pages**

[Stateful batch UDF task](/docs/data/guides/custom-python-udfs) · [State and cardinality semantics](/docs/data/concepts/udfs)

## Relation API

Relation methods are specialized Python table stages. They may return a new table shape or expose explicit backend and actor-pool controls. Vane does not expose SQL table-function syntax for these Relation UDFs.

#### `rel.map_batches`

**Purpose**

Transform Arrow batches into a declared table shape, including multi-column output.

**Signature**

```python
rel.map_batches(
    function: Callable[..., Any],
    schema: dict[str, DuckDBPyType] | None = None,
    *,
    batch_size: int | None = None,
    output_batch_size: int | None = None,
    cpus: float | None = None,
    gpus: float | None = None,
    execution_backend: Literal[
        "subprocess_task", "subprocess_actor", "ray_task", "ray_actor"
    ] | None = None,
    actor_number: int | None = None,
    streaming_breaker: bool | None = None,
    queue_depth: int | None = None,
    max_outstanding_batches: int | None = None,
    max_ready_rows: int | None = None,
    generator_backpressure_num_objects: int | None = None,
) -> DuckDBPyRelation
```

**Parameters**

| Parameter | Contract |
| --- | --- |
| `function` | Function or zero-argument callable class that consumes an Arrow table/batch and returns Arrow-compatible table output. |
| `schema` | Declared output column-to-type mapping. The signature defaults to `None`, but the current call contract requires a schema dict. |
| `batch_size` | Optional positive input batch size. |
| `output_batch_size` | Optional positive chunk size for emitted output. |
| `cpus`, `gpus` | Optional non-negative execution resource requests. |
| `execution_backend` | Explicit subprocess/Ray task or actor backend. `None` resolves from the active runner and callable shape. |
| `actor_number` | Positive actor count for an actor backend. |
| `streaming_breaker` | Controls the table-stage streaming boundary; `None` uses the current enabled default. |
| `queue_depth`, `max_outstanding_batches` | Optional positive admission and in-flight limits. |
| `max_ready_rows` | Optional positive bound for ready output rows. |
| `generator_backpressure_num_objects` | Optional positive generator backpressure limit. |

**Returns**

A lazy `DuckDBPyRelation` containing the columns declared by `schema`.

**Behavior and data contract**

The callable receives Arrow batches with the relation's input column names. Output names, types, and shape come from `schema`; multi-column results are flattened into ordinary relation columns. The stage may return a different row count and does not automatically retain input columns.

**Minimal example**

Relation API is needed here because the callable emits a two-column table stage rather than one Expression column.

```python
import duckdb
import pyarrow as pa
import vane

def build_features(table: pa.Table) -> pa.Table:
    ids = table.column("id").to_pylist()
    texts = table.column("text").to_pylist()
    return pa.table({
        "id": ids,
        "normalized": [text.strip().lower() for text in texts],
    })

con = vane.connect()
rel = con.sql("SELECT 1::BIGINT AS id, ' Hello ' AS text")
schema = {
    "id": duckdb.sqltypes.BIGINT,
    "normalized": duckdb.sqltypes.VARCHAR,
}
features = rel.map_batches(build_features, schema=schema, batch_size=1024)
```

**Restrictions and errors**

`schema` must be a dict and callable output must match its names, types, and column count. Batch, queue, and actor counts must be positive; resources must be non-negative. Task backends accept functions and reject actor-only options; actor backends require a callable class plus explicit actor resources.

**Related pages**

[Multi-column Relation UDF task](/docs/data/guides/custom-python-udfs) · [Relation execution model](/docs/data/concepts/udfs)

#### `rel.flat_map`

**Purpose**

Apply a row callable that emits zero, one, or many output rows under a declared schema.

**Signature**

```python
rel.flat_map(
    function: Callable[..., Any],
    schema: dict[str, DuckDBPyType] | None = None,
    *,
    batch_size: int | None = None,
    output_batch_size: int | None = None,
    cpus: float | None = None,
    gpus: float | None = None,
    execution_backend: Literal[
        "subprocess_task", "subprocess_actor", "ray_task", "ray_actor"
    ] | None = None,
    actor_number: int | None = None,
    streaming_breaker: bool | None = None,
    queue_depth: int | None = None,
    max_outstanding_batches: int | None = None,
    max_ready_rows: int | None = None,
    generator_backpressure_num_objects: int | None = None,
) -> DuckDBPyRelation
```

**Parameters**

| Parameter | Contract |
| --- | --- |
| `function` | Row callable returning `None`, one mapping, or an iterable/generator of mappings. |
| `schema` | Declared output mapping. The signature defaults to `None`, but the current call contract requires a schema dict. |
| `batch_size`, `output_batch_size` | Optional positive input and output batch sizes. |
| `cpus`, `gpus` | Optional non-negative execution resource requests. |
| `execution_backend` | Explicit subprocess/Ray task or actor backend; `None` resolves from runner and callable shape. |
| `actor_number` | Positive actor count for actor execution. |
| `streaming_breaker` | Controls the stage boundary; `None` uses the current enabled default. |
| `queue_depth`, `max_outstanding_batches` | Optional positive admission and in-flight limits. |
| `max_ready_rows`, `generator_backpressure_num_objects` | Optional positive output/backpressure limits. |

**Returns**

A lazy `DuckDBPyRelation` with zero, one, or many output rows per input row and columns declared by `schema`.

**Behavior and data contract**

Returning `None` emits no row, returning one mapping emits one row, and yielding mappings emits multiple rows. Only fields declared by the output schema are produced, so pass-through values must be included explicitly.

**Minimal example**

Relation API is needed here because one input row expands into multiple output rows.

```python
import duckdb
import vane

def split_words(row):
    for word in row["text"].split():
        yield {"id": row["id"], "word": word}

con = vane.connect()
rel = con.sql("SELECT 1::INTEGER AS id, 'hello vane' AS text")
schema = {
    "id": duckdb.sqltypes.INTEGER,
    "word": duckdb.sqltypes.VARCHAR,
}
words = rel.flat_map(split_words, schema=schema)
```

**Restrictions and errors**

Every emitted mapping must conform to `schema`. This is an explicit cardinality-changing boundary; it cannot be represented as a registered SQL Expression alias. Invalid backend/callable combinations and non-positive batch, queue, or actor controls fail preflight.

#### `rel.map`

**Purpose**

Apply a row-wise scalar Python UDF to all relation columns and append its typed result.

**Signature**

```python
rel.map(
    map_function: Callable[..., Any],
    *,
    return_type: DuckDBPyType,
    batch_size: int | None = None,
    cpus: float | None = None,
    gpus: float | None = None,
    execution_backend: Literal[
        "subprocess_task", "subprocess_actor", "ray_task", "ray_actor"
    ] | None = None,
    actor_number: int | None = None,
    queue_depth: int | None = None,
    max_outstanding_batches: int | None = None,
    side_effects: bool = False,
) -> DuckDBPyRelation
```

**Parameters**

| Parameter | Contract |
| --- | --- |
| `map_function` | Scalar callable receiving the input row's column values as positional arguments. |
| `return_type` | Required DuckDB type of the appended scalar result. |
| `batch_size` | Optional positive execution batch size. |
| `cpus`, `gpus` | Optional non-negative execution resource requests. |
| `execution_backend` | Explicit subprocess/Ray task or actor backend; `None` resolves from runner and callable shape. |
| `actor_number` | Positive actor count for actor execution. |
| `queue_depth`, `max_outstanding_batches` | Optional positive admission and in-flight limits. |
| `side_effects` | Marks the callable as having side effects; default `False`. It does not add transaction or exactly-once guarantees. |

**Returns**

A lazy `DuckDBPyRelation` containing every input column plus the scalar result in a column named `value`.

**Behavior and data contract**

The method is row-preserving: one scalar result is appended for each input row. The active runner and callable shape choose the backend when none is supplied. Use a following projection to rename or select the `value` column.

**Minimal example**

Relation API is used here because the stage needs an explicit Relation execution backend and its standardized pass-through-plus-`value` contract.

```python
import duckdb
import vane

def plus_one(value):
    return value + 1

con = vane.connect()
rel = con.sql("SELECT 1::INTEGER AS id")
mapped = rel.map(
    plus_one,
    return_type=duckdb.sqltypes.INTEGER,
    execution_backend="subprocess_task",
)
result = mapped.project("id, value AS next_id")
```

**Restrictions and errors**

`return_type` is required. Function callables use task backends; callable classes use actor backends and require valid actor/resource controls. `side_effects=True` records the effect boundary but does not make external writes transactional, idempotent, or exactly-once.

## Shared UDF Constraints

| Area | Lookup rule |
| --- | --- |
| Expression and SQL output types | Declarations accept SQL type strings, `DuckDBPyType`, and supported `pyarrow.DataType` values. Supported families are booleans; signed and unsigned 8/16/32/64-bit integers; `FLOAT`/`DOUBLE`; `VARCHAR`/`BLOB`; decimal128; `DATE`; timezone-naive timestamps; recursive lists; and fixed-size arrays/lists. |
| Unsupported Expression outputs | `TIME`, `INTERVAL`, `UUID`, `ENUM`, `STRUCT`, `MAP`, timezone-aware timestamps, Arrow duration/dictionary, and other unmapped Arrow types are unsupported by this output adapter. Relation `map_batches` schemas are a separate, broader table contract. |
| Placement | Expression UDFs and registered aliases run only in a `SELECT projection`. They are unsupported in `WHERE`, `GROUP BY`, `HAVING`, join conditions, aggregate arguments, `CASE`, `AND`, `OR`, and `COALESCE`. |
| Cardinality | Scalar Expressions and SQL aliases are row-preserving. Batch `row_preserving=True` requires one output row per input and may compose with ordinary columns. `row_preserving=False` may change cardinality only as the sole top-level projection. Relation stages define their own output shape. |
| Stateful class contract | v1 requires exact `actor_number=1`. State is query-scoped, actor-local, ephemeral, not checkpointed, and not exactly-once. It is not keyed, shared, global, or durable. Actor loss fails the query rather than rebuilding empty state. |
| Failure, retries, and effects | Distributed fragment retry or replay can re-execute user code. External effects are not rolled back with a failed SQL query; design callables to be idempotent or deduplicated where effects matter. |
| GPU resources | On Ray, `gpus` becomes a task or actor resource request. A local subprocess does not reserve or isolate a GPU; it inherits process-visible devices and warns for non-zero requests. |
| Concurrency names | Relation actor pools use `actor_number`. AI provider `concurrency` is a separate option that maps to provider actors; it is not a `rel.map_batches` argument. |

For complete setup, validation, and output workflows, use the [Custom Python UDFs Guide](/docs/data/guides/custom-python-udfs). For the reasons behind placement, cardinality, state, and failure rules, use [UDF Concepts](/docs/data/concepts/udfs) and the [Execution Model](/docs/data/concepts/execution-model).
