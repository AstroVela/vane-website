---
title: Embeddings at Scale
---

Use this guide when your POC needs to turn many documents or text records into a table of embeddings that can be inspected, rerun, and handed to a downstream search or retrieval system.

The embedding model call is only one stage. A useful POC path is:

```text
read and filter source rows -> chunk documents -> embed chunks -> validate output -> write an embedding table
```

This guide stops at the embedding table. It is not a vector database, retrieval-ranking, or production RAG guide.

## Start with a small source relation

Begin with real business rows, but keep the first run small. Select stable identifiers and source metadata before any AI call.

```python
import vane

con = vane.connect()

documents = con.sql("""
    select
        cast(document_id as varchar) as document_id,
        source_uri,
        document_type,
        text
    from read_parquet('data/documents/*.parquet')
    where text is not null
      and length(text) > 0
    order by document_id
    limit 100
""")
```

Keep the first relation ordered and limited so you can compare source rows, chunks, and embedding output deterministically. Remove the `limit` only after the chunking and validation checks are correct.

## Chunk with stable IDs

Chunking is a one-to-many transform: one document can produce zero, one, or many chunks. Use `flat_map(...)` for this stage and return all columns needed later.

```python
CHUNK_SIZE = 1200
OVERLAP = 150
CHUNKER_VERSION = "fixed-char-v1"

def chunk_document(row: dict) -> list[dict]:
    document_id = str(row["document_id"])
    text = str(row["text"] or "")
    source_uri = row.get("source_uri")
    document_type = row.get("document_type")

    chunks = []
    start = 0
    chunk_index = 0

    while start < len(text):
        end = min(start + CHUNK_SIZE, len(text))
        chunk_text = text[start:end].strip()

        if chunk_text:
            chunks.append({
                "document_id": document_id,
                "chunk_id": f"{document_id}:{CHUNKER_VERSION}:{chunk_index:06d}",
                "chunk_index": chunk_index,
                "chunk_text": chunk_text,
                "source_uri": source_uri,
                "document_type": document_type,
                "chunker_version": CHUNKER_VERSION,
            })

        if end == len(text):
            break
        start = max(end - OVERLAP, 0)
        chunk_index += 1

    return chunks

chunks = documents.flat_map(
    chunk_document,
    schema={
        "document_id": "VARCHAR",
        "chunk_id": "VARCHAR",
        "chunk_index": "BIGINT",
        "chunk_text": "VARCHAR",
        "source_uri": "VARCHAR",
        "document_type": "VARCHAR",
        "chunker_version": "VARCHAR",
    },
    execution_backend="subprocess_task",
)
```

For a real POC, make the chunking rule deterministic and versioned. If you change tokenization, overlap, cleaning, or page extraction rules, update the chunker version and rerun from the source rows. For model-quality checks, use a tokenizer-aware chunker that matches the embedding model context window; the fixed-character chunker above is only a compact structure example.

Chunk output should preserve:

- `document_id`: the stable source or business document ID.
- `chunk_id`: deterministic from document ID, chunking version, and chunk index or offset.
- `chunk_text`: the exact text sent to the embedding model.
- `source_uri`: the file, object, or source record used for traceability.
- business metadata such as type, tenant, language, date, or case ID when it helps review failures.

## Validate chunks before embedding

Do cheap checks before sending data to a model or hosted API.

```python
chunk_table = chunks.to_arrow_table()

if chunk_table.num_rows == 0:
    raise ValueError("chunking produced no rows")

chunk_ids = chunk_table["chunk_id"].to_pylist()
if len(chunk_ids) != len(set(chunk_ids)):
    raise ValueError("chunk_id values must be unique")

empty_chunks = [
    chunk_id
    for chunk_id, text in zip(chunk_ids, chunk_table["chunk_text"].to_pylist(), strict=False)
    if not text
]
if empty_chunks:
    raise ValueError(f"empty chunks: {empty_chunks[:5]}")
```

Also inspect a few chunks manually. Check that boundaries, overlap, metadata, and language handling match the POC question before increasing the sample size.

## Choose a provider

Choose the smallest provider setup that answers the POC question.

| Choice | Use it when | Watch for |
| --- | --- | --- |
| Hosted API | You want managed models and can use external calls. | Credentials, cost, latency, rate limits, retries, and provider-specific dimensions. |
| Local model | You want local execution or private data handling with installed model libraries. | Dependency versions, model download, CPU/GPU memory, and model initialization time. |
| Custom UDF | You need full control over IDs, retries, status columns, or embedding schema in one step. | More code to maintain; validate schema and errors carefully. |

AI helper methods such as `embed_text(...)` are convenient, but their output relation contains the configured embedding column. If your final table needs IDs and metadata, explicitly combine helper output with the chunk rows after validating row counts and ordering, or use a custom UDF that returns the full schema.

## Embed with a hosted API

Hosted APIs are useful for quick model access, but tune concurrency against your account limits.

```python
embedding_only = chunks.embed_text(
    "chunk_text",
    provider="openai",
    model="text-embedding-3-small",
    output_column="embedding",
    batch_size=64,
    concurrency=4,
    execution_backend="subprocess_actor",
)
```

Start with conservative `batch_size` and `concurrency`. Increase one setting at a time and record provider errors, timeouts, and rate-limit responses.

## Embed with a local model

Local models avoid hosted API calls but add local dependencies and model-loading cost. Actor backends are usually the right first choice because the model can be initialized once per actor and reused across batches.

```python
embedding_only = chunks.embed_text(
    "chunk_text",
    provider="transformers",
    model="sentence-transformers/all-MiniLM-L6-v2",
    output_column="embedding",
    batch_size=64,
    execution_backend="subprocess_actor",
)
```

For strict GPU placement, custom retry behavior, or a single UDF that returns identifiers, text, status, errors, and embeddings together, use a callable class with `map_batches(...)` instead of the high-level helper.

## Combine and validate output

When you use `embed_text(...)`, combine the embedding column with the source chunk rows only after validation.

```python
import pyarrow as pa

PROVIDER = "openai"
MODEL = "text-embedding-3-small"
RUN_ID = "embedding-poc-run-001"

chunk_table = chunks.to_arrow_table()
embedding_table = embedding_only.to_arrow_table()

if embedding_table.num_rows != chunk_table.num_rows:
    raise ValueError("embedding row count does not match chunk row count")

embeddings = embedding_table["embedding"]
if chunk_table.num_rows and not embeddings[0].as_py():
    raise ValueError("first embedding is empty")

embedded_table = (
    chunk_table
    .append_column("embedding", embeddings)
    .append_column("embedding_provider", pa.array([PROVIDER] * chunk_table.num_rows))
    .append_column("embedding_model", pa.array([MODEL] * chunk_table.num_rows))
    .append_column("embedding_run_id", pa.array([RUN_ID] * chunk_table.num_rows))
)

embedded = con.from_arrow(embedded_table)
```

If you cannot make row order stable, do not rely on positional recombination. Write a custom embedding UDF that returns the full output schema, including IDs and error/status columns, in one relation.

## Output schema

Write an embedding table that a downstream system and a reviewer can understand without re-reading the whole pipeline.

| Column | Purpose |
| --- | --- |
| `document_id` | Stable source or business document ID. |
| `chunk_id` | Stable, deterministic chunk ID. |
| `chunk_index` | Numeric order of the chunk within the document. |
| `chunk_text` | Text sent to the embedding model. |
| `embedding` | Vector returned by the provider or local model. |
| `source_uri` | Original file, object, or source record. |
| `chunker_version` | Reproducibility metadata for the chunking rule. |
| `embedding_provider` | Provider name, such as `openai` or `transformers`. |
| `embedding_model` | Exact model identifier used for the run. |
| `embedding_run_id` | Run identifier for repeatability and review. |

Add business metadata columns that help you debug bad chunks or explain a sample to domain reviewers. Avoid silently dropping identifiers even if the downstream vector store does not require them.

## Write a reviewable result

Write to a run-specific path so you can compare runs without overwriting earlier POC output.

```python
embedded.write_parquet(f"output/embeddings/{RUN_ID}/")
```

For distributed writes, make sure every worker can access the output filesystem or object store.

## Scale after the sample is correct

Move from local sample to Ray or larger batches only after chunk and embedding validation is passing on a small representative slice.

```python
vane.configure(runner="ray")

embedding_only = chunks.embed_text(
    "chunk_text",
    provider="transformers",
    model="sentence-transformers/all-MiniLM-L6-v2",
    output_column="embedding",
    batch_size=128,
    concurrency=8,
    execution_backend="ray_actor",
)
```

Every Ray worker needs access to the provider libraries, local model files or hosted API credentials, source data, and output path. Keep the same chunking configuration when you scale; otherwise you are measuring a different pipeline.

## Tuning checklist

- Filter and project before chunking so you do not embed irrelevant rows.
- Keep chunk IDs deterministic and preserve `document_id`, `chunk_text`, `source_uri`, provider, model, and run metadata.
- Validate 10-100 representative rows before increasing data size, `batch_size`, `concurrency`, or Ray resources.
- Increase `batch_size` until memory, provider payload size, or latency becomes the bottleneck.
- Increase `concurrency` only when local workers, GPU capacity, or hosted provider quota have headroom.
- Use actor backends for local models so model initialization is reused across batches.
- Use Ray when one process is no longer enough for scan, embedding, or write throughput.
- Estimate output size early: vector dimension times chunk count can dominate storage and write time.
- Keep run-specific output paths and record chunker version, provider, model, batch size, concurrency, backend, input row count, chunk count, and failures.

## Boundary

This guide produces a reproducible embedding dataset for a POC. It does not cover vector database schema design, approximate nearest-neighbor indexing, retrieval evaluation, online serving, access control, or production RAG operations. Use the embedding table as the handoff point to those downstream systems.
