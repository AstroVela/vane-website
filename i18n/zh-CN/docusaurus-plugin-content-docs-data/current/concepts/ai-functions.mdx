title: AI 函数
---

Vane Data AI Function 有两种语义输出模型：**Expression API** 和 **Relation API**。它们不是三种语法入口。需要在源列旁为每行增加一个保持行数的 AI 结果时，先从 Expression 内的 SQL 入口开始。

两种模型使用同一个 provider 与 UDF 执行层。区别在于操作产出一个 projection 结果，还是一次专用表转换。

## Expression API

Expression API 在 `SELECT` projection 中生成一个保持行数的 AI 列。SQL 是推荐的默认入口：稳定标识和源字段可以明确地保留在模型结果旁边。

### SQL 入口（推荐）

当每一输入行旁只需要一个模型结果时，使用 SQL `ai_prompt` 或 `ai_embed`。下面的短 projection 会保留生成每个 response 的标识与 prompt：

```python
import vane

con = vane.connect()
source = con.sql("""
    SELECT *
    FROM (VALUES
        (1, 'Summarize the refund request'),
        (2, 'Summarize the shipping delay')
    ) AS t(document_id, prompt)
""")

reviewed = con.sql("""
    SELECT
        document_id,
        prompt,
        ai_prompt(prompt) AS response
    FROM source
""")
```

SQL 调用仍属于 Expression API。它的 provider options 是常量 planning 输入，`document_id` 和 `prompt` 则继续作为普通 relation 列流动。

### Python 入口

Python Expression 是语义等价的语法，具有相同的保持行数 projection 语义。当周围的 projection 已经用 Python expression 组合时，它更方便：

```python
projected = source.select(
    vane.col("document_id"),
    vane.col("prompt"),
    vane.ai.prompt(vane.col("prompt")).alias("response"),
)
```

从 SQL 改成 Python 不会改变输出模型。两种形式都会增加一个 AI 结果列，并通过显式 projection 保留源字段。

## Relation API

只有当某项能力需要表转换而不是普通的单个 projection 结果时，才转到 Relation AI：

- `rel.embed_text` 通过 `max_chunk_chars` 提供内置分块。
- `rel.classify_text` 提供内置分类。
- `rel.prompt` 通过 `return_format` 提供结构化输出，通过 `image_columns` 提供多模态输入，并通过 `output_column` 命名 Relation 输出。
- Relation 方法可以显式选择 `execution_backend`，以及 provider `concurrency` 或 `actor_number` 控制。

内置分类需要 Relation API，因此最后这个示例使用 Relation 方法，而不会假装分类是 SQL table function：

```python
classified = source.classify_text(
    "prompt",
    labels=["refund", "shipping", "other"],
    output_column="label",
)
```

Vane 不提供直接的 SQL Relation 或 table-function API。SQL 可以在 Relation 调用之前过滤、projection 输入，也可以在调用之后消费结果，但专用 Relation 方法本身目前仍通过 Python 调用。

## Provider 与 Runner 生命周期

Provider descriptor 会跨越 plan 边界，而不会把 live client 或 model 放进 plan。所选 client 或 model 会延迟到 worker 才实例化并复用，复用范围是每个 actor；这样既不会为没有执行的行初始化资源，也能在多个 batch 之间摊薄 setup 成本。

对于 SQL 和 Python Expression，active runner 会在 query planning 时选择 actor backend：local 执行使用 subprocess actor，Ray run 使用 Ray actor。Relation 方法还可以显式选择 backend。Provider 默认值和精确参数应留在 [AI Function API 参考](/zh-CN/docs/data/reference/ai-api)，而不是放进这里的概念性生命周期说明。

## 凭据与外部副作用

凭据应放在 worker 环境中，因为 provider client 是在 worker 上创建的。SQL option binding 会拒绝类似凭据的字段，因此 API key、token 和 secret 不应嵌入 SQL plan。

Provider 调用是在 SQL 事务之外发生的外部副作用，不提供 exactly-once：失败或 retry 的 query 可能重复 request，transaction rollback 也无法撤销已经完成的 provider 调用。请让稳定 ID 随 projection 一起传递，保留可审查的输入与输出，并在重复调用会造成影响时使用幂等的下游写入。

## 选择模型

- 每一输入行只需要一个 prompt response 或 embedding 时，首先选择 SQL Expression。
- 需要 Python 组合方式但仍是同一个保持行数结果时，选择 Python Expression。
- 只有在需要内置分块或分类、结构化或多模态 prompt、命名 Relation 输出或显式 Relation 执行控制时，才选择 Relation。

精确签名和 provider 参数规则请查阅 [AI Function API 参考](/zh-CN/docs/data/reference/ai-api)。包含 setup、校验和输出处理的完整任务请使用 [AI Function 指南](/zh-CN/docs/data/guides/ai-functions)。
