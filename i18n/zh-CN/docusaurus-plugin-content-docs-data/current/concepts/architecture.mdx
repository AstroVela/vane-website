title: 架构
---

Vane Data 是一个多模态原生数据引擎。它提供兼容 DuckDB 的 SQL 与 Python 接口，支持 Expression 和 Relation 两种 UDF 与 AI 操作，并且可以在本地或通过 Ray 执行选定工作。

设计目标很简单：让 pipeline 保持表格形态，同时让 SQL、Python 库、模型推理和分布式执行处在同一个 workflow 中。

## 系统视图

一个 Vane Data pipeline 有四个用户可见层：

1. API 层：`import vane`、`vane.connect()`、兼容 DuckDB 的 SQL、Expression 与 Relation API、UDF 和 AI Function。
2. relation pipeline：SQL 操作、投影、过滤、连接、自定义 Expression 结果、Relation 转换和写出都组合为 relation 步骤。
3. 执行 runner：本地执行用于开发和单节点任务；Ray runner 负责分发适合的扫描、写出、UDF 和 AI 工作。
4. UDF 运行时：Python 函数、可调用类和 AI provider 通过 task 或 actor backend 运行。

在 API 层，Expression 与 Relation 描述操作返回什么。Expression 为 projection 贡献一个有类型的结果，Relation 则负责一次表形转换。SQL 是 Expression UDF 和 AI Function 的推荐入口；Python 同时提供 Expression 语法与 Relation 方法。

这种分层让 pipeline 可以先从本地开始；当数据量、模型吞吐量或资源放置需要时，再把选定工作迁移到 Ray。

## 公开用户接口

主要公开入口如下：

- `import vane`
- `vane.connect()`
- `vane.configure(...)`
- `vane.current_config()`
- `vane.env`
- 通过 `con.sql(...)` 和 `con.execute(...)` 使用兼容 DuckDB 的 SQL
- 使用 `vane.attach_function(...)` 注册、使用 `vane.detach_function(...)` 移除的 SQL Expression UDF alias
- SQL AI Expression：`ai_prompt(...)` 和 `ai_embed(...)`
- Python Expression 输入：`vane.col(...)`、`vane.lit(...)` 和 `vane.sql_expr(...)`
- Python Expression UDF：`vane.func`、`vane.func.batch`、`vane.cls` 和 `vane.cls.batch`
- Python AI Expression：`vane.ai.prompt(...)` 和 `vane.ai.embed(...)`
- 兼容 DuckDB 的 relation API
- `rel.map_batches(...)`
- `rel.flat_map(...)`
- `rel.map(...)`
- `rel.embed_text(...)`
- `rel.classify_text(...)`
- `rel.prompt(...)`

当自定义工作属于 projection，通常需要在每一输入行旁增加一个有类型的结果时，选择 Expression。当操作负责输出列或行数时，选择 Relation。Vane 当前不提供直接的 SQL Relation 或 table-function API；SQL 可以在 Python Relation 调用前准备 relation，也可以在调用后继续消费其结果。

为了兼容已有代码，DuckDB 风格的 import 仍然可用；但新的 Vane Data 项目应优先使用 `import vane`。

## runner 模式

Vane Data 有两种主要执行模式。

本地模式：

- 在本机的 driver 进程中运行 relation 工作。
- 适合作为开发、测试、轻量 SQL 和单节点 UDF 任务的起点。
- 使用本地 subprocess task 或 actor worker 执行 UDF 和 AI 工作。

Ray 模式：

- 通过 `vane.configure(runner="ray")` 或 `VANE_RUNNER=ray` 选择。
- 使用 Ray 执行分布式扫描、分布式写出、分布式 UDF，以及基于 actor 的模型 worker。
- 要求每个 worker 都能访问 pipeline 所需的相同 Python 依赖、存储系统、模型文件和凭证。

如果当前进程已经初始化 Ray，runner 选择可能会与该环境相互影响。为了让生产脚本行为可预测，请在创建连接和运行 pipeline 前显式配置目标 runner。

## UDF 执行层

UDF 通过两种公开输出契约进入同一执行层。Expression UDF 会增加一个有类型的 projection 结果：推荐的 SQL 路径调用通过 `vane.attach_function(...)` 注册的 alias，`vane.func` 和 `vane.cls` 则提供 Python Expression 语法。`rel.map_batches(...)`、`rel.flat_map(...)` 和 `rel.map(...)` 等 Relation UDF 负责一个表形阶段，并返回它声明的输出。

根据可调用对象形态和资源需求选择 backend：

| Backend | 可调用对象形态 | 适用场景 |
| --- | --- | --- |
| `subprocess_task` | Python 函数 | 本地无状态 batch 转换 |
| `subprocess_actor` | 可调用类 | 本地模型或客户端复用 |
| `ray_task` | Python 函数 | 分布式无状态转换 |
| `ray_actor` | 可调用类 | 分布式模型复用或 GPU 支持的推理 |

task backend 适用于没有长期状态的函数。actor backend 会在每个 actor 中构造一个可调用类实例，并在多个 batch 之间复用；不同 actor 各自保留独立状态。active runner 会把 Expression 工作映射到本地或 Ray backend，Relation 方法还可以提供显式执行控制。

精确签名、基数例外、放置规则和资源限制请查阅 UDF API Reference。

## AI Function 层

AI Function 使用相同的 Expression 与 Relation 输出契约，也共用同一套 worker 执行模型：

- 当一个 AI 结果属于 `SELECT` projection 时，推荐使用 SQL `ai_prompt(...)` 和 `ai_embed(...)`。
- Python `vane.ai.prompt(...)` 和 `vane.ai.embed(...)` 为同一个 projection 模型提供另一种 Expression 语法。
- `rel.prompt(...)`、`rel.embed_text(...)` 和 `rel.classify_text(...)` 等 Relation 方法支持结构化或多模态输出、分块、分类和显式 Relation 执行控制等表形能力。

AI provider 配置以可序列化数据进入计划。实际客户端或模型会延迟到 worker 中创建，并可由所在 actor 跨 batch 复用。由 provider 支持的常见操作可以使用 AI Function；当你需要完全控制模型加载、batch 处理、输出 schema 或 provider 行为时，使用自定义 UDF。

## 配置模型

代码配置：

```python
import vane

vane.configure(runner="ray")
```

环境配置：

```bash
export VANE_RUNNER=ray
```

`vane.configure(...)` 会通过进程环境应用已注册的 `VANE_*` 设置。使用 `vane.current_config()` 或 `vane.env.as_dict()` 查看公开配置快照。

## worker 环境

worker 需要具备其执行工作所需的运行时上下文。实践中，请确认 worker 可以访问：

- UDF 和 AI provider 使用的 Python 模块。
- 对象存储凭证和端点设置。
- 模型文件或模型下载位置。
- 相关 `VANE_*`、`AWS_*`、`S3FS_*`、`DUCKDB_*`、`PYTHONPATH` 和 `RAY_ADDRESS` 环境变量值。

AI provider 客户端在 worker 中创建，因此 provider 凭据也必须在那里可用。API key、token 和 secret 应优先通过 worker 环境变量提供；SQL AI options 会拒绝类似凭据的字段，避免把凭据写入计划。

在创建连接或运行 pipeline 之前设置所需环境。

## 数据流动

常见数据路径：

- SQL 将本地或远程数据读取为兼容 DuckDB 的 relation。
- Expression 执行把所需参数列发送给 UDF 或 AI worker，再把有类型的结果与周围的 projection 组合。
- Relation 方法把 Arrow batch 流式传入 Python worker，并返回声明的表形输出。当下游阶段仍然需要源列时，应把这些列纳入 Relation 输出。
- Ray 模式可以通过 Ray worker 物化分区。
- 写出会根据 runner 和目标，选择兼容 DuckDB 的 relation 写出路径或 Vane Data 的 relation 写出路径。

把过滤和列投影放在昂贵的 Python 或模型阶段之前。这样可以减少进入 Python 的数据量；在 Ray 模式下，也可以减少集群内的数据移动。
