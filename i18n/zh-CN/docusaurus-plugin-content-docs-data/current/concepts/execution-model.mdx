---
title: 执行模型
---

Vane Data 的 pipeline 是以 relation 为核心的 pipeline。每一步都描述一次表格转换；只有在通过显示、拉取、转换或写出结果来物化 relation 时，实际工作才会开始执行。

借助这种模型，Vane Data 可以组合三类工作：

- 用兼容 DuckDB 的 SQL 表达关系型处理。
- 用 UDF 表达 Python 处理。
- 用 AI Function 的 relation 方法表达 AI 处理。

## Relation 是 pipeline 的基本单元

relation 表示一个表格形态的计算。你可以用 SQL 构建 relation：

```python
rel = con.sql("""
    select id, text
    from read_parquet('data/documents/*.parquet')
    where text is not null
""")
```

然后添加 Python 或 AI 阶段：

```python
features = rel.map_batches(
    model_batch,
    schema={"id": "BIGINT", "label": "VARCHAR"},
    batch_size=64,
    execution_backend="subprocess_task",
)
```

每个阶段都会返回另一个 relation，因此 pipeline 仍然可组合。UDF 阶段返回 UDF 产出的列；如果后续阶段需要标识符和元数据，也要一并返回。

## 执行从物化开始

与执行相比，构建 relation 的成本很低。只有调用消费操作时，工作才会开始，例如：

- `show()`
- `fetchall()`
- `to_arrow_table()`
- `to_record_batch()`
- `write_parquet(...)`
- `to_table(...)`

这样 Vane Data 可以先把 SQL、Python 和 AI 阶段放在一起，再决定如何执行 pipeline。

## Arrow batch 连接 SQL 与 Python

`map_batches` 是 relation 执行与 Python 代码之间的主要边界。

- 输入以 Arrow 数据的形式进入。
- Python 可调用对象返回 Arrow 数据。
- 输出 schema 在 relation 边界声明。
- `batch_size` 控制每次 UDF 调用处理的行数。

这样 schema 变更会保持显式，而不是隐藏在 Python 对象内部。

## 本地执行

本地执行适合作为以下场景的起点：

- 开发和调试。
- 单节点预处理。
- 小中型数据 pipeline。
- 在扩展规模之前校验 UDF 正确性。

优先使用本地 UDF backend：

- 对无状态函数使用 `subprocess_task`。
- 对具有可复用状态的可调用类使用 `subprocess_actor`。

## Ray 执行

使用以下方式选择 Ray runner：

```python
import vane

vane.configure(runner="ray")
```

或：

```bash
export VANE_RUNNER=ray
```

Ray 执行适用于：

- 大文件扫描。
- 分布式写入。
- 使用 `ray_task` 的分布式无状态 UDF。
- 使用 `ray_actor` 的有状态模型 worker。
- 跨 Ray 集群的 CPU 和 GPU 资源放置。

为了获得可预测结果，请确认每个 Ray worker 都能导入相同的 Python 模块、访问相同的存储系统，并看到 UDF 所需的凭证或模型文件。

## UDF 调度

根据可调用对象形态和资源模式选择 backend：

| 工作负载 | 可调用对象形态 | Backend |
| --- | --- | --- |
| 简单的本地 batch 转换 | Python 函数 | `subprocess_task` |
| 本地模型或客户端复用 | 可调用类 | `subprocess_actor` |
| 分布式无状态转换 | Python 函数 | `ray_task` |
| 分布式模型复用或 GPU 推理 | 可调用类 | `ray_actor` |

对不需要长期状态的函数使用 task backend。当初始化成本较高，且应在每个 worker 或 actor 上只发生一次时，使用 actor backend。

## 流式阶段与阻塞阶段

许多 relation 阶段可以让 batch 流式穿过 pipeline。有些阶段则需要先收集或协调数据，才能产生输出。

常见的阻塞或部分阻塞阶段包括：

- 全局聚合。
- 需要数据交换的 join。
- 排序和 shuffle。
- 写入。
- 引入执行边界的 UDF 阶段。

做性能调优时，把过滤和列投影放在昂贵的 UDF 与模型调用之前。这样可以减少进入 Python 的数据；在 Ray 模式下，也会减少集群内的数据移动。

## 操作检查清单

将 pipeline 从本地执行迁移到 Ray 之前：

1. 在有代表性的样本上本地运行 pipeline。
2. 显式声明 UDF 输出 schema。
3. 先在本地调优 `batch_size`。
4. 只有当分布式执行确实有帮助时，才把无状态函数迁移到 `ray_task`。
5. 只有当需要 actor 复用、GPU 或集群资源放置时，才把有状态模型类迁移到 `ray_actor`。
6. 确认存储、凭证、Python 依赖和模型文件对每个 Ray worker 都可见。
7. 度量吞吐量、内存使用、重试次数和输出行数。
