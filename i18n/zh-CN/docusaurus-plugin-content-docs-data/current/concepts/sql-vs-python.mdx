---
title: SQL 与 Python
---

Vane Data 面向需要同时使用 SQL 与 Python 的关系型 workflow。两者可以在同一条 pipeline 中协作，而不是把关系型处理和 Python 逻辑割裂成两套系统。

关系型处理交给 SQL。某个步骤需要 Python 库、自定义状态、模型调用或行展开时，使用 Python UDF。操作符合内置 embedding、分类或提示词模式时，使用 AI Function。

## 决策指南

| 需求 | 使用 | 原因 |
| --- | --- | --- |
| 过滤行、投影列、连接表、聚合指标 | SQL | 声明式、可检查，并且可优化 |
| 解码媒体、调用 Python 库、运行自定义模型 | Python UDF | 让非 SQL 逻辑保持显式且可测试 |
| 生成 embedding、文本分类、用提示词调用模型 | AI Function | 提供针对 provider 的封装，并继承 Vane Data 的执行控制 |
| 复用初始化成本较高的模型或客户端 | Actor UDF backend | 初始化一次，然后处理多个 batch |
| 扩展无状态转换 | `ray_task` | 将函数调用分发到 Ray worker |
| 扩展模型推理或 GPU 工作负载 | `ray_actor` | 结合 actor 复用与 Ray 资源调度 |

## 使用 SQL 处理关系型工作

天然属于表转换的操作应由 SQL 负责：

- 读取 Parquet、CSV、JSON 以及其他 DuckDB 支持的数据源。
- 在昂贵步骤之前过滤行。
- 只选择下一阶段需要的列。
- 连接元数据表。
- 聚合质量指标。
- 写出干净的表格化输出。

```python
raw = con.sql("""
    select id, uri, text
    from read_parquet('data/documents/*.parquet')
    where text is not null
      and length(text) > 80
""")
```

## 使用 Python UDF 处理自定义工作

需要运行时库或自定义逻辑的操作应由 Python 负责：

- batch 模型推理。
- 图片、音频、视频或 PDF 解码。
- 使用 Python 包做文本规范化。
- 调用私有服务或客户端。
- 每个输入行生成多行输出。
- 返回带有显式 schema 的自定义列。

```python
import pyarrow as pa

def split_sentences(batch: pa.Table) -> pa.Table:
    ids = batch.column("id").to_pylist()
    texts = batch.column("text").to_pylist()
    out_id: list[int] = []
    out_sentence: list[str] = []

    for doc_id, text in zip(ids, texts):
        for sentence in str(text).split("."):
            sentence = sentence.strip()
            if sentence:
                out_id.append(doc_id)
                out_sentence.append(sentence)

    return pa.table({"id": out_id, "sentence": out_sentence})

sentences = raw.map_batches(
    split_sentences,
    schema={"id": "BIGINT", "sentence": "VARCHAR"},
    batch_size=512,
    execution_backend="subprocess_task",
)
```

## 使用 AI Function 处理常见 AI 任务

当操作匹配内置 provider 模式时，AI Function 是更合适的选择：

```python
embedded = raw.embed_text(
    "text",
    provider="transformers",
    model="sentence-transformers/all-MiniLM-L6-v2",
    output_column="embedding",
)
```

AI Function 会返回一个包含已配置输出列的 relation。如果下游阶段还需要源 ID、文本或元数据，请使用自定义 UDF 返回完整 schema，或者在经过校验的步骤中显式地将 AI Function 的输出与源行组合起来。

## 推荐的 pipeline 形态

先用 SQL 完成扫描、过滤和投影；只有在昂贵或自定义步骤中切换到 Python；随后回到 SQL 做校验和输出。

```python
raw = con.sql("""
    select id, uri, text
    from read_parquet('s3://bucket/raw/*.parquet')
    where text is not null
""")

features = raw.map_batches(
    ModelBatch,
    schema={"id": "BIGINT", "label": "VARCHAR"},
    batch_size=64,
    execution_backend="ray_actor",
    gpus=1,
    concurrency=4,
)

features.to_table("features")

con.sql("""
    select label, count(*) as n
    from features
    group by label
    order by n desc
""").show()
```

将 `ModelBatch` 替换为一个会加载并复用模型的可调用类。

## Python 优先的工作负载

在以下场景中选择 Python 优先路径：

- 模型吞吐量是主要瓶颈。
- 模型或服务客户端需要 actor 复用。
- 转换用 batch 函数表达比 SQL 更清晰。
- 你正在调优 `batch_size`、`gpus`、`cpus`、`concurrency` 和 Ray 资源。

推荐路径：

1. 使用 SQL 做扫描和剪枝。
2. 使用 `map_batches` 执行模型或解码工作。
3. 初始化成本较高时，在本地使用 `subprocess_actor`。
4. 只有在本地 actor 路径正确后，才切换到 `ray_actor`。
5. 使用有代表性的数据进行调优。

## SQL 优先的工作负载

在以下场景中选择 SQL 优先路径：

- 数据可以放在单机或单个分析引擎中处理。
- 转换需要易于检查和审计。
- Python UDF 只是一个小型补充增强步骤。
- 输出会被另一个 SQL 系统、数据仓库、湖仓或 BI pipeline 消费。

推荐路径：

1. 将数据保持在 relation、表和视图中。
2. 只在非 SQL 步骤中使用 Python。
3. 写出 Parquet 或其他 DuckDB 支持的输出格式。
4. 通过下游系统自身的数据导入路径完成交接。

## 经验法则

如果操作可以用 SQL 清晰表达，就留在 SQL 中。如果它需要 Python 状态、外部库、模型执行或 provider 调用，就放到 UDF 或 AI Function 中，并保持输入和输出 schema 显式。
