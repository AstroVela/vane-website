---
title: SQL vs Python
---

Vane Data 面向同时需要 SQL 与 Python 的 relation workflow。两者可以在同一条 pipeline 中协作，而不是把关系型处理和 Python 逻辑割裂成两套系统。

关系型处理交给 SQL。某个步骤需要 Python library、自定义状态、model call 或 row expansion 时，使用 Python UDF。操作符合内置 embedding、classification 或 prompting 模式时，使用 AI helper。

## 决策指南

| 需求 | 使用 | 原因 |
| --- | --- | --- |
| 过滤行、投影列、join 表、聚合指标 | SQL | 声明式、可检查，并且可优化 |
| 解码 media、调用 Python library、运行自定义 model | Python UDF | 让非 SQL 逻辑保持显式且可测试 |
| 生成 embeddings、文本分类、prompt model | AI helper | 提供 provider-specific wrapper，并继承 Vane Data 的执行控制 |
| 复用初始化成本较高的 model 或 client | Actor UDF backend | 初始化一次，然后处理多个 batch |
| 扩展无状态 transform | `ray_task` | 将 function call 分发到 Ray worker |
| 扩展 model inference 或 GPU 工作负载 | `ray_actor` | 结合 actor 复用与 Ray resource placement |

## 使用 SQL 处理关系型工作

天然属于 table transformation 的操作应由 SQL 负责：

- 读取 Parquet、CSV、JSON 以及其他 DuckDB 支持的数据源。
- 在昂贵步骤之前过滤行。
- 只选择下一阶段需要的列。
- Join metadata table。
- 聚合质量指标。
- 写出干净的 tabular output。

```python
raw = con.sql("""
    select id, uri, text
    from read_parquet('data/documents/*.parquet')
    where text is not null
      and length(text) > 80
""")
```

## 使用 Python UDF 处理自定义工作

需要 runtime library 或自定义逻辑的操作应由 Python 负责：

- Batch model inference。
- 图片、音频、视频或 PDF 解码。
- 使用 Python package 做文本规范化。
- 调用私有 service 或 client。
- 每个输入行生成多行输出。
- 返回带有显式 schema 的自定义列。

```python
import pyarrow as pa

def split_sentences(batch: pa.Table) -> pa.Table:
    ids = batch.column("id").to_pylist()
    texts = batch.column("text").to_pylist()
    out_id: list[int] = []
    out_sentence: list[str] = []

    for doc_id, text in zip(ids, texts):
        for sentence in str(text).split("."):
            sentence = sentence.strip()
            if sentence:
                out_id.append(doc_id)
                out_sentence.append(sentence)

    return pa.table({"id": out_id, "sentence": out_sentence})

sentences = raw.map_batches(
    split_sentences,
    schema={"id": "BIGINT", "sentence": "VARCHAR"},
    batch_size=512,
    execution_backend="subprocess_task",
)
```

## 使用 AI helper 处理常见 AI 任务

当操作匹配内置 provider pattern 时，AI helper 是更合适的选择：

```python
embedded = raw.embed_text(
    "text",
    provider="transformers",
    model="sentence-transformers/all-MiniLM-L6-v2",
    output_column="embedding",
)
```

helper 会返回一个包含已配置输出列的 relation。如果 downstream stage 还需要 source ID、文本或 metadata，请使用自定义 UDF 返回完整 schema，或者在经过校验的步骤中显式地将 helper output 与 source row 组合起来。

## 推荐的 pipeline 形态

先用 SQL 完成扫描、过滤和投影；只有在昂贵或自定义步骤中切换到 Python；随后回到 SQL 做校验和输出。

```python
raw = con.sql("""
    select id, uri, text
    from read_parquet('s3://bucket/raw/*.parquet')
    where text is not null
""")

features = raw.map_batches(
    ModelBatch,
    schema={"id": "BIGINT", "label": "VARCHAR"},
    batch_size=64,
    execution_backend="ray_actor",
    gpus=1,
    concurrency=4,
)

features.to_table("features")

con.sql("""
    select label, count(*) as n
    from features
    group by label
    order by n desc
""").show()
```

将 `ModelBatch` 替换为一个会加载并复用 model 的 callable class。

## Python-first workload

在以下场景中选择 Python-first 路径：

- Model throughput 是主要瓶颈。
- model 或 service client 需要 actor 复用。
- transformation 用 batch function 表达比 SQL 更清晰。
- 你正在调优 `batch_size`、`gpus`、`cpus`、`concurrency` 和 Ray resource。

推荐路径：

1. 使用 SQL 做扫描和剪枝。
2. 使用 `map_batches` 执行 model 或解码工作。
3. 初始化成本较高时，在本地使用 `subprocess_actor`。
4. 只有在本地 actor 路径正确后，才切换到 `ray_actor`。
5. 使用有代表性的数据进行调优。

## SQL-first workload

在以下场景中选择 SQL-first 路径：

- 数据可以放在单机或单个 analytical engine 中处理。
- transformation 需要易于检查和审计。
- Python UDF 只是一个小型 enrichment step。
- 输出会被另一个 SQL system、warehouse、lakehouse 或 BI pipeline 消费。

推荐路径：

1. 将数据保持在 relation、table 和 view 中。
2. 只在非 SQL 步骤中使用 Python。
3. 写出 Parquet 或其他 DuckDB 支持的输出格式。
4. 通过 downstream system 自身的 ingestion path 完成交接。

## 经验法则

如果操作可以用 SQL 清晰表达，就留在 SQL 中。如果它需要 Python state、外部 library、model execution 或 provider call，就放到 UDF 或 AI helper 中，并保持输入和输出 schema 显式。
