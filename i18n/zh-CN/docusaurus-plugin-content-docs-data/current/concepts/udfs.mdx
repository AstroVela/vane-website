Vane Data UDF 有两种语义输出模型：**Expression API** 和 **Relation API**。它们不是三种语法入口。SQL 与 Python 是同一个 Expression 模型的两种入口；Relation 则是用于表形转换的独立模型。

这一区分关注查询阶段会产出什么。先根据输出契约选择模型，再选择语法与执行控制。

## Expression API

Expression API 在 `SELECT` projection 中增加一个有类型的输出列，并且通常保持行数。SQL 是推荐的默认入口，因为它能在一条关系语句中同时清楚呈现过滤、projection、源列和 UDF 结果。

### SQL 入口（推荐）

用 `vane.attach_function` 注册 Python callable，会为 Expression API 提供一个 SQL 名称；注册仍属于该模型。注册后的 alias 是一个 projection expression。SQL alias 在 v1 中始终保持行数。

```python
import vane

con = vane.connect()
source = con.sql("""
    SELECT *
    FROM (VALUES
        (1, '  Hello  '),
        (2, ' Vane Data ')
    ) AS t(id, text)
""")

@vane.func(return_dtype="VARCHAR")
def normalize_text(value):
    return str(value).strip().lower()

vane.attach_function(
    normalize_text,
    alias="normalize_text_sql",
    connection=con,
    parameters=["VARCHAR"],
)
normalized = con.sql("""
    SELECT id, text, normalize_text_sql(text) AS normalized_text
    FROM source
""")
normalized.show()
vane.detach_function("normalize_text_sql", connection=con)
```

在 projection 中保留 `id` 和 `text`，可以直接看出保持行数的契约，并让普通 SQL 继续负责周围的表逻辑。

### Python 入口

Python Expression 是同一个 projection 模型的另一种语法。调用收到 `vane.col(...)` 时会构建 lazy expression，而不是立即执行 Python 函数：

```python
projected = source.select(
    vane.col("id"),
    vane.col("text"),
    normalize_text(vane.col("text")).alias("normalized_text"),
)
```

通常的契约是生成一个有类型、保持行数的输出列。Python batch Expression 有一个显式例外：`row_preserving=False` 可以改变基数，但它必须是唯一的顶层 projection，不能与 passthrough 列并列组合。这个例外不适用于注册的 SQL alias；SQL alias 在 v1 中仍然保持行数。

## Relation API

结果为表形输出时使用 Relation API，例如多列输出、完整表阶段或显式改变基数。当前方法包括 `rel.map_batches`、`rel.flat_map` 和 `rel.map`；`rel.map` 是逐行 scalar Relation API，不是 pandas batch 转换。

例如，term 展开属于 Relation，因为一行输入可能生成多行输出：

```python
def split_terms(row):
    return [
        {"id": row["id"], "term": term}
        for term in str(row["text"]).split()
    ]

terms = source.flat_map(
    split_terms,
    schema={"id": "BIGINT", "term": "VARCHAR"},
    execution_backend="subprocess_task",
)
```

Vane 不提供直接的 SQL Relation 或 table-function API。SQL 可以准备输入 relation，也可以消费 Relation 结果，但改变基数或生成表形结果的方法调用目前仍是 Python Relation API。

## 共享规划与执行

两种模型使用同一个惰性规划与执行层。SQL alias 和 Python expression 会成为规划后的 projection 工作，Relation 方法会成为规划后的表阶段。执行以 Arrow batch 传递有类型的数据，因此 batching、worker placement 和资源选择不会改变语义输出模型。

正因为共享这一层，SQL 与 Python Expression 入口遵守相同的 projection 和行数保持规则。选择 Relation 的原因是它的表契约，而不仅仅是 callable 使用了 Python。

## Actor 复用与可变状态

Actor-backed 执行会在每个 actor 内复用 callable instance，从而复用已经加载的模型、客户端或缓存。这是生命周期和吞吐量工具；多个 actor 不会自动形成共享状态语义。

`vane.cls` 和 `vane.cls.batch` 是范围更窄的 Expression UDF 可变状态契约。在 v1 中，它们使用 `actor_number=1`，让一个 actor instance 在单次 query 的多个 batch 之间携带状态。一般的 actor-backed callable 复用不应被描述成持久应用状态。

## 状态、失败与副作用

可变的 `vane.cls` 状态是 query-scoped、actor-local 且 ephemeral 的。它不会 checkpoint，既不是 global state，也不是 keyed state，并且不提供 exactly-once 保证。对于有状态 Expression UDF，actor 丢失会让 query 失败，而不是用空状态重建 actor。

任何 UDF 产生的外部副作用都不会在 SQL transaction 或 query 失败时回滚，retry 还可能重复这些副作用。分布式 relation 执行也没有稳定的全局行顺序。因此，依赖状态的逻辑不应依赖输入顺序，外部写入应使用稳定标识和幂等处理。

## 选择模型

- 当一个有类型的结果需要保留在现有行旁边时，首先选择 SQL Expression。
- 当 Python 组合方式更清楚且仍符合相同 projection 契约时，选择 Python Expression。
- 当 callable 负责表形、多列或基数时，选择 Relation。
