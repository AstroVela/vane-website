---
title: 容量规划
---

Vane Data 作业的容量规划主要取决于数据量、UDF 成本和模型资源。

## 从小型运行开始

记录：

- 输入行数
- 输入字节数
- 投影列
- UDF backend
- batch 大小
- Relation `actor_number`、AI `concurrency` 和（对 prompt provider）`max_api_concurrency`
- CPU/GPU 资源
- 输出行数
- 运行耗时

之后每次只扩大一个变量。

## 本地 CPU UDF

使用：

```python
execution_backend="subprocess_task"
```

起始配置：

- 文本清洗使用 `batch_size=512`
- 图像/音频解码使用 `batch_size=32`

根据内存和延迟调整。

## 本地模型 actor

使用：

```python
execution_backend="subprocess_actor"
gpus=0.0
actor_number=1
```

起始时请选择能让模型保持忙碌、且不会造成内存压力的 batch 大小。

只有在 CPU/GPU 资源仍然空闲时，才增加 `actor_number`。

## Ray task

使用：

```python
execution_backend="ray_task"
```

适合 CPU 密集的无状态工作负载。如果 batch 太小，请关注调度开销。

## Ray actor

使用：

```python
execution_backend="ray_actor"
```

适合 actor-backed 模型推理。

容量规划规则：

```text
required GPUs = actor_number * gpus
```

如果 `actor_number=4` 且 `gpus=1`，Ray 集群至少需要 4 个可用 GPU 才能实现完整并行。

## 扫描容量规划

相关公开设置：

- `ray_scan_task_size_grouping`
- `ray_max_task_backlog`
- `ray_scan_task_open_cost_bytes`
- `ray_scan_task_min_partition_num`

当大量小文件原本会创建过多扫描任务时，使用扫描任务分组。

## 输出容量规划

embedding 可能主导输出大小。

估算：

```text
rows * dimensions * 4 bytes
```

对于 1000 万行、768 维 float32，单是原始向量值在计入文件格式开销前就约为 30 GB。

Relation actor 池使用 `actor_number`。公开的 AI provider 选项 `concurrency` 会映射为外层 AI UDF actor 数；对 prompt provider，`max_api_concurrency` 会限制每个 actor 内同时进行的 provider request 数。有状态 `vane.cls` v1 始终只使用一个 actor，不能按并行 actor 池规划容量。详见 [UDF API 参考](/zh-CN/docs/data/reference/udf-api)和 [AI Function API 参考](/zh-CN/docs/data/reference/ai-api)。

## 风险信号

- 单个 UDF batch 包含未使用的大列。
- Worker 在每个 batch 都下载模型权重。
- 分布式作业将输出写入节点本地磁盘。
- AI 作业因 actor 过多、每个 actor 内同时请求过多，或两者叠加而超过 provider 速率限制。
- 单个巨大文件阻碍并行度。

在增加更多机器前，先修复列投影、actor 复用、共享存储、provider 限制和分区问题。
