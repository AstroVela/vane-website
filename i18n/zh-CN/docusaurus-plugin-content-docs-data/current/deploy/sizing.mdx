---
title: 容量规划
---

Vane Data 作业的容量规划主要取决于数据量、UDF 成本和模型资源。

## 从小型运行开始

记录：

- 输入行数
- 输入字节数
- 投影列
- UDF backend
- batch 大小
- 并发度
- CPU/GPU 资源
- 输出行数
- 运行耗时

之后每次只扩大一个变量。

## 本地 CPU UDF

使用：

```python
execution_backend="subprocess_task"
```

起始配置：

- 文本清洗使用 `batch_size=512`
- 图像/音频解码使用 `batch_size=32`

根据内存和延迟调整。

## 本地模型 actor

使用：

```python
execution_backend="subprocess_actor"
```

起始配置：

- `concurrency=1`
- 选择能让模型保持忙碌、且不会造成内存压力的 batch 大小

只有在 CPU/GPU 资源仍然空闲时，才增加并发度。

## Ray task

使用：

```python
execution_backend="ray_task"
```

适合 CPU 密集的无状态工作负载。如果 batch 太小，请关注调度开销。

## Ray actor

使用：

```python
execution_backend="ray_actor"
```

适合有状态的模型推理。

容量规划规则：

```text
required GPUs = concurrency * gpus
```

如果 `concurrency=4` 且 `gpus=1`，Ray 集群至少需要 4 个可用 GPU 才能实现完整并行。

## 扫描容量规划

相关公开设置：

- `ray_scan_task_size_grouping`
- `ray_max_task_backlog`
- `ray_scan_task_open_cost_bytes`
- `ray_scan_task_min_partition_num`

当大量小文件原本会创建过多扫描任务时，使用扫描任务分组。

## 输出容量规划

embedding 可能主导输出大小。

估算：

```text
rows * dimensions * 4 bytes
```

对于 1000 万行、768 维 float32，单是原始向量值在计入文件格式开销前就约为 30 GB。

## 风险信号

- 单个 UDF batch 包含未使用的大列。
- Worker 在每个 batch 都下载模型权重。
- 分布式作业将输出写入节点本地磁盘。
- API 型 provider 的并发度高于速率限制。
- 单个巨大文件阻碍并行度。

在增加更多机器前，先修复列投影、actor 复用、共享存储、provider 限制和分区问题。
