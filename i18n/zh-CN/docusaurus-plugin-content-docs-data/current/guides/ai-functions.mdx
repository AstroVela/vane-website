本指南将完成一个可审查的文档分析任务：首先使用 SQL `ai_prompt` 和 `ai_embed`，然后把 Python Expression 作为同一保持行数模型的备选入口，最后仅针对必须使用 Relation API 的能力切换到 Relation。

精确签名与选项请查阅 [AI Function API 参考](/zh-CN/docs/data/reference/ai-api)，Expression 与 Relation 两种模型的设计原因请阅读 [AI Function 概念](/zh-CN/docs/data/concepts/ai-functions)。

**前提条件**

- 在每个可能执行 relation 的环境中安装 OpenAI client，包括所有 worker 环境：`python -m pip install openai`。
- 在每个 worker 的环境中设置 `OPENAI_API_KEY`。不要把凭据放入 SQL 文本或 options。
- 在 `data/documents/*.parquet` 提供 Parquet 文件，其中包含可转换为 `VARCHAR` 的 `document_id`、`document_type`、`source_uri` 与 `text` 列。在选中的切片中，ID 必须非 null 且唯一，source URI 与文本必须非 null。
- 运行前确保输出父目录已存在，例如执行 `mkdir -p output`。

## 1. 准备可审查的源数据

首先准备稳定的业务 ID、将发送给模型的源文本，以及足以复现和审查输出的元数据：

```python
import vane

con = vane.connect()
source = con.sql(
    """
    SELECT
        CAST(document_id AS VARCHAR) AS document_id,
        CAST(document_type AS VARCHAR) AS document_type,
        CAST(source_uri AS VARCHAR) AS source_uri,
        CAST(text AS VARCHAR) AS source_text
    FROM read_parquet('data/documents/*.parquet')
    """
)
source.to_table("ai_review_source")
```

`document_id` 是审查和写入时使用的 key。`source_uri`、`document_type` 与 `source_text` 用于说明每个模型结果来自哪里，以及 provider 实际看到了什么。

## 2. 用 SQL Expression 完成任务

### 选择候选切片

在任何 provider 调用之前先用普通 SQL 过滤。第一次运行应保持规模小、顺序确定并具有代表性：

```python
candidates = con.sql(
    """
    SELECT
        'ai-review-2026-07-11-001'::VARCHAR AS run_id,
        document_id,
        document_type,
        source_uri,
        source_text,
        (
            'Summarize the business issue and required follow-up in one sentence: '
            || source_text
        )::VARCHAR AS review_prompt
    FROM ai_review_source
    WHERE source_text IS NOT NULL
      AND trim(source_text) <> ''
      AND document_type IN ('invoice', 'claim', 'memo')
    ORDER BY document_id
    LIMIT 100
    """
)
candidates.to_table("ai_review_candidates")

expected_ids = [row[0] for row in con.sql(
    "SELECT document_id FROM ai_review_candidates ORDER BY document_id"
).fetchall()]
expected_row_count = len(expected_ids)

if expected_row_count == 0:
    raise ValueError("AI review candidate slice is empty")
if len(set(expected_ids)) != expected_row_count:
    raise ValueError("document_id must be unique in the candidate slice")
```

稳定排序让本次校验运行可以复现。后续每个 projection 都会保留 ID 与源字段；模型输出的先后顺序绝不能充当身份契约。

### 先生成审查说明

执行 relation 前，在每个 worker 的环境中设置 `OPENAI_API_KEY`。不要把 key、token、password 或 secret 放入 SQL options。

```python
prompted = con.sql(
    """
    SELECT
        run_id,
        document_id,
        document_type,
        source_uri,
        source_text,
        review_prompt,
        ai_prompt(
            review_prompt,
            struct_pack(
                provider := 'openai',
                model := 'gpt-4o-mini',
                concurrency := 2,
                max_api_concurrency := 4,
                batch_size := 8,
                max_retries := 2,
                on_error := 'log',
                system_message := 'Return one concise factual review note.',
                temperature := 0.0
            )
        ) AS review_note
    FROM ai_review_candidates
    ORDER BY document_id
    """
)
prompted.to_table("ai_review_prompted")
```

第二个参数是一个 foldable constant `STRUCT`。Provider、model、执行参数与请求设置都不能逐行变化。设置 `on_error := 'log'` 后，经过 retry 仍失败的响应会以 null `review_note` 留给校验步骤，而不会悄悄消失。

### 需要检索时添加 Embedding

本指南的任务会生成同时用于审查与检索的产物，因此后续的校验和写出流程假定 `embedding` 列存在。请在保留源数据与 prompt 输出的同时，再增加一个 SQL Expression 列：

```python
reviewed = con.sql(
    """
    SELECT
        run_id,
        document_id,
        document_type,
        source_uri,
        source_text,
        review_prompt,
        review_note,
        ai_embed(
            source_text,
            struct_pack(
                provider := 'openai',
                model := 'text-embedding-3-small',
                dimensions := 512,
                normalize := true,
                concurrency := 2,
                batch_size := 32,
                max_retries := 2,
                on_error := 'raise'
            )
        ) AS embedding
    FROM ai_review_prompted
    ORDER BY document_id
    """
)
reviewed.to_table("ai_review_model_output")
```

两个调用都位于 `SELECT` projection 内，也可以嵌套在普通 projection expression 中。每个调用只增加一个有类型、保持行数的输出列；predicate、grouping、join、short-circuit 组合与逐行变化的配置都应放在 AI expression 之外。

### 校验并写入审查产物

先把可容忍的 prompt 失败转换成显式审查状态，再在写入前检查行数、ID、null/error 输出与精确 schema：

```python
reviewable = con.sql(
    """
    SELECT
        run_id,
        document_id,
        document_type,
        source_uri,
        source_text,
        review_prompt,
        review_note,
        embedding,
        CASE
            WHEN review_note IS NULL OR trim(review_note) = ''
                THEN 'prompt_error'
            WHEN embedding IS NULL OR array_length(embedding) <> 512
                THEN 'embedding_error'
            ELSE 'ready'
        END::VARCHAR AS ai_status
    FROM ai_review_model_output
    ORDER BY document_id
    """
)
reviewable.to_table("ai_review_result")

validation = con.sql(
    """
    SELECT
        count(*) AS row_count,
        count(DISTINCT document_id) AS distinct_id_count,
        count(*) FILTER (WHERE document_id IS NULL) AS null_id_count,
        count(*) FILTER (
            WHERE source_uri IS NULL OR source_text IS NULL
        ) AS null_source_count,
        count(*) FILTER (WHERE ai_status <> 'ready') AS error_output_count
    FROM ai_review_result
    """
).fetchone()
(
    actual_row_count,
    distinct_id_count,
    null_id_count,
    null_source_count,
    error_output_count,
) = validation

assert actual_row_count == expected_row_count
assert distinct_id_count == expected_row_count
assert null_id_count == 0
assert null_source_count == 0

actual_ids = [row[0] for row in con.sql(
    "SELECT document_id FROM ai_review_result ORDER BY document_id"
).fetchall()]
assert actual_ids == expected_ids

expected_columns = [
    "run_id",
    "document_id",
    "document_type",
    "source_uri",
    "source_text",
    "review_prompt",
    "review_note",
    "embedding",
    "ai_status",
]
expected_types = [
    "VARCHAR",
    "VARCHAR",
    "VARCHAR",
    "VARCHAR",
    "VARCHAR",
    "VARCHAR",
    "VARCHAR",
    "FLOAT[512]",
    "VARCHAR",
]
assert list(reviewable.columns) == expected_columns
assert [str(dtype) for dtype in reviewable.types] == expected_types

print({"error_output_count": error_output_count})
reviewable.write_parquet("output/ai-review-2026-07-11-001.parquet")
```

这次写入会把失败行与其 ID、源数据、prompt 和状态一起保留下来，供后续审查。只有当下游只接受完全干净的结果时，才要求 `error_output_count == 0`。真正开始一次新的 provider 运行时，请使用新的 `run_id` 和输出路径；写入持久表时，应按稳定 key merge 或 replace，不要盲目追加 retry 结果。

## 3. 将 Python Expression 作为备选入口

当有类型的 provider/configuration 更适合在 Python 中构造时，使用 Python Expression。它与 SQL Expression 具有相同的单列、保持行数 projection 语义；它并不是获得结构化、多模态或表形输出的入口。

```python
python_prompt_provider_options = vane.ai.OpenAIProviderOptions(
    concurrency=2,
    max_api_concurrency=4,
)
python_embedding_provider_options = vane.ai.OpenAIProviderOptions(
    concurrency=2,
)
python_prompt_options = vane.ai.OpenAIPromptOptions(
    temperature=0,
    on_error="log",
)
python_embedding_options = vane.ai.OpenAIEmbeddingOptions(
    on_error="raise",
)

python_prompted = candidates.select(
    vane.col("run_id"),
    vane.col("document_id"),
    vane.col("document_type"),
    vane.col("source_uri"),
    vane.col("source_text"),
    vane.col("review_prompt"),
    vane.ai.prompt(
        vane.col("review_prompt"),
        provider="openai",
        model="gpt-4o-mini",
        provider_options=python_prompt_provider_options,
        prompt_options=python_prompt_options,
        system_message="Return one concise factual review note.",
    ).alias("review_note"),
)

python_reviewed = python_prompted.select(
    vane.col("run_id"),
    vane.col("document_id"),
    vane.col("document_type"),
    vane.col("source_uri"),
    vane.col("source_text"),
    vane.col("review_prompt"),
    vane.col("review_note"),
    vane.ai.embed(
        vane.col("source_text"),
        provider="openai",
        model="text-embedding-3-small",
        provider_options=python_embedding_provider_options,
        embedding_options=python_embedding_options,
        dimensions=512,
        normalize=True,
    ).alias("embedding"),
)
python_reviewed.to_table("ai_review_python_model_output")
```

对 `ai_review_python_model_output` 应执行与 `ai_review_model_output` 相同的行数、有序 ID、null/error、schema 及按 run 写入检查。每个 AI Expression 都应直接放在 `rel.select(...)` 中并设置 alias；不要在过滤条件中使用，也不要手工把输出与源行重新组合。

## 4. 仅在专用能力需要时使用 Relation API

Relation helper 是 Python 表变换。Vane 不提供直接的 SQL Relation 或 table-function API。只有当任务需要 SQL 或 Python Expression 未公开的能力时，才使用这些 helper。

### 结构化或多模态 Prompt

调用 `rel.prompt` 前先定义 response model 并完成 import。对于支持图像输入的 provider/model 组合，同一 helper 还接受 `image_columns`：

```python
from pydantic import BaseModel


class ReviewDecision(BaseModel):
    label: str
    reason: str


multimodal_candidates = con.sql(
    """
    SELECT
        CAST(document_id AS VARCHAR) AS document_id,
        CAST(review_prompt AS VARCHAR) AS review_prompt,
        image
    FROM read_parquet('data/review-items/*.parquet')
    WHERE review_prompt IS NOT NULL
    ORDER BY document_id
    LIMIT 20
    """
)

decision_only = multimodal_candidates.prompt(
    "review_prompt",
    image_columns=["image"],
    provider="openai",
    model="gpt-4o-mini",
    provider_options=vane.ai.OpenAIProviderOptions(
        max_api_concurrency=4,
    ),
    prompt_options=vane.ai.OpenAIPromptOptions(
        temperature=0,
        on_error="log",
    ),
    return_format=ReviewDecision,
    output_column="decision_json",
    execution_backend="subprocess_task",
)
```

发布前，应使用 `ReviewDecision` 校验每个非 null 的 `decision_json`。纯文本输入可以省略 `image_columns`；只有所选 provider 支持结构化响应时才保留 `return_format`。

### 内置分类

`rel.classify_text` 是内置的 zero-shot 分类路径：

```python
label_only = candidates.classify_text(
    "source_text",
    labels=["billing", "delivery", "policy", "other"],
    provider="transformers",
    model="facebook/bart-large-mnli",
    output_column="review_label",
    batch_size=16,
    execution_backend="subprocess_task",
)
```

扩容前，应验证每个非 null label 都属于声明的集合，并从每个标签分组中抽样检查。

### 内置长文本分块

`rel.embed_text` 通过 `max_chunk_chars` 负责内置字符分块：

```python
chunk_embedding_only = candidates.embed_text(
    "source_text",
    provider="transformers",
    model="sentence-transformers/all-MiniLM-L6-v2",
    max_chunk_chars=2000,
    chunk_overlap_chars=200,
    output_column="chunked_embedding",
    execution_backend="subprocess_task",
)
```

如果每个现有行只需要增加一个 embedding 列，请使用 SQL `ai_embed` 或 `vane.ai.embed`。只有需要 helper 自行拆分并组合长文本时才使用这个 Relation helper；如果每个 chunk 都需要独立的稳定 ID 和行，请阅读 [大规模 Embedding 指南](/zh-CN/docs/data/guides/embeddings-at-scale)。

### 显式选择 Relation Backend 与 Actor Pool

Relation 方法可以显式选择 `execution_backend`。对于 Ray-backed vLLM actor pool，provider `concurrency` 会映射为 actor 数，而 `gpus_per_actor` 会为每个 actor 请求对应的 Ray GPU 资源：

```python
vane.configure(runner="ray")

gpu_response_only = candidates.prompt(
    "review_prompt",
    provider="vllm",
    model="Qwen/Qwen3-1.7B",
    provider_options=vane.ai.VLLMProviderOptions(
        concurrency=2,
        gpus_per_actor=1,
    ),
    prompt_options=vane.ai.VLLMPromptOptions(
        max_tokens=128,
        temperature=0,
        on_error="log",
    ),
    output_column="gpu_review_note",
    execution_backend="ray_actor",
)
```

示例中的 `_only` 命名是刻意的：Relation helper 可能只返回配置的输出列，而不会自动保留所有源字段。必须在调用前规划源数据保留方式。只有下游不需要源 key 时，才把内置 helper 用作隔离的仅输出阶段；需要 key 时，应使用自定义 row-preserving batch UDF，在同一个已声明 schema 中返回稳定 ID 与输出。绝不能按行位置重新关联结果。

## 5. 安全运行 Provider 与写入

- **Provider 与 model：** 记录精确的 provider、model、prompt/run 版本与输出路径。Hosted provider 带来网络、quota、延迟和成本问题；local provider 则需要关注依赖、模型文件、内存与启动开销。
- **凭据：** Provider descriptor 会被序列化，并在 worker 上延迟实例化。凭据应放入每个 worker 的环境，而不是 SQL 文本或序列化 options；worker 的网络访问也应只开放必要 endpoint。
- **执行控制：** SQL 与 Python Expression 由 active runner 选择 local subprocess actor 或 Ray actor；Relation 方法可以显式选择 backend。Provider `concurrency` 控制 actor 数，`max_api_concurrency` 限制单个 actor 内的调用数，batch size 则影响吞吐、内存和失败范围。
- **Retry 与错误：** 如果失败应该终止样本校验，请使用 `on_error="raise"`。容错模式可能把 prompt 失败转换成 null；已知维度时，还可能把 embedding 失败转换成 zero vector，因此必须检查所选输出契约。Wrapper 与 provider SDK 的 retry 都可能重复调用。
- **外部副作用：** Provider 请求既不是事务操作，也不提供 exactly-once。Query retry 可能产生新的请求与费用，rollback 也无法撤销已完成的调用。请保留 `(run_id, document_id)`，写入可审查的 run 产物，并让下游 merge 具备幂等性。
