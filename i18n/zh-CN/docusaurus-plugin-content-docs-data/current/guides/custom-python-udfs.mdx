---
title: 自定义 Python UDF
---

当 AI 或 multimodal workflow 需要 SQL 不适合承载的逻辑时，使用 Custom Python UDF：文本清理、media metadata 检查、domain rule、外部 Python library、model preprocessing、自定义 inference 或 client call。

目标是在一个小型真实数据切片上先把单个 UDF 做正确，并保持 table contract 显式；只有当 distribution、actor 复用或 GPU placement 值得引入额外配置时，才把同一套逻辑迁移到 Ray。

## UDF contract

对于 `map_batches`，contract 是：

1. callable 接收一个 `pyarrow.Table`。
2. callable 返回一个 `pyarrow.Table`。
3. 返回的 table 必须匹配声明的 `schema`。

`map_batches` 返回 UDF 产出的列。它不会自动保留 upstream column。请返回稳定 ID，以及后续阶段用于 join、validation、review 或 audit 的所有 source column。

```python
import vane

con = vane.connect()

documents = con.sql("""
    select
        cast(document_id as varchar) as document_id,
        source_uri,
        media_type,
        document_type,
        text
    from read_parquet('data/documents/*.parquet')
    where text is not null
    order by document_id
    limit 100
""")
```

保持第一个 sample 小且真实。学习阶段可以使用 inline data，但第一个真正有用的 UDF 应该接触真实 pipeline 会使用的列、source metadata 和脏数据值。

## 第一个 stateless batch UDF

当工作是确定性的，并且不需要 model、network client、cache 或其他可复用状态时，从普通 function 开始。

```python
import re

import pyarrow as pa


SUPPORTED_MEDIA_TYPES = {"application/pdf", "text/plain", "text/html"}
RISK_TERMS = ("missing signature", "expired", "redacted incorrectly")


def prepare_document_text(batch: pa.Table) -> pa.Table:
    document_ids = batch.column("document_id").to_pylist()
    source_uris = batch.column("source_uri").to_pylist()
    media_types = batch.column("media_type").to_pylist()
    document_types = batch.column("document_type").to_pylist()
    texts = batch.column("text").to_pylist()

    normalized_text = []
    text_length = []
    needs_review = []
    review_reason = []
    status = []
    error = []

    for media_type, text in zip(media_types, texts, strict=False):
        try:
            clean = re.sub(r"\s+", " ", str(text or "")).strip()
            lower = clean.lower()

            normalized_text.append(clean)
            text_length.append(len(clean))

            if media_type not in SUPPORTED_MEDIA_TYPES:
                needs_review.append(True)
                review_reason.append("unsupported_media_type")
            elif not clean:
                needs_review.append(True)
                review_reason.append("empty_text")
            elif any(term in lower for term in RISK_TERMS):
                needs_review.append(True)
                review_reason.append("risk_term")
            else:
                needs_review.append(False)
                review_reason.append("none")
            status.append("ok")
            error.append("")
        except Exception as exc:
            normalized_text.append("")
            text_length.append(0)
            needs_review.append(True)
            review_reason.append("udf_error")
            status.append("failed")
            error.append(f"prepare_document_text: {type(exc).__name__}: {exc}")

    return pa.table({
        "document_id": document_ids,
        "source_uri": source_uris,
        "media_type": media_types,
        "document_type": document_types,
        "normalized_text": normalized_text,
        "text_length": text_length,
        "needs_review": needs_review,
        "review_reason": review_reason,
        "status": status,
        "error": error,
    })


prepared = documents.map_batches(
    prepare_document_text,
    schema={
        "document_id": "VARCHAR",
        "source_uri": "VARCHAR",
        "media_type": "VARCHAR",
        "document_type": "VARCHAR",
        "normalized_text": "VARCHAR",
        "text_length": "BIGINT",
        "needs_review": "BOOLEAN",
        "review_reason": "VARCHAR",
        "status": "VARCHAR",
        "error": "VARCHAR",
    },
    batch_size=64,
    execution_backend="subprocess_task",
)

prepared.show()
```

这个示例会在返回新字段的同时返回 source `document_id`、`source_uri`、`media_type` 和 `document_type` 列，以及显式的 `status`/`error` 列。这是有意为之。Vane 不会自动创建 row-level diagnostics。未捕获的 exception 会让 batch 或 query 失败；如果你希望失败行可检查，请在 UDF 内捕获 per-row failure。

## 使用 SQL 校验

当你需要关系型检查时，将 UDF output 转成具名 table。

```python
prepared.to_table("prepared_documents")

summary = con.sql("""
    select
        document_type,
        media_type,
        needs_review,
        review_reason,
        count(*) as rows
    from prepared_documents
    group by document_type, media_type, needs_review, review_reason
    order by document_type, media_type, needs_review, review_reason
""")

summary.show()
```

第一次 validation run 应检查 row count、ID、不支持的 media type、空文本，以及不应该被标记的行。在引入 model、provider 或 Ray cluster 之前，正确性问题更容易修复。

## Schema 与 Arrow output

将 schema 视为 UDF interface 的一部分。

| 主题 | 建议 |
| --- | --- |
| Column name | 精确返回 `schema` 中声明的列。 |
| 必需的 source column | 返回后续所需的 ID、source URI/path、media type、business key、文本或 metadata。 |
| Type | 使用 Arrow-compatible value：string、number、boolean、bytes、list、struct 或 Arrow array。 |
| Null | 保持 null 行为显式，尤其是 numeric、bytes、list 和 model-output 列。 |
| Row count | `map_batches` UDF 通常每个输入行返回一个输出行。当一个输入可能生成零行或多行时，使用 `flat_map`。 |

对于 `flat_map`，callable 会接收一行 Python `dict`。它可以返回 `None`、一个 output `dict`，或 output dict 的 iterable/generator。声明的 schema 会从这些 dict 中选择 key；缺失的 key 会变成 null，额外 key 会被忽略。对于 chunking、frame extraction summary、split record，或任何一个 input row 可能产生零行、一行或多行 output 的 stage，请使用 `flat_map`。

如果 Arrow 无法推断你需要的类型，请构造显式 array：

```python
return pa.table({
    "document_id": pa.array(document_ids, type=pa.string()),
    "review_reason": pa.array(review_reason, type=pa.string()),
})
```

避免返回 model class、file handle、自定义 dataclass 或 raw exception 等 Python object。返回前请将它们转换为 string、number、bytes、list、struct、JSON string 或显式 Arrow array。

## Batch size

`batch_size` 控制 Vane 每次 `map_batches` 调用传入多少输入行。

调试时从小 batch 开始：

```python
debugged = documents.map_batches(
    prepare_document_text,
    schema={
        "document_id": "VARCHAR",
        "source_uri": "VARCHAR",
        "media_type": "VARCHAR",
        "document_type": "VARCHAR",
        "normalized_text": "VARCHAR",
        "text_length": "BIGINT",
        "needs_review": "BOOLEAN",
        "review_reason": "VARCHAR",
        "status": "VARCHAR",
        "error": "VARCHAR",
    },
    batch_size=8,
    execution_backend="subprocess_task",
)
```

在 schema 和输出行正确之后，再增大 batch。CPU text transform 通常可以接受更大的 batch。Media decoding、embeddings、LLM call 和 GPU inference 通常需要更小的 batch，以适配内存和 provider limit。

Batch size 不应改变结果语义。如果调整 batch size 后输出发生变化，UDF 很可能依赖了隐藏的 batch state。

## Callable class 与 actor

当 setup 成本较高，或状态需要跨 batch 复用时，使用 callable class：

- model 或 tokenizer loading
- API client setup
- database 或 object-store client setup
- lookup table 或 cache
- GPU model state

Actor backend 会在 worker 内构造 class，并在多个 batch 间复用它。请传入 class 本身，而不是已经创建好的 instance 或普通 function。Actor class 不能有必填 constructor argument。请把配置放在环境变量、class-level default、带默认值的 optional argument，或 no-argument wrapper class 中。

```python
import pyarrow as pa


class PromptBuilder:
    def __init__(self) -> None:
        self.template_by_type = {
            "claim": "Extract claim decision and one short reason:",
            "invoice": "Extract invoice number, supplier, and total:",
            "policy": "Summarize policy issue and required follow-up:",
        }

    def __call__(self, batch: pa.Table) -> pa.Table:
        document_ids = batch.column("document_id").to_pylist()
        source_uris = batch.column("source_uri").to_pylist()
        document_types = batch.column("document_type").to_pylist()
        texts = batch.column("normalized_text").to_pylist()
        upstream_status = batch.column("status").to_pylist()
        upstream_error = batch.column("error").to_pylist()

        prompts = []
        prompt_template = []
        status = []
        error = []

        for document_type, text, previous_status, previous_error in zip(
            document_types,
            texts,
            upstream_status,
            upstream_error,
            strict=False,
        ):
            if previous_status != "ok":
                prompt_template.append("")
                prompts.append("")
                status.append("skipped")
                error.append(previous_error)
                continue
            template = self.template_by_type.get(document_type, "Summarize this document:")
            prompt_template.append(template)
            prompts.append(f"{template}\n\n{text}")
            status.append("ok")
            error.append("")

        return pa.table({
            "document_id": document_ids,
            "source_uri": source_uris,
            "prompt_template": prompt_template,
            "prompt": prompts,
            "status": status,
            "error": error,
        })


prompts = prepared.map_batches(
    PromptBuilder,
    schema={
        "document_id": "VARCHAR",
        "source_uri": "VARCHAR",
        "prompt_template": "VARCHAR",
        "prompt": "VARCHAR",
        "status": "VARCHAR",
        "error": "VARCHAR",
    },
    batch_size=64,
    execution_backend="subprocess_actor",
    concurrency=2,
)
```

在真实 model UDF 中，请在 `__init__` 中或第一次调用时 lazy load model、tokenizer 或 client，而不是在 module import 时加载。这样可以保持 import 轻量，并让每个 subprocess 或 Ray actor 拥有自己的状态。

## Backend 选择

选择能验证当前问题的最简单 backend。

| Backend | Callable 形态 | 适用场景 |
| --- | --- | --- |
| `subprocess_task` | Function | 第一个 stateless UDF、本地隔离、CPU transform。 |
| `subprocess_actor` | Callable class | 不使用 Ray 的本地 model 或 client 复用。 |
| `ray_task` | Function | 本地正确性验证后，再执行 distributed stateless batch work。 |
| `ray_actor` | Callable class | Distributed model/client 复用、GPU placement，或更大的 actor pool。 |

唯一有效的 backend name 是 `subprocess_task`、`subprocess_actor`、`ray_task` 和 `ray_actor`。`local`、`subprocess`、`ray=True`、`use_process`、`actor_count`、`prefer_actor` 等旧写法或 shorthand 不是当前 API。`concurrency` 支持 `subprocess_task`、`subprocess_actor` 和 `ray_actor`；不要将它与 `ray_task` 一起使用。

对于 distributed execution 和 `ray_actor` pool，请在创建并运行 plan 前配置 Ray/Vane runner。部分直接 Ray task path 可以为小型运行初始化 Ray，但示例应显式配置 Ray，让 worker environment、path 和 resource 更可控。

```python
import vane

vane.configure(runner="ray")

ray_prepared = documents.map_batches(
    prepare_document_text,
    schema={
        "document_id": "VARCHAR",
        "source_uri": "VARCHAR",
        "media_type": "VARCHAR",
        "document_type": "VARCHAR",
        "normalized_text": "VARCHAR",
        "text_length": "BIGINT",
        "needs_review": "BOOLEAN",
        "review_reason": "VARCHAR",
        "status": "VARCHAR",
        "error": "VARCHAR",
    },
    batch_size=256,
    execution_backend="ray_task",
)
```

对于 stateful 或 GPU-backed model，保持 class 形态，并切换 backend：

```python
ray_prompts = prepared.map_batches(
    PromptBuilder,
    schema={
        "document_id": "VARCHAR",
        "source_uri": "VARCHAR",
        "prompt_template": "VARCHAR",
        "prompt": "VARCHAR",
        "status": "VARCHAR",
        "error": "VARCHAR",
    },
    batch_size=128,
    execution_backend="ray_actor",
    concurrency=4,
)
```

对于 GPU placement，请按 actor 请求 GPU resource：

```python
class GpuEmbedder:
    def __init__(self) -> None:
        # 在这里加载 tokenizer/model，使每个 actor 拥有自己的 GPU state。
        self.model_name = "local-or-mounted-model"

    def __call__(self, batch: pa.Table) -> pa.Table:
        document_ids = batch.column("document_id").to_pylist()
        source_uris = batch.column("source_uri").to_pylist()
        texts = batch.column("normalized_text").to_pylist()

        embeddings = [[0.0, 0.0, 0.0] for _ in texts]
        return pa.table({
            "document_id": document_ids,
            "source_uri": source_uris,
            "embedding": embeddings,
            "status": ["ok"] * len(document_ids),
            "error": [""] * len(document_ids),
        })


embedded = prepared.map_batches(
    GpuEmbedder,
    schema={
        "document_id": "VARCHAR",
        "source_uri": "VARCHAR",
        "embedding": "FLOAT[]",
        "status": "VARCHAR",
        "error": "VARCHAR",
    },
    batch_size=32,
    execution_backend="ray_actor",
    concurrency=2,
    cpus=0.0,
    gpus=1.0,
)
```

`gpus=1.0` 是每个 actor 的资源请求。使用 `concurrency=2` 时，Ray 需要两个可用 GPU slot 才能满并行运行。

每个 Ray worker 都必须能够 import 该 callable，并访问相同的数据路径、model file、credential 和 Python dependency。Callable 会被序列化进 plan，但被引用的 module 和 runtime dependency 仍需要存在于 worker environment 中。

## 本地调试顺序

在将 UDF 放入更大的 pipeline 之前，按以下顺序调试：

1. 选择带有稳定 ID 的小型真实 sample。
2. 直接在 Arrow table 上调用 Python function。
3. 使用较小的 `batch_size`，通过 `map_batches` 运行同一个 function。
4. 使用 SQL 校验 row count、ID、source column 和 output type。
5. 在本地增大 `batch_size`。
6. 只有在本地 contract 正确后，才迁移到 `ray_task` 或 `ray_actor`。

```python
sample = con.sql("""
    select document_id, source_uri, media_type, document_type, text
    from read_parquet('data/documents/*.parquet')
    where text is not null
    order by document_id
    limit 20
""")

sample_arrow = sample.to_arrow_table()
debug_arrow = prepare_document_text(sample_arrow)

assert len(debug_arrow) == len(sample_arrow)
print(debug_arrow.schema)
print(debug_arrow.to_pydict())
```

然后通过 Vane 运行：

```python
debug_relation = sample.map_batches(
    prepare_document_text,
    schema={
        "document_id": "VARCHAR",
        "source_uri": "VARCHAR",
        "media_type": "VARCHAR",
        "document_type": "VARCHAR",
        "normalized_text": "VARCHAR",
        "text_length": "BIGINT",
        "needs_review": "BOOLEAN",
        "review_reason": "VARCHAR",
        "status": "VARCHAR",
        "error": "VARCHAR",
    },
    batch_size=4,
    execution_backend="subprocess_task",
)

debug_relation.show()
```

如果这里失败，请先将问题限制在本地。UDF contract 修复前，Ray run 会额外引入 import、dependency、storage 和 resource 等变量。

## 常见失败

| 失败 | 通常意味着 | 修复方式 |
| --- | --- | --- |
| Schema mismatch | 返回的列或类型与 `schema` 不匹配。 | 打印返回的 Arrow schema，然后对齐 column name 和 Arrow-compatible type。 |
| Missing ID or source columns | UDF 只返回了 derived column。 | 返回 downstream SQL、join 和 review 所需的稳定 ID 与 source field。 |
| Missing status or error fields | UDF 依赖 exception 或 log 作为 row-level diagnostics。 | 当失败行需要保持可审查时，返回显式 `status` 和 `error` 列。 |
| Arrow type incompatibility | UDF 返回了 Arrow 无法序列化的 Python object。 | 将输出转换为 primitive、bytes、list、struct、JSON string 或显式 `pa.array` value。 |
| Wrong callable shape | task backend 收到了 class/instance，或 actor backend 收到了 function/instance。 | 对 `subprocess_task`/`ray_task` 使用 function；对 `subprocess_actor`/`ray_actor` 使用无必填构造参数的 callable class。 |
| Worker import failure | subprocess 或 Ray worker 无法 import 定义 UDF 的 module。 | 将被引用的代码放入可 import 的 module，并在每个 worker 上安装相同 Python dependency。 |
| Model loads at import time | 每次 worker import 都触发重型 setup，或在 actor 启动前失败。 | 将 model、tokenizer 和 client setup 移到 callable class 的 `__init__` 或首次调用中。 |
| Batch works locally but fails on Ray | Worker 无法访问数据、credential、model file 或兼容的 package version。 | 增大数据规模前，先用极小 Ray sample 验证 worker path 和 environment。 |
| Memory spikes | Batch size、media bytes 或 model output 过大。 | 投影更少列，降低 `batch_size`，拆分 stage，或使用 actor 复用 model state。 |

## 与其他文档的关系

- 当主要任务是为 AI 或 multimodal workflow 做 table-shaped cleanup、join、deduplication、validation 或 output shaping 时，使用 [Structured Transformation](/zh-CN/docs/data/guides/structured-transformation)，只在 SQL 不适合承载的逻辑上添加 UDF。
- 当 UDF 需要读取 bytes、解码文件、提取 media feature，或记录 per-stage status 和 error 时，使用 [Multimodal Pipeline](/zh-CN/docs/data/guides/multimodal-pipeline)。
- 对于标准 embedding、classification 或 prompt enrichment，使用 [AI Functions](/zh-CN/docs/data/guides/ai-functions)。当你需要 custom model、custom output schema、external client behavior，或完全控制 preprocessing 与 postprocessing 时，再回到 Custom Python UDFs。
- 阅读 [UDFs](/zh-CN/docs/data/concepts/udfs)，了解 `map_batches`、callable class 和 execution backend 背后的概念模型。
