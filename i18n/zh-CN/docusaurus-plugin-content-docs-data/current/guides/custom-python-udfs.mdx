---
title: 自定义 Python UDF
---

当 Vane Data pipeline 中的 Python 逻辑不适合用 SQL 或内置 AI Function 表达时，可以使用 Custom Python UDF：外部库、自定义规则、媒体处理、模型预处理、自定义推理、私有客户端，或完全可控的 output schema。

本文聚焦 UDF 的类型，以及不同场景下的选择方式。如果只需在现有列旁增加一个有类型的结果，优先使用 Expression UDF：推荐通过 SQL alias 进入 SQL 工作流，也可以用 Python Expression API 构造同一个 projection。只有当 UDF 负责多列输出或改变行数的表阶段时，才使用 Relation API。

## 选择 UDF 类型

先确定输出形态，再确定可调用对象形态；backend 是另一层选择，它决定可调用对象在哪里执行。

| UDF 类型 | 推荐 API | 可调用对象形态 | 适用场景 |
| --- | --- | --- | --- |
| 无状态 scalar Expression UDF | 注册 SQL alias；在 Python 中构造 expression 时使用 `vane.func` | Scalar 函数 | 在现有列旁增加一个有类型、保持行数的值。 |
| 无状态 batch Expression UDF | 注册 SQL alias；在 Python 中构造 expression 时使用 `vane.func.batch` | Arrow 函数 | Batch 处理有帮助，但结果仍是一个有类型的列。 |
| 有状态 Expression UDF | 注册 SQL alias；Python 使用 `vane.cls` 或 `vane.cls.batch` | 可调用类 | 一个有类型的结果需要 query scope 的 actor 状态。 |
| 无状态 batch Relation UDF | `map_batches` | Python 函数 | 返回多列或完整表结果，且不需要复用状态。 |
| 有状态 batch Relation UDF | `map_batches` | 可调用类 | 返回表形结果，同时跨 batch 复用昂贵的初始化结果。 |
| 行扩展 Relation UDF | `flat_map` | 行函数 | 一个输入行可能产生零行、一行或多行输出。 |
| Scalar Relation UDF | `map` | Scalar 函数 | 需要显式 Relation executor 及其透传结果形态。 |

普通的保持行数补充列任务，应先从 connection scope 的 SQL alias 开始。过滤、join 和源列选择仍然使用普通 SQL，Python callable 只提供一个有类型的 projection 值：

```python
import vane

con = vane.connect()


def normalize_document_type(value):
    return str(value or "unknown").strip().lower()


vane.attach_function(
    normalize_document_type,
    alias="normalize_document_type_sql",
    connection=con,
    parameters=["VARCHAR"],
    return_dtype="VARCHAR",
)

try:
    normalized = con.sql("""
        select
            document_id,
            normalize_document_type_sql(document_type) as document_type
        from (values
            ('doc-001', ' Invoice '),
            ('doc-002', null)
        ) as documents(document_id, document_type)
    """)
    normalized.show()
finally:
    vane.detach_function("normalize_document_type_sql", connection=con)
```

Alias 只属于 `con`，并且支持在 `SELECT` projection 中使用。应在 UDF projection 之前完成过滤、join 和分组。当 projection 需要作为 Python Expression 对象组装时，再使用 `vane.func`、`vane.func.batch`、`vane.cls` 或 `vane.cls.batch`。注册签名和放置限制见 [UDF API 参考](/zh-CN/docs/data/reference/udf-api)。

## 无状态 Batch UDF

当每个 batch 都可以独立处理时，使用无状态 batch UDF：

- 文本清理
- 基于规则的标注
- 元数据检查
- 轻量解析
- 确定性特征提取
- 不需要可复用状态的 Python 库调用

无状态 batch UDF 是一个 Python 函数，接收 `pyarrow.Table`，返回 `pyarrow.Table`。如果它只产出一个有类型的列并保持输入行数，可以把它注册为 SQL Expression alias，或通过 Python `vane.func.batch(..., row_preserving=True)` 使用。下面的示例会同时返回源字段、诊断字段和多个派生列，因此更适合使用表形的 `map_batches` Relation API。

```python
import re

import pyarrow as pa
import vane

con = vane.connect()

documents = con.sql("""
    select
        cast(document_id as varchar) as document_id,
        source_uri,
        media_type,
        document_type,
        text
    from read_parquet('data/documents/*.parquet')
    where text is not null
    order by document_id
    limit 100
""")

SUPPORTED_MEDIA_TYPES = {"application/pdf", "text/plain", "text/html"}
RISK_TERMS = ("missing signature", "expired", "redacted incorrectly")


def prepare_document_text(batch: pa.Table) -> pa.Table:
    document_ids = batch.column("document_id").to_pylist()
    source_uris = batch.column("source_uri").to_pylist()
    media_types = batch.column("media_type").to_pylist()
    document_types = batch.column("document_type").to_pylist()
    texts = batch.column("text").to_pylist()

    normalized_text = []
    text_length = []
    needs_review = []
    review_reason = []
    status = []
    error = []

    for media_type, text in zip(media_types, texts, strict=False):
        try:
            clean = re.sub(r"\s+", " ", str(text or "")).strip()
            lower = clean.lower()

            normalized_text.append(clean)
            text_length.append(len(clean))

            if media_type not in SUPPORTED_MEDIA_TYPES:
                needs_review.append(True)
                review_reason.append("unsupported_media_type")
            elif not clean:
                needs_review.append(True)
                review_reason.append("empty_text")
            elif any(term in lower for term in RISK_TERMS):
                needs_review.append(True)
                review_reason.append("risk_term")
            else:
                needs_review.append(False)
                review_reason.append("none")
            status.append("ok")
            error.append("")
        except Exception as exc:
            normalized_text.append("")
            text_length.append(0)
            needs_review.append(True)
            review_reason.append("udf_error")
            status.append("failed")
            error.append(f"prepare_document_text: {type(exc).__name__}: {exc}")

    return pa.table({
        "document_id": document_ids,
        "source_uri": source_uris,
        "media_type": media_types,
        "document_type": document_types,
        "normalized_text": normalized_text,
        "text_length": text_length,
        "needs_review": needs_review,
        "review_reason": review_reason,
        "status": status,
        "error": error,
    })


prepared_schema = {
    "document_id": "VARCHAR",
    "source_uri": "VARCHAR",
    "media_type": "VARCHAR",
    "document_type": "VARCHAR",
    "normalized_text": "VARCHAR",
    "text_length": "BIGINT",
    "needs_review": "BOOLEAN",
    "review_reason": "VARCHAR",
    "status": "VARCHAR",
    "error": "VARCHAR",
}

prepared = documents.map_batches(
    prepare_document_text,
    schema=prepared_schema,
    batch_size=64,
    execution_backend="subprocess_task",
)

prepared.show()
```

返回的表同时包含新字段和后续阶段需要的源字段。`map_batches` 返回 UDF 产出的列，不会自动保留上游列。

作为对比，保持行数的 Python batch Expression 会显式命名输入列，并且只声明一个输出列：

```python
def normalize_text_batch(table: pa.Table) -> pa.Table:
    values = table.column("text").to_pylist()
    return pa.table({
        "normalized_text": [
            None if value is None else str(value).strip().lower()
            for value in values
        ]
    })


normalized_text = vane.func.batch(
    normalize_text_batch,
    inputs={"text": vane.col("text")},
    schema={"normalized_text": "VARCHAR"},
    row_preserving=True,
).alias("normalized_text")

normalized = documents.select(
    vane.col("document_id"),
    normalized_text,
)
```

如果这个 projection 属于 SQL pipeline，使用已注册的 SQL 形式；如果需要动态组装 expression，则使用这里的 Python 形式。

增大 batch size 或迁移到 Ray 之前，先校验这个无状态 UDF：

```python
prepared.to_table("prepared_documents")

summary = con.sql("""
    select
        document_type,
        media_type,
        needs_review,
        review_reason,
        count(*) as rows
    from prepared_documents
    group by document_type, media_type, needs_review, review_reason
    order by document_type, media_type, needs_review, review_reason
""")

summary.show()
```

本地输出正确后，同一个函数可以用 `ray_task` 运行：

```python
import vane

vane.configure(runner="ray")

ray_prepared = documents.map_batches(
    prepare_document_text,
    schema=prepared_schema,
    batch_size=256,
    execution_backend="ray_task",
)
```

只有在本地契约正确后才使用 `ray_task`。Ray 会额外引入 worker 导入、依赖、存储和资源等变量，可能掩盖普通的 UDF 问题。

## 有状态 Batch UDF

当初始化结果需要跨 batch 复用时，使用有状态 batch UDF：

- 模型或 tokenizer 加载
- API 客户端初始化
- 数据库或对象存储客户端初始化
- 查找表或缓存
- GPU 模型状态

有状态 UDF 是可调用类。对于只返回一个有类型、保持行数的 Expression 结果，可以使用 `vane.cls` 或 `vane.cls.batch`，实例化被装饰的类，再把该实例注册为 SQL alias，或在 Python projection 中调用。v1 Expression 契约每个 query 使用一个 actor，其状态仅限 query scope 且是临时的。

下面的示例返回多列表，因此使用 Relation 形式：传给 `map_batches` 的应该是类本身，而不是已经创建好的实例。Actor backend 会在 worker 内构造这个类，并在多个 batch 间复用同一个对象。

Actor 类不应有必填构造参数。可以把配置放在环境变量、类级默认值、带默认值的可选参数，或无参数包装类中。

```python
import pyarrow as pa


class PromptBuilder:
    def __init__(self) -> None:
        self.template_by_type = {
            "claim": "Extract claim decision and one short reason:",
            "invoice": "Extract invoice number, supplier, and total:",
            "policy": "Summarize policy issue and required follow-up:",
        }

    def __call__(self, batch: pa.Table) -> pa.Table:
        document_ids = batch.column("document_id").to_pylist()
        source_uris = batch.column("source_uri").to_pylist()
        document_types = batch.column("document_type").to_pylist()
        texts = batch.column("normalized_text").to_pylist()
        upstream_status = batch.column("status").to_pylist()
        upstream_error = batch.column("error").to_pylist()

        prompts = []
        prompt_template = []
        status = []
        error = []

        for document_type, text, previous_status, previous_error in zip(
            document_types,
            texts,
            upstream_status,
            upstream_error,
            strict=False,
        ):
            if previous_status != "ok":
                prompt_template.append("")
                prompts.append("")
                status.append("skipped")
                error.append(previous_error)
                continue

            template = self.template_by_type.get(document_type, "Summarize this document:")
            prompt_template.append(template)
            prompts.append(f"{template}\n\n{text}")
            status.append("ok")
            error.append("")

        return pa.table({
            "document_id": document_ids,
            "source_uri": source_uris,
            "prompt_template": prompt_template,
            "prompt": prompts,
            "status": status,
            "error": error,
        })


prompt_schema = {
    "document_id": "VARCHAR",
    "source_uri": "VARCHAR",
    "prompt_template": "VARCHAR",
    "prompt": "VARCHAR",
    "status": "VARCHAR",
    "error": "VARCHAR",
}

prompts = prepared.map_batches(
    PromptBuilder,
    schema=prompt_schema,
    batch_size=64,
    execution_backend="subprocess_actor",
    actor_number=2,
    gpus=0.0,
)
```

在真实的模型 UDF 中，请在 `__init__` 中或第一次调用时延迟加载模型、tokenizer 或客户端，而不是在模块导入时加载。这样可以保持导入轻量，并让每个 subprocess 或 Ray actor 拥有自己的状态。

如果需要分布式 actor 复用，把 backend 切换到 `ray_actor`：

```python
ray_prompts = prepared.map_batches(
    PromptBuilder,
    schema=prompt_schema,
    batch_size=128,
    execution_backend="ray_actor",
    actor_number=4,
    gpus=0.0,
)
```

对于依赖 GPU 的状态，请按 actor 请求 GPU 资源：

```python
class GpuEmbedder:
    def __init__(self) -> None:
        # 在这里加载 tokenizer/model，使每个 actor 拥有自己的 GPU state。
        self.model_name = "local-or-mounted-model"

    def __call__(self, batch: pa.Table) -> pa.Table:
        document_ids = batch.column("document_id").to_pylist()
        source_uris = batch.column("source_uri").to_pylist()
        texts = batch.column("normalized_text").to_pylist()

        embeddings = [[0.0, 0.0, 0.0] for _ in texts]
        return pa.table({
            "document_id": document_ids,
            "source_uri": source_uris,
            "embedding": embeddings,
            "status": ["ok"] * len(document_ids),
            "error": [""] * len(document_ids),
        })


embedded = prepared.map_batches(
    GpuEmbedder,
    schema={
        "document_id": "VARCHAR",
        "source_uri": "VARCHAR",
        "embedding": "FLOAT[]",
        "status": "VARCHAR",
        "error": "VARCHAR",
    },
    batch_size=32,
    execution_backend="ray_actor",
    actor_number=2,
    cpus=0.0,
    gpus=1.0,
)
```

`gpus=1.0` 是每个 actor 的资源请求。使用 `actor_number=2` 时，Ray 需要两个可用 GPU 资源位才能满并行运行。更多 GPU 执行模式，请参考 [GPU 推理](/zh-CN/docs/data/guides/gpu-inference)。

每个 Ray worker 都必须能够导入该可调用对象，并访问相同的数据路径、模型文件、凭据和 Python 依赖。

## 行扩展 UDF

当一个输入行可能产生零行、一行或多行输出时，使用 `flat_map`。典型场景包括文档分块、帧提取摘要、嵌套记录规范化、URL 展开，以及通过返回空结果来过滤行。

Python `vane.func.batch(..., row_preserving=False)` 也可以生成改变行数的 batch Expression，但它必须是唯一的顶层 projection，而且 v1 不能把它注册为 SQL alias。当转换天然表示为显式的零行或多行输出时，优先使用 `flat_map`。

可调用对象接收一行 Python `dict`。它可以返回 `None`、一个输出 `dict`，或多个输出 `dict` 组成的可迭代对象。

```python
def chunk_document(row: dict) -> list[dict]:
    text = str(row.get("normalized_text") or "")
    chunks = [text[index:index + 1000] for index in range(0, len(text), 1000)]

    return [
        {
            "document_id": row["document_id"],
            "chunk_id": f"{row['document_id']}:{position}",
            "chunk_text": chunk,
        }
        for position, chunk in enumerate(chunks)
        if chunk
    ]


chunks = prepared.flat_map(
    chunk_document,
    schema={
        "document_id": "VARCHAR",
        "chunk_id": "VARCHAR",
        "chunk_text": "VARCHAR",
    },
    execution_backend="subprocess_task",
)
```

声明的 schema 会从每个输出 `dict` 中选择键。缺失的键会变成 null，额外键会被忽略。

## 标量 UDF

如果只需在现有列旁增加一个简单的逐行值，优先使用上文展示的已注册 SQL scalar Expression；当 projection 在 Python 中构造时，使用 `vane.func`。只有明确需要 Relation executor 及其透传结果形态时，才使用 Relation `map`。

```python
def normalize_document_type(document_type: str) -> str:
    return str(document_type or "unknown").strip().lower()


normalized_types = documents.project("document_type").map(
    normalize_document_type,
    return_type="VARCHAR",
    execution_backend="subprocess_task",
)
```

对于标准 embedding 或 LLM 调用，优先使用内置 AI Function。对于自定义模型调用、媒体解码或任何可以从 batch 处理中获益的操作，一个有类型且保持行数的结果使用 batch Expression，表形结果使用 `map_batches`。

## 共同输出契约

请把每个 UDF 输出都视为显式契约。Expression UDF 向 projection 增加一个声明过的结果列；Relation UDF 则负责它返回的表形。

| 主题 | 建议 |
| --- | --- |
| 返回列 | Expression batch schema 只声明一个输出列。对于 Relation `map_batches` 和 `flat_map`，精确返回 `schema` 中声明的列。 |
| 源字段 | 在 Expression 结果旁选择源字段。Relation UDF 的后续阶段需要哪些源字段，就显式返回哪些字段。 |
| 行诊断 | 当失败行需要保持可审查时，返回显式 `status` 和 `error` 列。 |
| 类型 | 使用 Arrow 兼容值：字符串、数字、布尔值、bytes、list、struct 或 Arrow array。 |
| Null | 保持 null 行为显式，尤其是数值型、bytes、list 和模型输出列。 |
| 行数 | 保持行数的 Expression 必须为每个输入行返回一个输出行。Relation `map_batches` 通常也如此；显式零行或多行输出使用 `flat_map`。 |

如果 Arrow 无法推断你需要的类型，请构造显式 array：

```python
return pa.table({
    "document_id": pa.array(document_ids, type=pa.string()),
    "review_reason": pa.array(review_reason, type=pa.string()),
})
```

避免返回模型类、文件句柄、自定义 dataclass 或原始异常等 Python 对象。返回前请将它们转换为字符串、数字、bytes、list、struct、JSON 字符串或显式 Arrow array。

## Backend 与扩展

Expression UDF 根据可调用对象确定执行形态：`vane.func` 和 `vane.func.batch` 使用 task，`vane.cls` 和 `vane.cls.batch` 使用 actor。通过 Vane runner 配置选择本地或 Ray 执行。Relation UDF 则直接提供 `execution_backend`，应根据可调用对象形态选择：

| Backend | 可调用对象形态 | 适用场景 |
| --- | --- | --- |
| `subprocess_task` | 函数 | 首个无状态 UDF、本地隔离、CPU 转换。 |
| `subprocess_actor` | 可调用类 | 不使用 Ray 的本地模型或客户端复用。 |
| `ray_task` | 函数 | 本地正确性校验后，再执行分布式无状态 batch 工作。 |
| `ray_actor` | 可调用类 | 分布式模型/客户端复用、GPU 资源放置，或更大的 actor 池。 |

唯一有效的 backend 名称是 `subprocess_task`、`subprocess_actor`、`ray_task` 和 `ray_actor`。`local`、`subprocess`、`ray=True`、`use_process`、`actor_count`、`prefer_actor` 等旧写法或缩写写法不是当前 API。

建议按这个顺序推进：

1. 使用 `subprocess_task` 或 `subprocess_actor` 在本地构建 UDF。
2. 在小型真实样本上校验 schema、行数、ID、源列和错误行为。
3. 输出正确后，在本地增大 `batch_size`。
4. 只有当分布式执行有帮助时，才把无状态函数迁移到 `ray_task`。
5. 只有当 actor 复用、GPU 资源放置或集群调度是必要条件时，才把有状态类迁移到 `ray_actor`。

`batch_size` 控制 Vane 每次 `map_batches` 调用传入多少输入行。batch size 不应改变结果语义。如果调整 `batch_size` 后输出发生变化，UDF 很可能依赖了隐藏的 batch 状态。

使用 `actor_number` 设置 `subprocess_actor` 和 `ray_actor` 的 actor 池大小。Task backend 不接受这个 actor 池选项。

## 相关文档

- 当主要任务是表形态清理、join、去重、校验或输出整理时，使用 [结构化转换](/zh-CN/docs/data/guides/structured-transformation)。
- 当 UDF 需要读取 bytes、解码文件、提取媒体特征，或记录逐阶段状态和错误时，使用 [多模态 Pipeline](/zh-CN/docs/data/guides/multimodal-pipeline)。
- 对于标准 embedding、文本分类或提示词增强，使用 [AI 函数](/zh-CN/docs/data/guides/ai-functions)。当你需要自定义模型、自定义 output schema、外部客户端行为，或完全控制预处理与后处理时，使用自定义 Python UDF。
- 使用 [UDF API 参考](/zh-CN/docs/data/reference/udf-api) 查询 SQL 注册、Python Expression 和 Relation API 的精确签名与限制。
- 阅读 [UDF](/zh-CN/docs/data/concepts/udfs)，了解 Expression projection、Relation 表阶段、可调用类和执行 backend 背后的概念模型。
