---
title: 大规模 Embedding
---

使用 Vane Data 构建 embedding pipeline，可以在一个 workflow 中完成 SQL 过滤、batch 模型执行和表输出。

常见形态如下：

1. 用 SQL 读取并过滤源行。
2. 将长文本切成稳定的分块。
3. 优先用 SQL `ai_embed(...)` 对分块做 embedding；当配置更适合放在 Python 中时，也可以使用 Python Expression API。
4. 带着稳定的文档 ID 和分块 ID 写出 embedding。

## 基本文本 embedding

推荐从 SQL Expression API 开始。`ai_embed` 为每个输入行增加一个 embedding 值，因此标识符和源字段可以保留在同一个 projection 中：

```python
import vane

con = vane.connect()
embedded = con.sql("""
    SELECT
        id,
        text,
        ai_embed(
            text,
            struct_pack(
                provider := 'transformers',
                model := 'sentence-transformers/all-MiniLM-L6-v2'
            )
        ) AS embedding
    FROM read_parquet('data/documents/*.parquet')
    WHERE text IS NOT NULL
""")
```

`ai_embed` 在 `SELECT` projection 中执行，可以和普通列组合。未指定 `dimensions` 时，其结果类型是 `FLOAT[]`。options 参数是整个任务共用的常量 `STRUCT`，不是逐行取值的列。

## 先切分长文本

除非你明确希望把整篇文档送入 embedding 模型，否则应创建边界清晰、ID 稳定的分块。

```python
import pyarrow as pa

def chunk_rows(batch: pa.Table) -> pa.Table:
    doc_ids = batch.column("id").to_pylist()
    texts = batch.column("text").to_pylist()
    out_doc_id = []
    out_chunk_id = []
    out_text = []

    for doc_id, text in zip(doc_ids, texts):
        parts = [part.strip() for part in str(text).split(".") if part.strip()]
        for idx, part in enumerate(parts):
            out_doc_id.append(doc_id)
            out_chunk_id.append(idx)
            out_text.append(part)

    return pa.table({
        "id": out_doc_id,
        "chunk_id": out_chunk_id,
        "chunk_text": out_text,
    })

rel = con.sql("""
    SELECT id, text
    FROM read_parquet('data/documents/*.parquet')
    WHERE text IS NOT NULL
""")

chunks = rel.map_batches(
    chunk_rows,
    schema={
        "id": "BIGINT",
        "chunk_id": "BIGINT",
        "chunk_text": "VARCHAR",
    },
    batch_size=512,
    execution_backend="subprocess_task",
)
chunks.to_table("document_chunks")
con.sql("SELECT * FROM document_chunks").write_parquet("data/document-chunks.parquet")
```

这里使用 Relation API，是因为一篇文档可能产生多个输出行。写出的 Parquet 副本会作为下方独立 Ray 示例的输入。生产系统中，应使用能感知 tokenizer 的分块器，使其匹配模型上下文窗口，并保留足够的重叠区间以保证检索质量。

## 使用本地 embedding 模型

本地模型通常最适合 actor 执行，因为模型初始化成本较高。SQL `ai_embed` 会创建相应的执行路径，同时把源列保留在 projection 中：

```python
embedded_local = con.sql("""
    SELECT
        id,
        chunk_id,
        chunk_text,
        ai_embed(
            chunk_text,
            struct_pack(
                provider := 'transformers',
                model := 'sentence-transformers/all-MiniLM-L6-v2',
                batch_size := 64
            )
        ) AS embedding
    FROM document_chunks
""")
```

当 typed 配置或 Python 组合更方便时，可以改用 Python Expression API：

```python
embedded_python = chunks.select(
    vane.col("id"),
    vane.col("chunk_id"),
    vane.col("chunk_text"),
    vane.ai.embed(
        vane.col("chunk_text"),
        provider="transformers",
        model="sentence-transformers/all-MiniLM-L6-v2",
    ).alias("embedding"),
)
```

两种写法都是保持行数的 Expression 操作。当 provider 检测到可用 GPU 资源时，可能会向 UDF 执行层申请 GPU。

## 使用托管 embedding API

当你希望使用托管模型，并且可以接受 API 延迟和速率限制时，托管 provider 很有用。SQL 形式仍然是首选入口：

```python
embedded_hosted = con.sql("""
    SELECT
        id,
        chunk_id,
        chunk_text,
        ai_embed(
            chunk_text,
            struct_pack(
                provider := 'openai',
                model := 'text-embedding-3-small',
                dimensions := 512,
                normalize := true,
                batch_size := 64,
                concurrency := 8
            )
        ) AS embedding
    FROM document_chunks
""")
```

请在每个 worker 的环境中设置 `OPENAI_API_KEY` 等凭据；SQL options 会拒绝 credential-like 字段。`concurrency` 控制 AI actor 数量，请求级并发则取决于具体 provider。只有在有界输入上测量过速率限制、retry 和成本后，才应提高它们。

## 使用 Ray 处理更大任务

本地路径正确后，再迁移到 Ray。应在创建连接或输入 relation 前配置 runner；SQL Expression 本身不需要改变：

```python
import vane

vane.configure(runner="ray")
con = vane.connect()

embedded_ray = con.sql("""
    SELECT
        id,
        chunk_id,
        chunk_text,
        ai_embed(
            chunk_text,
            struct_pack(
                provider := 'transformers',
                model := 'sentence-transformers/all-MiniLM-L6-v2',
                batch_size := 64
            )
        ) AS embedding
    FROM read_parquet('data/document-chunks.parquet')
""")
```

每个 Ray worker 都必须能够 import provider 库，并访问模型文件或 provider 凭证。本地 subprocess 执行只会继承可见设备，不会预留或隔离 GPU；需要明确的资源调度时应使用 Ray。

## 输出形态

写出完整 relation，让下游检索和审计更直接：

- `document_id`
- `chunk_id`
- `chunk_text`
- `embedding`
- `source_uri`
- 模型名和 provider
- 需要时记录创建时间戳或 run ID

Expression API 会把标识符和 embedding 保留在同一个 projection 中，因此校验后可以直接写出结果：

```python
embedded_hosted.write_parquet("output/embeddings/part-00000.parquet")
```

写出前应校验行数、稳定 ID、null 和向量维度。对于分布式写出，请确保每个 worker 都能访问输出文件系统。

## 调优清单

- 在 embedding 前先过滤并投影。
- 保持分块边界清晰且结果具有确定性。
- 本地模型使用 actor 执行。
- 增大 `batch_size`，直到延迟或内存成为瓶颈。
- 只有 worker 和 API quota 仍有余量时，才增大 actor 数或 provider 请求并发。
- 当扫描、推理或写入吞吐量超出单进程能力，或需要资源调度时使用 Ray。
- 使用稳定的文档 ID 和分块 ID，使输出行保持确定性。
- 只有 provider 和 model 支持目标向量大小时，才设置固定的 `dimensions`。
