使用 Vane Data 构建 embedding pipeline，可以在一个 workflow 中完成 SQL 过滤、batch model execution 和 table output。

通常形态是：

1. 用 SQL 读取并过滤 source row。
2. 将长文本切成稳定 chunk。
3. 用 `rel.embed_text(...)` 或自定义 UDF 对 chunk 做 embedding。
4. 带着稳定的 document ID 和 chunk ID 写出 embedding。

## 基本文本 Embedding

```python
import vane

con = vane.connect()
rel = con.sql("""
    select id, text
    from read_parquet('data/documents/*.parquet')
    where text is not null
""")

embedding_only = rel.embed_text(
    "text",
    provider="transformers",
    model="sentence-transformers/all-MiniLM-L6-v2",
    output_column="embedding",
    execution_backend="subprocess_actor",
)
```

`embed_text` 返回包含所配置 embedding output column 的 relation。这个 high-level helper 会把 embedding 写成 `FLOAT[]`；它不会自动包含 `id` 或 `text` 等 source column。

## 先切分长文本

除非你明确希望把整篇文档送进 embedding model，否则应创建有边界且 ID 稳定的 chunk。

```python
import pyarrow as pa

def chunk_rows(batch: pa.Table) -> pa.Table:
    doc_ids = batch.column("id").to_pylist()
    texts = batch.column("text").to_pylist()
    out_doc_id = []
    out_chunk_id = []
    out_text = []

    for doc_id, text in zip(doc_ids, texts):
        parts = [part.strip() for part in str(text).split(".") if part.strip()]
        for idx, part in enumerate(parts):
            out_doc_id.append(doc_id)
            out_chunk_id.append(idx)
            out_text.append(part)

    return pa.table({
        "id": out_doc_id,
        "chunk_id": out_chunk_id,
        "chunk_text": out_text,
    })

chunks = rel.map_batches(
    chunk_rows,
    schema={
        "id": "BIGINT",
        "chunk_id": "BIGINT",
        "chunk_text": "VARCHAR",
    },
    batch_size=512,
    execution_backend="subprocess_task",
)
```

生产系统中，应使用 tokenizer-aware chunker，使其匹配 model context window，并保留足够 overlap 以保证 retrieval quality。

## 使用本地 Embedding Model

本地 model 通常最适合 actor backend，因为 model initialization 成本较高：

```python
embedding_only = chunks.embed_text(
    "chunk_text",
    provider="transformers",
    model="sentence-transformers/all-MiniLM-L6-v2",
    output_column="embedding",
    execution_backend="subprocess_actor",
    batch_size=64,
)
```

当 provider 检测到可用 GPU resource 时，可能会向 UDF execution layer 传递 GPU resource request。如果需要严格控制 device，或需要包含 ID 的完整 output row，请使用自定义 `map_batches` callable class 返回 identifier、text 和 embedding。

## 使用 Hosted Embedding API

当你希望使用 managed model，并且可以接受 API latency 和 rate limit 时，hosted provider 很有用：

```python
embedding_only = chunks.embed_text(
    "chunk_text",
    provider="openai",
    model="text-embedding-3-small",
    output_column="embedding",
    execution_backend="subprocess_actor",
    batch_size=64,
    concurrency=8,
)
```

请谨慎调节 `batch_size` 和 `concurrency`。更高的 concurrency 可以提升 throughput，但也可能触发 provider rate limit 或放大 retry。如果在 helper call 之后把 provider output 与 source column 组合，请在该步骤中校验 row count 并保持稳定顺序。

## 使用 Ray 处理更大任务

本地路径正确后，再迁移到 Ray：

```python
import vane

vane.configure(runner="ray")

embedding_only = chunks.embed_text(
    "chunk_text",
    provider="transformers",
    model="sentence-transformers/all-MiniLM-L6-v2",
    output_column="embedding",
    execution_backend="ray_actor",
)
```

每个 Ray worker 都必须能够 import provider library，并访问 model file 或 provider credential。

## 输出形态

写出完整 relation，让下游 retrieval 和 audit 更直接：

- `document_id`
- `chunk_id`
- `chunk_text`
- `embedding`
- `source_uri`
- model name 和 provider
- 需要时记录 creation timestamp 或 run ID

使用 relation writer 写出结果：

```python
embedded_with_ids.write_parquet("output/embeddings/")
```

`embedded_with_ids` 应包含上面列出的 identifier 和 metadata column。它可以来自返回完整 schema 的自定义 embedding UDF，也可以来自 AI helper output 与 source row 的显式组合；组合前需要校验 row count 和 ordering。

对于 distributed write，请确保每个 worker 都能访问 output filesystem。

## Tuning Checklist

- 在 embedding 前先 filter 和 project。
- 保持 chunk 有边界且 deterministic。
- 本地 model 使用 actor backend。
- 增大 `batch_size`，直到 latency 或 memory 成为瓶颈。
- 只有 worker 或 API quota 仍有余量时，才增大 `concurrency`。
- 当 scan、inference 或 write throughput 超出单进程能力时使用 Ray。
- 使用稳定 document ID 和 chunk ID 保持 output row deterministic。
