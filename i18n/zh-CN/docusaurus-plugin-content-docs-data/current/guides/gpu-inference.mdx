---
title: GPU 推理
---

当你需要批量将多模态或文本数据交给本地或私有 LLM 推理时，请使用 GPU Inference。典型输入包括抽取后的文档、字幕、转写文本、标签，或前序 Vane 阶段准备好的提示词行。

本指南聚焦 Vane 中用于 batch 提示词推理的 vLLM 路径。对于不依赖 vLLM 的通用 GPU 模型代码，请使用 [Custom Python UDFs](/zh-CN/docs/data/guides/custom-python-udfs)。

常见 workflow 如下：

```text
load source rows -> build prompt column -> keep IDs and source metadata -> run vLLM batches -> validate output -> write result table
```

## 何时使用 GPU Inference

在以下场景中使用本指南：

- 你有大量提示词，而不是少量交互式聊天请求。
- 模型需要在本地或你自己的 Ray 环境中运行。
- 模型或引擎初始化成本较高，需要跨 batch 复用。
- 你需要验证 GPU actor 复用、多 GPU 调度，或基于 Ray 的 vLLM actor pool。
- 相似提示词共享系统前缀或模板，而缓存局部性可能影响吞吐量。

## Vane 中的 vLLM

Vane 通过 AI Function `prompt(...)` 暴露 vLLM：

```python
response_only = prompts.prompt(
    "prompt",
    provider="vllm",
    model="Qwen/Qwen3-1.7B",
    output_column="response_text",
    batch_size=8,
    execution_backend="subprocess_actor",
)
```

## 构建提示词 relation

从一小段真实源行切片开始。将提示词文本与后续需要的标识符和元数据分开保留。

```python
import vane

con = vane.connect()

prompts = con.sql("""
    select
        case_id,
        document_id,
        source_uri,
        document_type,
        'Extract the claim decision and one short reason from this document: '
            || text as prompt
    from read_parquet('data/claims/documents/*.parquet')
    where text is not null
    order by case_id, document_id
    limit 50
""")

prompt_table = prompts.to_arrow_table()
```

推荐的输入列：

| 列 | 作用 |
| --- | --- |
| `case_id`、`document_id`、`asset_id` 或其他稳定 key | 将响应关联回源系统。 |
| `source_uri` 或源元数据 | 审查异常输出并复现失败。 |
| `document_type`、`media_type`、类别、租户或任务类型 | 对提示词分组，并按来源分段比较输出质量。 |
| `prompt` | 发送给 vLLM 的最终文本。 |
| `system_prefix` 或模板 ID，如果使用 | 用于解释提示词分组和缓存局部性实验。 |

第一次运行时保持提示词构造过程可复现。如果修改模板，请在输出中记录 `run_id` 或提示词版本。

## 运行本地 vLLM 样例

在引入 Ray actor pool 的复杂度前，先使用本地路径验证模型、提示词形态、输出长度和 GPU 内存行为。

```python
response_only = prompts.prompt(
    "prompt",
    provider="vllm",
    model="Qwen/Qwen3-1.7B",
    system_message="Return concise JSON-like business notes.",
    output_column="response_text",
    batch_size=8,
    execution_backend="subprocess_actor",
    engine_args={
        "trust_remote_code": True,
        "max_model_len": 2048,
    },
    generate_args={
        "sampling_params": {
            "temperature": 0.0,
            "max_tokens": 128,
        }
    },
    on_error="log",
)
```

`execution_backend="subprocess_actor"` 会将 vLLM prompter 状态保留在 actor 进程中，使引擎可以在同一次运行的多个 batch 之间复用。第一次正确性检查不需要 Ray。

## 保留 ID 并写出结果

在将辅助输出追加到原始 relation 前，先完成校验。

```python
import pyarrow as pa

response_table = response_only.to_arrow_table()

if response_table.num_rows != prompt_table.num_rows:
    raise ValueError("vLLM output row count does not match prompt row count")

responses = response_table["response_text"].to_pylist()
status = ["ok" if value else "error" for value in responses]
error_message = [
    None if value else "vLLM returned null or empty output"
    for value in responses
]

result_table = prompt_table.append_column(
    "response_text",
    pa.array(responses, type=pa.string()),
)
result_table = result_table.append_column(
    "status",
    pa.array(status, type=pa.string()),
)
result_table = result_table.append_column(
    "error_message",
    pa.array(error_message, type=pa.string()),
)
result_table = result_table.append_column(
    "model",
    pa.array(["Qwen/Qwen3-1.7B"] * prompt_table.num_rows, type=pa.string()),
)
result_table = result_table.append_column(
    "run_id",
    pa.array(["claims-vllm-validation-001"] * prompt_table.num_rows, type=pa.string()),
)

results = con.from_arrow(result_table)
results.write_parquet("output/claims-vllm-validation-001/")
```

建议的输出列：

| 列 | 作用 |
| --- | --- |
| 业务 ID 列 | 将每个响应追溯到源行。 |
| 来源元数据 | 复现并审查失败。 |
| `prompt` 或提示词版本 | 解释模型看到了什么。 |
| `response_text` | 生成的文本。 |
| `status` | `ok`、`error`，或你定义的业务审核状态。 |
| `error_message` | 用于审核的可空诊断列。 |
| `model`、`run_id` 和配置摘要 | 比较不同运行，而无需猜测哪些设置发生了变化。 |

对于更严格的结构化输出，将 `return_format` 传给 `prompt(...)`，并在写入最终表之前校验返回的 JSON 字符串。

## 本地 vLLM 与基于 Ray 的 vLLM

使用能够回答当前问题的最小执行形态。

| 形态 | 适用场景 | 校验内容 |
| --- | --- | --- |
| 本地 vLLM 样例 | 提示词正确性、模型加载、输出质量，以及第一次 GPU 内存检查。 | 10-100 行、非空输出、预期输出格式，并确认没有每行重新加载模型。 |
| Ray actor 执行 | 多 GPU 校验、actor 复用和更大的提示词 batch。 | Ray 能看到预期 GPU，worker 能导入 vLLM，并且每个 actor 都有模型文件和依赖。 |
| 原生 vLLM Ray actor pool | 更大规模的 batch 推理，Vane 需要复用 vLLM actor 并在它们之间路由工作。 | actor pool 大小、GPU 分配、共享池复用、进行中负载分布和吞吐量。 |

对于基于 Ray 的 vLLM 校验，请在构建分布式运行前配置 Ray，并保持模型选项与本地样例一致。

```python
import vane

vane.configure(runner="ray")

ray_response_only = prompts.prompt(
    "prompt",
    provider="vllm",
    model="Qwen/Qwen3-1.7B",
    output_column="response_text",
    batch_size=32,
    concurrency=2,
    gpus_per_actor=1,
    execution_backend="ray_actor",
    use_ray=True,
    ray_actor_pool_name="claims-vllm-validation",
    engine_args={
        "trust_remote_code": True,
        "max_model_len": 2048,
    },
    generate_args={
        "sampling_params": {
            "temperature": 0.0,
            "max_tokens": 128,
        }
    },
    on_error="log",
)
```

需要区分两层配置：

- `execution_backend` 控制 Vane 在哪里运行提示词 UDF 阶段。
- `use_ray`、`concurrency`、`gpus_per_actor` 和 `ray_actor_pool_name` 等 vLLM 选项控制 vLLM 执行器和 actor pool 行为。

开始时一次只改变一层。如果本地样例还不正确，Ray 会让失败更难诊断。

## Batch 缓存优化

vLLM 吞吐量不只取决于 GPU 数量。对于 batch 推理运行，需要围绕引擎复用、有界 batch 和提示词局部性调优。

### 复用 actor 和引擎

避免每行或每个很小的 batch 都重新加载模型。对 vLLM 提示词使用 actor 式执行，并在一次运行中保持 `model`、`engine_args` 和依赖版本稳定。

对于基于 Ray 的 vLLM，当多个分布式任务需要共享同一组 vLLM actor 时，使用具名 actor pool：

```python
ray_actor_pool_name="claims-vllm-validation"
```

本地代码支持具名 vLLM actor pool，并按池名称共享进行中请求跟踪。这使多个执行器可以看到每个 actor 的当前负载，并以更好的负载感知能力路由新的提示词 batch。它也能避免在分布式计划中反复创建相同 actor pool。

### 对相似前缀分组

当大量提示词共享同一个系统消息、任务模板或长前缀，并且分组不会改变本次运行的业务语义时，可以在推理前分组。

```python
prompts = con.sql("""
    select
        case_id,
        document_id,
        document_type,
        source_uri,
        case
            when document_type = 'claim'
                then 'You review insurance claim documents.'
            when document_type = 'invoice'
                then 'You review invoice support documents.'
            else 'You review business documents.'
        end as system_prefix,
        'Return a short decision and reason: ' || text as prompt
    from source_documents
    where text is not null
    order by document_type, case_id, document_id
""")
```

当一个前缀适用于所有行时，使用 `system_message`。如果前缀按行变化，请将前缀放入提示词文本，或按任务类型拆分 relation。最终输出仍应以业务 ID 为 key，避免分组执行让审核变得含糊。

原生 vLLM 执行器包含 `do_prefix_routing`、`min_bucket_size`、`prefix_match_threshold` 和 `load_balance_threshold` 等前缀路由控制项。请在已有基线后再把它们作为调优控制项使用。它们不是通用 UDF 路由选项。

### 联合调优 batch 和并发度

为 relation 中的 vLLM 提示词显式设置 `batch_size`。更大的 batch 可以改善 GPU 利用率和 vLLM continuous batching，但也可能增加延迟、内存压力和故障影响范围。

只有在 GPU 和 actor 内存有余量时，才增加 `concurrency`。一个简单的容量检查是：

```text
required GPUs = concurrency * gpus_per_actor
```

如果 `concurrency=2` 且 `gpus_per_actor=1`，Ray 环境至少需要两个可用 GPU 才能实现完整并行。如果模型使用张量并行或其他 vLLM 引擎级 GPU 设置，请把它们纳入自己的容量估算。

## 关键配置概念

| 概念 | 出现位置 | 建议 |
| --- | --- | --- |
| `model` | `prompt(..., model=...)` | 使用你已在本地验证过的准确模型标识符。 |
| `engine_args` | vLLM 引擎配置 | 放置模型加载和引擎级设置，例如上下文长度或信任设置。 |
| `generate_args` | vLLM 生成请求 | 放置采样参数，通常位于 `sampling_params` 下。 |
| `batch_size` | Vane UDF batch 和 vLLM 提交 | 从小值开始，然后增大到 GPU 利用率、延迟或内存成为瓶颈。 |
| `concurrency` | Actor 或 vLLM 池大小 | 只有 worker 和 GPU 仍有余量时才增加。 |
| `gpus_per_actor` | vLLM actor 的 Ray GPU 资源请求 | 与模型和引擎设置匹配。 |
| `ray_actor_pool_name` | 原生 vLLM Ray actor pool | 在分布式运行中用于共享 actor 复用。 |
| `on_error` | vLLM 执行器错误行为 | 调试时使用 `raise`；对于需要保留失败行以便审核的校验运行，使用 `log`。 |

Vane 为原生 vLLM 引擎调度器应用面向吞吐量的默认值，你可以通过 `engine_args` 覆盖引擎级设置。除非明确知道自己在测试哪个设置，否则不要把完整的 vLLM 服务配置复制到校验运行中。

## 边界

本指南不是 vLLM 手册。它展示如何将 vLLM batch 推理放入 Vane relation workflow，以及在扩大规模前需要验证什么。

它也不是生产服务指南，不覆盖自动扩缩容、SLA、流量管理、在线请求路由或长期运行服务的运维。

对于非 vLLM GPU 模型、自定义分词器、多模态预处理，或完整的自定义输出 schema，请使用 [Custom Python UDFs](/zh-CN/docs/data/guides/custom-python-udfs)。对于托管 provider 的提示词调用和小型 AI 增强任务，请使用 [AI Functions](/zh-CN/docs/data/guides/ai-functions)。
