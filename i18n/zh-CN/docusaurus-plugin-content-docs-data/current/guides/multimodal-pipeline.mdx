---
title: SQL 多模态 Pipeline
---

本指南展示一种 SQL 优先的多模态数据准备模式。SQL 负责元数据过滤和校验；Python UDF 处理媒体专用步骤；结果再回到 SQL 中进行检查并生成输出。

当 pipeline 的大部分逻辑都需要让 SQL 用户读懂，但其中某个阶段需要 Python 库完成解码、特征提取或模型推理时，可以使用这种模式。

## Pipeline 结构

这个模式包含五个阶段：

1. 使用 SQL 选择候选行。
2. 加载字节，或保留稳定的媒体引用路径。
3. 运行 Python UDF 做解码或推理。
4. 回到 SQL 做校验和汇总。
5. 写出整理后的表格输出。

Vane Data 不会为 SQL 增加图像、音频或视频语法。多模态处理会表示为表中的列和显式 Python UDF 阶段。

## 1. 从 SQL 元数据开始

第一步保持关系型处理。先过滤行、规范化元数据，并在数据进入 Python 前减少输入量。

```python
import vane

con = vane.connect()

items = con.sql("""
    select
        id,
        path,
        media_type
    from read_parquet('data/media_index/*.parquet')
    where media_type in ('image/png', 'image/jpeg')
      and path is not null
""")
```

## 2. 添加字节或引用路径

如果媒体字节已经在 Parquet 中，就保留在 relation 里并跳过这一步。如果表中存储的是路径，使用 UDF 加载字节，并返回后续阶段需要的列。

```python
from pathlib import Path

import pyarrow as pa

def load_image_bytes(batch: pa.Table) -> pa.Table:
    ids = batch.column("id").to_pylist()
    paths = batch.column("path").to_pylist()
    media_types = batch.column("media_type").to_pylist()

    return pa.table({
        "id": ids,
        "path": paths,
        "media_type": media_types,
        "image_bytes": [Path(path).read_bytes() for path in paths],
    })

with_bytes = items.map_batches(
    load_image_bytes,
    schema={
        "id": "BIGINT",
        "path": "VARCHAR",
        "media_type": "VARCHAR",
        "image_bytes": "BLOB",
    },
    batch_size=32,
    execution_backend="subprocess_task",
)
```

`map_batches` 返回 UDF 产出的列。当下游阶段仍然需要 `id`、`path` 和 `media_type` 等输入列时，请从 UDF 中返回它们。

## 3. 运行模型阶段

当模型、分词器、图像处理器或客户端需要只初始化一次并跨 batch 复用时，使用 actor backend。

```python
class ImageClassifier:
    def __init__(self) -> None:
        # Initialize model state here in a real pipeline.
        self.labels = ("invoice", "claim", "other")

    def __call__(self, batch: pa.Table) -> pa.Table:
        ids = batch.column("id").to_pylist()
        paths = batch.column("path").to_pylist()
        media_types = batch.column("media_type").to_pylist()
        image_bytes = batch.column("image_bytes").to_pylist()

        labels = []
        scores = []
        for value in image_bytes:
            labels.append(self.labels[0] if value else self.labels[2])
            scores.append(1.0 if value else 0.0)

        return pa.table({
            "id": ids,
            "path": paths,
            "media_type": media_types,
            "label": labels,
            "score": scores,
        })

classified = with_bytes.map_batches(
    ImageClassifier,
    schema={
        "id": "BIGINT",
        "path": "VARCHAR",
        "media_type": "VARCHAR",
        "label": "VARCHAR",
        "score": "DOUBLE",
    },
    batch_size=16,
    execution_backend="subprocess_actor",
)
```

这个示例分类器有意保持简单，方便看清 relation 契约。在生产中，把解码、预处理和推理放进 actor，同时保持返回 schema 稳定。

## 4. 回到 SQL

当你想用普通 SQL 做校验、汇总或下游 join 时，从 relation 创建表。

```python
classified.to_table("classified_media")

summary = con.sql("""
    select
        label,
        count(*) as rows,
        avg(score) as avg_score
    from classified_media
    group by label
    order by rows desc
""")

summary.show()
```

## 5. 写出输出

只写出下游系统需要的列。大块媒体字节在推理期间通常有用，但在整理后的输出里往往不需要。

```python
curated = con.sql("""
    select id, path, media_type, label, score
    from classified_media
""")

curated.write_parquet("output/classified_media/")
```

## 迁移到 Ray

将开销较高的阶段迁移到 Ray 时，SQL 文本和 UDF 契约不需要变化。请在创建并执行 pipeline 前配置 runner：

```python
import vane

vane.configure(runner="ray")
```

然后为这个开销较高的阶段选择 Ray UDF backend：

```python
classified = with_bytes.map_batches(
    ImageClassifier,
    schema={
        "id": "BIGINT",
        "path": "VARCHAR",
        "media_type": "VARCHAR",
        "label": "VARCHAR",
        "score": "DOUBLE",
    },
    batch_size=16,
    execution_backend="ray_actor",
    gpus=1,
    concurrency=4,
)
```

每个 Ray worker 都必须能够导入 UDF 类，并访问相同的存储、凭证、模型文件和 Python 依赖。

## 实用检查

在推广多模态 pipeline 前：

- 使用小型本地运行验证 UDF 输出 schema。
- 将过滤和列投影放在字节加载和推理之前。
- 保留稳定的标识符，使模型输出可以关联回源记录。
- 决定媒体字节属于最终输出，还是只属于中间阶段。
- 扩展规模前，先从每个 Ray worker 测试路径、存储和凭证访问。
