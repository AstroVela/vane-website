---
title: 什么是 Vane Data？
---

Vane Data 是一个多模态原生数据引擎，面向多模态数据准备、面向 AI 的数据集和 batch 推理 pipeline。它把兼容 DuckDB 的 SQL、Python/Arrow UDF、AI Function，以及可选的 Ray 执行能力组合到一个基于 relation（关系对象）的 workflow 中。

当多模态对象、文本记录、模型输入、模型输出和业务元数据等数据，在结果进入训练、搜索、分析、审核或下游应用之前，还需要 Python 库、自定义逻辑、embedding、提示词调用、模型推理或分布式执行时，可以使用 Vane Data。

## Vane Data 为什么存在

Vane Data 的出发点很直接：AI 数据 pipeline 既需要更高吞吐量，也需要更少跨系统集成代码。

### 性能：更高吞吐量和更充分的 GPU 利用率

面向训练规模的多模态数据准备、离线描述生成、自动标注、质量评分、embedding、去重和历史数据重处理，通常会受限于吞吐量和 GPU 利用率。昂贵的加速器不应在 pipeline 读取数据、解码媒体、构造模型输入、执行推理、写出结果和处理失败时空转。

Vane Data 面向这种工作负载形态设计：

- 高效的异构执行会重叠媒体解码、GPU 上的描述生成、自动标注、embedding 和 I/O，而不是把每个阶段都拆成独立任务。
- 流式处理、反压和动态 batching 让大型媒体与传感器对象持续流过 pipeline，而不必把每个中间对象都加载进内存。
- 基于 Ray 的分布式执行可以把更大的历史重处理任务作为一个可扩展执行图运行，而不是变成跨多个系统、持续多天的交接流程。

### 简洁性：一个引擎，少写集成代码

今天的 AI 数据 pipeline 往往把文件选择、媒体解码、模型推理、质量过滤、embedding、去重和数据集打包分散在多个独立任务与手写集成代码里。

Vane Data 的目标是让这类工作像一个面向表的 pipeline：

```text
source data -> relation/SQL -> Python UDF or AI function -> validation -> tabular output
```

这意味着：

- 一个引擎、一个执行图，把兼容 DuckDB 的 SQL、Python UDF、AI Function 和可选的 Ray 执行能力组合起来。
- 兼容 DuckDB 的 API 让表格数据处理保持熟悉的使用方式，也降低从现有 Ray、Spark、Daft 或 DuckDB 风格 pipeline 迁移的成本。
- 从原始数据到发布的路径可以放在一个可读程序里，而不需要为每个阶段单独写编排集成代码。
- 每一步都会产生另一个带有显式列的 relation，因此 pipeline 始终可检查，并且可以在扩展规模之前先做校验。

## Vane Data 适合什么

当你希望在真实数据上运行早期校验 workflow，且工作负载具备以下形态时，Vane Data 会比较适合：

- 把文档、图像、音频、视频、文本记录、模型输入和模型输出表示为带有稳定 ID 与元数据的行。
- 使用 SQL 对关系型部分做筛选、投影、连接、聚合、抽样和校验。
- 当逻辑需要外部库、自定义规则、媒体处理或模型预处理/后处理时，使用 Python UDF。
- 对 embedding、分类、提示词调用和结构化模型输出等常见数据富集任务使用 AI Function。
- 当更大的提示词或模型工作负载需要 GPU 执行时，使用 GPU 推理，包括基于 vLLM 的 batch 推理。
- 从本地执行开始；当吞吐量、Ray actor 复用、重处理规模或 GPU 资源放置/调度变得重要时，再把选定阶段迁移到 Ray。

典型的场景包括：

- 将原始文本转换为训练、评估或检索数据集。
- 以带稳定 ID 和元数据的行来处理文档、图像、音频或视频。
- 在已有表上运行 batch AI 数据富集。
- 以更高吞吐量重处理历史媒体、文档或模型输出。
- 要求一个本地 workflow 可以快速迁移到基于 Ray 的执行方式。

## Vane Data 不是什么

不要把 Vane Data 当作：

- 持久化事务数据库。
- 低延迟在线模型服务。
- 向量数据库或检索服务层。
- workflow 调度器、编排平台或血缘系统。
- 完整的 MLOps 平台。
- 完整的湖仓、数仓或数据目录管理系统。
- 传统结构化 ETL 工具或数据仓库的通用替代品。


Vane Data 可以把表格输出交给下游系统，但服务化、事务、目录语义、治理和生产运维仍由这些系统负责。

## Vane Data 与 DuckDB 和 Ray 的关系

Vane Data 使用兼容 DuckDB 的 SQL 和 relation 概念，作为表格形态 AI 数据处理的基础。如果你熟悉 DuckDB，对连接、SQL、relation 和文件读取这些基本心智模型也会很熟悉。

Vane Data 在这个模型周围加入面向 AI 的能力和基于 Ray 的分布式执行能力。SQL 层的作用是让多模态和 AI pipeline 保持可检查，而不是替代成熟的数仓或传统 ETL 技术栈。

## 现在是否应该试用 Vane Data？

如果以下多数条件成立，Vane Data v0.1.0 值得试用：

- 你的 workflow 涉及多模态数据、文本密集型 AI 数据、模型推理、embedding、提示词调用或数据集准备。
- 你的 workflow 同时需要关系型操作和 Python、AI 或模型推理工作。
- 你关心吞吐量、GPU 利用率、历史数据重处理，或希望减少跨独立任务的集成代码。
- 你希望输出保持表格形态且可审计。
- 你可以先从本地执行开始，再迁移到 Ray。
